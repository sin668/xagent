# 第二阶段小范围运行部署与运行手册

创建日期：2026-06-03  
适用阶段：第二阶段小范围运行  
适用对象：研发、运维、运营复核、合规负责人、客服/出口销售负责人

## 1. 运行目标

第二阶段目标是把 LLM 来源发现、来源人工复核、来源驱动线索抽取、AI 审计、移动端审核和管理后台治理串成可小范围运行的闭环。

当前已验证能力：

- FastAPI 后端可连接真实 PostgreSQL。
- Redis 可用于调度锁健康验证。
- `llm_prompt_templates`、`agent_task_runs`、`lead_source_candidates`、`staging_leads`、`ai_audit_logs` 等第二阶段核心表已随 Alembic migration 创建。
- `/llm-health` 可返回 LLM Provider 配置健康状态。
- `/agent-tasks/source-discovery/run` 可启动 SOURCE_DISCOVERY 任务并写入任务审计。
- `/agent-tasks/lead-extraction/from-sources/run` 可从已审核来源候选中创建 LEAD_EXTRACTION 任务。
- 移动端 H5 可查看和审核来源候选。
- 管理后台可查看第二阶段运行看板和 LLM/Prompt 治理信息。

当前限制：

- 真实外部 LLM 调用只有在 `LLM_API_KEY` 或 `VEHICLE_LEADS_LLM_API_KEY` 配置后才能正常运行。
- `AGENT_SCHEDULER_ENABLED` 默认关闭。
- 当前 `AgentSchedulerService` 已注册定时任务和 Redis lock 机制，但默认 handler 为 placeholder；启用自动定时前必须确认实际 handler 已绑定到 SOURCE_DISCOVERY、LEAD_EXTRACTION 和 retry 逻辑。

## 2. 合规边界

以下规则在第二阶段继续强制执行：

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不绕过验证码、登录墙、频控或反爬机制。
- 不抓取非公开数据。
- Forbidden 来源不得进入自动抽取。
- High 来源未人工批准不得进入自动抽取。
- Invalid 和 Watch 不进入触达队列。
- 勿扰客户不得再次进入触达队列。
- C 级线索报价、合同或实质交易前必须合规复核。
- 所有 AI 输出必须保留来源证据、prompt 版本、模型、token/延迟信息和任务审计。

## 4. 环境变量

后端读取 `apps/api/.env`，同一项支持普通变量名和 `VEHICLE_LEADS_` 前缀变量名。

| 变量 | 必填 | 说明 |
|---|---|---|
| `DATABASE_URL` 或 `VEHICLE_LEADS_DATABASE_URL` | 是 | PostgreSQL 连接串，示例：`postgresql+asyncpg://user:password@host:5432/db` |
| `REDIS_URL` 或 `VEHICLE_LEADS_REDIS_URL` | 第二阶段建议必填 | Redis 连接串 |
| `CORS_ORIGINS` 或 `VEHICLE_LEADS_CORS_ORIGINS` | 否 | 前端跨域白名单 |
| `LLM_PROVIDER` 或 `VEHICLE_LEADS_LLM_PROVIDER` | 是 | 当前默认 `deepseek` |
| `LLM_API_KEY` 或 `VEHICLE_LEADS_LLM_API_KEY` | 真实 LLM 必填 | LLM Provider API key，不得写入文档和日志 |
| `LLM_BASE_URL` 或 `VEHICLE_LEADS_LLM_BASE_URL` | 是 | OpenAI-compatible chat completions base URL，默认 `https://api.deepseek.com/v1` |
| `LLM_DEFAULT_MODEL` 或 `VEHICLE_LEADS_LLM_DEFAULT_MODEL` | 是 | 默认模型 |
| `LLM_SOURCE_DISCOVERY_MODEL` 或 `VEHICLE_LEADS_LLM_SOURCE_DISCOVERY_MODEL` | 是 | SOURCE_DISCOVERY 模型 |
| `LLM_EXTRACTION_MODEL` 或 `VEHICLE_LEADS_LLM_EXTRACTION_MODEL` | 是 | LEAD_EXTRACTION 模型 |
| `LLM_GRADING_MODEL` 或 `VEHICLE_LEADS_LLM_GRADING_MODEL` | 是 | LEAD_GRADING 模型 |
| `AGENT_SCHEDULER_ENABLED` 或 `VEHICLE_LEADS_AGENT_SCHEDULER_ENABLED` | 否 | 默认 `false`，启用自动定时需设为 `true` |
| `AGENT_SCHEDULER_LOCK_TTL_SECONDS` 或 `VEHICLE_LEADS_AGENT_SCHEDULER_LOCK_TTL_SECONDS` | 否 | Redis lock TTL，默认 `300` |

`.env` 示例：

```bash
DATABASE_URL=postgresql+asyncpg://xagent:<password>@127.0.0.1:5432/xagent
REDIS_URL=redis://127.0.0.1:6379/0
CORS_ORIGINS=http://localhost:5176,http://127.0.0.1:5176,http://localhost:5174,http://127.0.0.1:5174

LLM_PROVIDER=deepseek
LLM_BASE_URL=https://api.deepseek.com/v1
LLM_API_KEY=<secret>
LLM_DEFAULT_MODEL=deepseek-chat
LLM_SOURCE_DISCOVERY_MODEL=deepseek-chat
LLM_EXTRACTION_MODEL=deepseek-chat
LLM_GRADING_MODEL=deepseek-chat

AGENT_SCHEDULER_LOCK_TTL_SECONDS=300
```

## 8. LLM Agent 手动运行

### 8.1 Source Discovery

用途：自动发现新的来源候选并写入 `lead_source_candidates`。

请求：

```bash
curl -sS -X POST http://127.0.0.1:8000/agent-tasks/source-discovery/run \
  -H 'Content-Type: application/json' \
  -d '{
    "country": "Russia",
    "cities": ["Moscow"],
    "channel_strategy": "Low/Medium公开官网、公开目录、地图公开结果优先；High仅进入人工复核，不自动抽取；Forbidden禁止。",
    "keywords": ["автосалон", "used cars", "import cars"],
    "limit": 20
  }'
```

成功时返回：

```json
{
  "agent_task_run_id": "...",
  "status": "succeeded",
  "created_count": 1,
  "blocked_count": 0,
  "duplicate_count": 0
}
```

核查：

```sql

from agent_task_runs
order by created_at desc
limit 10;

select id, country, city, channel_name, risk_level, review_status, approved_for_extraction, source_url
from lead_source_candidates
```

### 8.2 来源人工复核

移动端 H5 页面：

```text
http://127.0.0.1:5176/#/pages/sources/index
```

复核要求：

- Low/Medium 可按证据和风险规则批准进入抽取。
- High 必须人工复核，确认只做允许范围内的公开文本读取后才可批准。
- Forbidden 必须拒绝，不得进入抽取。
- 任何登录墙、验证码、非公开内容、绕过反爬迹象必须拒绝。

### 8.3 Lead Extraction from Sources

用途：从 `lead_source_candidates` 中选择已批准来源并创建 LEAD_EXTRACTION 任务。

请求：

```bash
curl -sS -X POST http://127.0.0.1:8000/agent-tasks/lead-extraction/from-sources/run \
  -H 'Content-Type: application/json' \
  -d '{
    "limit": 20,
    "country": "Russia",
    "city": "Moscow",
    "trigger_source": "manual_api"
  }'
```

准入规则：

- `review_status` 必须为 `auto_approved` 或 `approved`。
- `approved_for_extraction` 必须为 `true`。
- `extraction_status` 必须为 `pending` 或 `retry`。
- Forbidden 直接阻断。
- High 未人工批准直接阻断。
- 关联渠道计划处于 paused 或 archived 时阻断。

核查：

```sql
select id, task_type, status, output_summary_json, error_message, created_at
from agent_task_runs
where task_type = 'LEAD_EXTRACTION'
order by created_at desc
limit 10;

select id, customer_name, country, city, recommended_grade, queue_status, source_url
from staging_leads
order by created_at desc
limit 20;

select id, task_type, model_name, prompt_version, created_at
from ai_audit_logs
order by created_at desc
limit 20;
```

## 9. APScheduler 定时

当前已实现 `AgentSchedulerService` 的定时任务注册：

| Job ID | 间隔 | 目标 |
|---|---:|---|
| `source_discovery_hourly` | 3600 秒 | 周期性来源发现 |
| `lead_extraction_interval` | 900 秒 | 周期性从已审核来源创建抽取任务 |
| `retry_failed_tasks` | 300 秒 | 技术失败重试和超时恢复 |

调度锁：

- Redis key 前缀：`agent-scheduler-lock`
- 默认 TTL：`300` 秒
- 未获取锁时任务返回 `lock_not_acquired`，避免多进程重复执行。

启用条件：

- `REDIS_URL` 已配置并可 `PING`。
- `/llm-health` 返回 `configuration_complete=true`。
- 已确认调度 job 绑定真实 handler，而不是 placeholder。
- 已有人工复核人和风险暂停机制在线。

启用方式：

```bash
AGENT_SCHEDULER_ENABLED=true
```

重要说明：当前代码中的默认 handler 返回 `not_implemented`。如未在应用启动处注入真实 handler，不得把定时任务作为生产自动运行能力宣称。

## 12. 常见故障

| 现象 | 可能原因 | 处理 |
|---|---|---|
| `/llm-health` 中 `api_key_configured=false` | 未配置 `LLM_API_KEY` | 在 `apps/api/.env` 配置 API key，重启 API |
| SOURCE_DISCOVERY 状态为 `failed` | LLM 配置错误、网络错误、Provider 限流 | 查询 `agent_task_runs.error_message` 和 `output_summary_json` |
| SOURCE_DISCOVERY 返回 422 | 缺少默认 active prompt template | 查询 `llm_prompt_templates`，确认 SOURCE_DISCOVERY 默认模板存在 |
| LEAD_EXTRACTION 返回 422 | 没有 approved 且 pending/retry 的来源 | 先在移动端审核来源候选 |
| High 来源被阻断 | 未人工批准 | 人工复核后再进入抽取；若风险不清晰，保持阻断 |
| Forbidden 来源被阻断 | 合规规则要求 | 不得恢复为自动抽取 |
| 移动端空白 | uni-app 依赖版本不匹配或 H5 插件缺失 | 确认 `@dcloudio/uni-app`、`@dcloudio/uni-h5`、`@dcloudio/vite-plugin-uni` 固定在同一版本 |
| 后台 404 | Vite 未启动在 5174 或访问路径错误 | 执行 `npm --prefix apps/admin run dev`，访问 `/` |
| Redis lock 报错 | `REDIS_URL` 未配置或 Redis 不可达 | 配置 Redis 并执行 `redis-cli ping` |
| migration 无法连接 | `.env` 数据库连接串错误或网络不可达 | 核查 `DATABASE_URL`，确认数据库、账号、端口和白名单 |

## 13. 风险暂停与恢复流程

必须暂停的条件：

- B 级线索比例持续低于 20%。
- 触达几乎无回复。
- 出现投诉、封禁、违规风险或平台警告。
- 销售/客服拒绝承接线索。
- 贸易路径、付款、物流或清关路径不可行。
- 任一自动流程尝试触发社交私信、加好友、登录后批量采集或反爬规避。

暂停动作：

1. 将 `AGENT_SCHEDULER_ENABLED=false`。
2. 暂停相关 `channel_plans`。
3. 将高风险来源保留为待复核或拒绝。
4. 查询并归档 `agent_task_runs`、`lead_source_candidates`、`staging_leads`、`ai_audit_logs`。
5. 在风险事件中记录原因、影响范围、修正动作和负责人。

恢复条件：

- 风险事件完成复盘。
- 合规负责人确认允许恢复。
- LLM health、Redis、PostgreSQL、移动端审核和后台看板全部通过。
- High/Forbidden 阻断规则经抽样验证未被绕过。
- 至少一轮人工小样本验证通过。
