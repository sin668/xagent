## 1. 冻结决策

本计划必须遵守以下已确认决策：

- Prompt 文件全部入库治理，`prompts/` 文件继续保留为开发基线和灾备，不在服务启动时自动覆盖数据库。
- 后台编辑 Prompt 只生成草稿；发布生成新版本；可切换默认版本和回滚。
- Q&A、邮件回复模板、合规话术、车型说明、流程 SOP 复用统一知识库底座。
- 只有 `published/active + embedding_ready + auto_reply_allowed=true` 的知识才可进入自动发送候选。
- 发布后的知识异步生成 pgvector embedding，发布接口不得阻塞等待 embedding 完成。
- 白名单客户、固定 FAQ、首次邮件触达、低风险场景可以进入自动发送候选。
- 其他场景由 AI 生成建议回复，运营/销售人工确认、编辑后发送或标记已发送。
- DNC/勿扰、Watch/Invalid（对外 D/E 级）、语言不确定、知识召回不足、缺少知识证据、价格、付款、合同、发票、税务、法律、交付、制裁、出口管制等场景必须阻断自动发送。
- AI 建议回复和最终发送内容必须分开保存。
- 邮件发送接入真实通道，但必须通过统一 `EmailSender` 适配层，支持 SMTP/SendGrid/Mailgun/企业邮箱扩展。
- `apps/agents` 新增独立 `EMAIL_REPLY` Agent/任务流，只做编排、LLM、节点追踪和结果回传，不直接写业务 core 表。
- `apps/api` 是唯一业务数据权威，所有业务写入、发送提交、规则判断和审计由 `apps/api` 控制。

## 3. 文件结构与职责边界

### 3.1 `apps/api`

建议职责：

- `apps/api/app/models/llm_prompt_template.py`：Prompt 模板治理字段、版本、状态和枚举扩展。
- `apps/api/app/models/knowledge.py`：知识库内容类型、metadata、embedding 状态和质量记录。
- `apps/api/app/models/*email*`：`email_threads`、`email_messages`、`email_reply_drafts`、`email_send_attempts` 等模型。
- `apps/api/app/services/*prompt*`：Prompt 文件解析、hash、入库、草稿、校验、发布、默认版本和回滚。
- `apps/api/app/services/*knowledge*`：Q&A、邮件模板、审核发布、召回过滤和质量记录。
- `apps/api/app/services/*embedding*`：pgvector embedding 任务状态、异步 worker、失败重试和 stale 处理。
- `apps/api/app/services/*email_reply*`：自动发送准入、硬拦截、上下文构建、AI 建议和最终发送审计。
- `apps/api/app/services/*email_sender*`：统一 EmailSender 适配层和发送结果处理。
- `apps/api/app/routers/internal*`：供 `apps/agents` 调用的内部接口，必须校验内部 API Key。
- `apps/api/app/routers/*metrics*`：Prompt、知识、embedding、邮件回复和 Go/No-Go 指标。
- `apps/api/alembic/versions/*`：第五阶段 migration。
- `apps/api/tests/*`：单元、契约、migration、服务间和 e2e 测试。

禁止职责：

- 不让前端直接绕过后端发送邮件。
- 不让 `apps/agents` 直接写业务表。
- 不让 Prompt 草稿直接覆盖线上 active 版本。
- 不让未审核或未 embedding ready 的知识进入自动发送候选。

### 3.2 `apps/agents`

建议职责：

- `apps/agents/app/schemas/*email*`：EMAIL_REPLY 输入、输出和 `schema_version=email-reply-v1`。
- `apps/agents/app/graphs/*email*`：EMAIL_REPLY LangGraph 流程：`load_context -> retrieve_knowledge -> draft_reply -> validate_schema -> request_auto_send_check -> route_decision -> write_summary`。
- `apps/agents/app/routers/*`：`POST /agent-runs/email-reply` 和 `GET /agent-runs/{id}`。
- `apps/agents/app/runtime/*`：run 状态、节点 trace、错误分类和重试。
- `apps/agents/tests/*`：schema、节点、HTTP API、内部接口契约和 dry_run 测试。

禁止职责：

- 不写 `customers`、`contact_methods`、`lead_sources`、`knowledge_items`、`email_reply_drafts`、`outreach_records` 等业务 core 表。
- 不自行绕过 `apps/api` 判断自动发送。
- 不直接调用邮件发送服务商。
- 不暴露公网服务。

### 3.3 `apps/admin`

建议职责：

- Prompt 入库治理页面：对接 `llm_prompt_templates` 真实 API，展示覆盖率、版本、来源 hash、校验、发布和回滚。
- Q&A 与邮件回复知识库页面：对接知识库真实 API，支持内容类型、语言、场景、风险、embedding 状态和召回测试。
- 邮件回复审核台：对接待回复邮件、AI 草稿、知识命中、硬拦截、人工确认发送、拒绝和标记已发送。
- 质量指标页面：展示 Prompt 覆盖率、embedding ready、AI 生成成功率、人工采纳率、自动发送成功率、退信率、DNC/D/E 阻断和风险事件。

禁止职责：

- 不使用 seed 静态数据作为验收依据。
- 不在前端自行决定最终自动发送。
- 不隐藏 DNC/D/E、硬拦截、知识不足和语言不确定原因。

## 4. API、状态与数据契约

### 4.1 Prompt 治理

Prompt 运行时读取规则：

```text
task_type + provider + model
  -> status=ACTIVE
  -> is_default=true
  -> validation_status=passed
```

Prompt 文件迁移规则：

```text
prompts/*.md
  -> parse
  -> hash
  -> source_file_path + source_file_hash + version 幂等入库
  -> hash 变化且已有 active 默认版本时创建 draft
  -> 不覆盖 active
```

### 4.2 知识库召回

自动发送场景的最小过滤条件：

```text
status = published/active
embedding_status = ready
language = reply_language
content_type in allowed_types
business_scene matches
risk_level != blocked
auto_reply_allowed = true
```

召回结果必须记录：

```json
{
  "knowledge_item_id": "uuid",
  "version": "string",
  "similarity_score": 0.88,
  "content_type": "qa_entry",
  "business_scene": "faq_reply",
  "filter_conditions": {}
}
```

### 4.3 EMAIL_REPLY Agent 输出

`EMAIL_REPLY` 输出必须使用：

```json
{
  "schema_version": "email-reply-v1",
  "detected_language": "ru",
  "reply_language": "ru",
  "language_confidence": 0.92,
  "suggested_subject": "string",
  "suggested_body": "string",
  "knowledge_hits": [],
  "risk_flags": [],
  "auto_send_allowed": false,
  "manual_review_required": true,
  "manual_review_reason": "string",
  "next_action": "send_after_review|auto_send|hold|block",
  "audit": {
    "prompt_version": "string",
    "model": "string"
  }
}
```

缺失字段必须输出 `Unknown`、`null` 或空数组，不允许编造。

### 4.4 邮件发送

邮件发送必须经过：

```text
EmailSender.send(message)
  -> send preview/check
  -> DNC/D/E/hard block
  -> eligibility
  -> provider send
  -> email_send_attempts
  -> outreach_records
  -> email_messages
  -> email_reply_drafts
  -> metrics
```

## 10. Go / No-Go 收口指标

第五阶段完成后必须以 P5-E9-S3 和 P5-E9-S4 的真实数据判断：

Go 条件：

- Prompt 入库覆盖率 100%。
- 已发布知识 embedding ready 率 >= 95%。
- AI 回复建议生成成功率 >= 90%。
- 人工采纳率 >= 50%。
- 自动发送硬拦截准确执行率 100%。
- DNC/勿扰客户不自动发送 100%。
- D/E 客户不自动发送 100%。
- 投诉、封禁、违规事件为 0。
- 语言不确定、缺少同语言知识、知识召回不足时不自动发送 100%。

重跑小范围条件：

- 功能跑通但采纳率低。
- 人工修改幅度高。
- embedding ready 不稳定。
- 邮件发送失败率偏高但无风险事件。
- 知识库召回质量不足。

暂停自动发送条件：

- 投诉、封禁、违规事件。
- DNC/勿扰误发。
- D/E 客户误发。
- 硬拦截失效。
- 语言误判外发。
- 缺少知识证据仍自动发送。
- 自动回复承诺价格、库存、付款、合同或交付事项。
