# 海外车辆采购 AI 获客系统总方案与 Codex 推进计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 基于已完成的 BMAD 产物，形成一份符合业务需求、原型设计、技术栈、合规边界和工程规范的完整实施方案，并提供可逐步提交给 Codex 的执行提示词。

**Architecture:** 项目采用“PoC 运营验证先行，MVP 产品化承接”的两阶段架构。PoC 使用飞书表格、AI prompts、人工复核和人工触达验证俄罗斯车商线索闭环；MVP 使用 uni-app(Vue3) 移动端、Vue3 管理后台、FastAPI、PostgreSQL、飞书单向同步、轻量任务队列和大模型 API 实现可审计、可交付、可扩展的 AI 辅助 SDR/CRM 系统。

**Tech Stack:** PoC：飞书表格、飞书文档、CSV、简单脚本、大模型 API。MVP：uni-app(Vue3)、Vue3、FastAPI、PostgreSQL、Feishu API、Celery/RQ/Arq 轻量任务队列、大模型 API、Markdown 文档。

## 1. 方案边界

### 1.1 已确认业务方向

- 第一阶段市场：俄罗斯。
- 主攻客户：当地车商/二级经销商。
- 拓展渠道：个人买家、KOL、汽车博主，仅作为线索拓展来源。
- 车辆品类：二手车、准新车、库存车。
- 产品节奏：2 周 PoC + 4 周 MVP。
- AI 定位：AI 线索研究员 + AI 销售助理，不是自动群发机器人。
- 线索交付规则：AI 先去重、补全、初筛、打标签，再分配给客服或出口销售。

### 1.2 明确不做事项

- 不做自动社交私信。
- 不做自动加好友。
- 不做登录后平台批量采集。
- 不做反爬规避。
- 不把未验证线索直接交付销售。
- 不做复杂黑箱评分模型。
- 不在 MVP 做多国家完整扩展。
- 不在 MVP 做完整 ERP/库存系统深度集成。

## 2. 产品方案

### 2.1 产品定位

本系统是面向海外车辆出口业务的 AI 辅助 SDR/CRM 系统，核心能力包括：

- 公开低/中风险渠道线索发现。
- AI 信息抽取、去重、补全、初筛和打标签。
- A/B/C 线索分级与人工复核。
- 俄语触达草稿和风险提示。
- B 级交付客服，C 级交付出口销售。
- 触达记录、勿扰机制、SLA 和管理仪表盘。
- C 级线索报价/合同前合规复核。

### 2.2 用户与权限

| 用户角色 | 核心工作 | 必备页面 |
|---|---|---|
| 线索运营 | 采集、清洗、去重、补全、复核、分配 | 智能体首页、线索池、线索详情、AI 任务中心 |
| AI/Agent 操作人员 | 运行 AI 抽取、分级、话术任务并处理失败案例 | AI 任务中心、同步与审计 |
| 客服 | 承接 B 级线索、审核话术、人工触达、记录结果 | 线索池、线索详情、触达助手 |
| 出口销售 | 承接 C 级线索、查看需求和车源、推进报价前复核 | 线索详情、车源匹配、触达记录 |
| 管理者 | 看渠道、回复、SLA、ROI 和团队承接 | 后台总览、数据洞察 |
| 合规/风控 | 管理渠道风险、禁止动作、勿扰、C 级复核 | 渠道风险配置、同步与审计 |

### 2.3 页面与原型对齐

MVP 开发必须对齐 `prototypes/mvp-mobile-agent/` 已完成原型。

移动端优先页面：

- `mobile-home.html`：智能体首页。
- `leads.html`：线索池。
- `lead-detail.html`：线索详情。
- `agent-tasks.html`：AI 任务中心。
- `outreach.html`：触达助手。
- `inventory.html`：车源匹配。
- `analytics.html`：数据洞察。
- `settings.html`：设置与合规。

管理后台页面：

- `admin-dashboard.html`：后台总览。
- `admin-risk.html`：渠道风险配置。
- `admin-sync.html`：同步与 AI 审计。

开发标准：

- 移动端使用 uni-app(Vue3)，以移动端工作流为主，不先做桌面 CRM 重页面。
- Web 管理后台使用 Vue3，服务管理者、合规和系统运营。
- 页面信息架构必须保留 AI 推荐理由、来源证据、风险提示和下一步动作。
- 触达相关页面必须显示勿扰、合规检查和人工确认。

## 3. 技术方案

### 3.1 推荐仓库结构

```text
apps/api/                 FastAPI 后端
apps/web/                 Vue3 管理后台
apps/mobile/              uni-app(Vue3) 移动端
docs/poc/                 PoC 运营文档、飞书字段、SOP

scripts/poc/              PoC 校验、去重、指标脚本
tests/                    后端、脚本和契约测试
```

### 3.2 后端模块

- `apps/api/app/main.py`：FastAPI 应用入口和健康检查。
- `apps/api/app/models/`：客户主体、来源、联系方式、触达、车源、渠道风险、AI 审计等模型。
- `apps/api/app/schemas/`：API 输入输出 schema。
- `apps/api/app/services/`：线索分级、去重、勿扰、合规阻断、车源匹配服务。
- `apps/api/app/integrations/feishu/`：飞书单向同步。
- `apps/api/app/integrations/ai/`：大模型调用封装和审计日志。
- `apps/api/app/api/`：REST API 路由。
- `apps/api/app/tasks/`：轻量任务队列任务。

### 3.3 核心数据对象

- Customer：客户主体。
- LeadSource：来源记录。
- ContactMethod：联系方式。
- OutreachRecord：触达记录。
- InventoryItem：车源/报价。
- ChannelRiskRule：渠道风险规则。
- AIAuditLog：AI 审计日志。
- ComplianceReview：C 级线索合规复核。
- SyncLog：飞书同步日志。

### 3.4 状态与等级

线索等级：

- A：基础线索，入库补全。
- B：增强线索，交付客服。
- C：高意向线索，交付出口销售。
- Invalid：无效。
- Watch：沉淀观察。

线索状态：

- 新采集。
- 待补全。
- 待复核。
- 可交付客服。
- 客服跟进中。
- 可交付销售。
- 销售跟进中。
- 无效/暂不匹配。
- 沉淀观察。
- 拒绝联系/勿扰。

SLA：

- B 级首次跟进：48 小时内。
- C 级首次跟进：24 小时内。

## 4. 标准、规范和流程

### 4.1 BMAD 文档规范

每个阶段必须有对应文档：

- Brainstorm：业务假设、问题、决策和待确认项。
- Product Plan：用户、功能、指标和路线图。
- Epic/Story：按业务价值拆解。
- Sprint Plan：按阶段排期。
- Story Files：每个 Story 独立可执行。
- Implementation Plan：工程执行计划。
- Retro：PoC/MVP 复盘。

文档必须中文优先，关键技术名词可保留英文。

### 4.2 Story 执行规范

每个 Story 必须满足：

- 有用户故事。
- 有业务价值。
- 有依赖。
- 有任务清单。
- 有验收标准。
- 有非目标。
- 有 QA/风控检查。

Story 状态建议：

- Draft：已创建，未排期。
- Ready：依赖明确，可进入 Sprint。
- In Progress：执行中。
- Review：待产品/业务/合规验收。
- Done：验收通过。
- Blocked：被外部依赖阻塞。

### 4.3 工程规范

- 先写测试或可验证脚本，再实现功能。
- 后端优先覆盖服务层规则测试。
- AI 输出必须保存输入、输出、模型、时间、来源链接和执行状态。
- High/Forbidden 渠道阻断逻辑必须在后端实现，不能只依赖前端。
- 勿扰状态必须作为客户主体级约束。
- C 级线索未合规复核前不能标记为已报价。
- 所有 API 必须有清晰 schema。
- 配置和密钥不得提交到仓库。

### 4.4 AI 使用规范

AI 可做：

- 公开信息抽取。
- 去重建议。
- 补全任务建议。
- A/B/C/Invalid/Watch 分级建议。
- 客户摘要。
- 俄语触达草稿。
- 风险提示。

AI 不可做：

- 编造缺失字段。
- 直接替代人工复核。
- 自动发送社交私信。
- 自动加好友。
- 绕过平台规则。
- 生成法律意见或贸易合规最终结论。
- 承诺最终价格、物流、清关、付款或交付周期。

### 4.5 合规流程

渠道风险等级：

- Low：官网公开联系方式、公开企业目录、地图标注、公开搜索结果。
- Medium：YouTube、公开 Telegram 频道、公开 VK 页面、公开论坛。
- High：需要登录的平台、私域群、批量私信、加好友、频繁访问、模拟人工操作。
- Forbidden：绕过反爬、盗用账号、购买异常账号、批量骚扰、规避平台限制、抓取非公开数据。

执行规则：

- PoC 只使用 Low/Medium 风险渠道。
- High 渠道仅做政策研究和人工小样本评估。
- Forbidden 行为系统不可执行。
- C 级报价/合同前必须人工合规复核。
- 客户拒绝后必须进入勿扰。

### 4.6 验收流程

每个 Sprint 结束必须检查：

- 是否新增平台风险。
- 是否触碰自动社交私信、加好友、登录平台批量采集边界。
- 是否保留来源证据。
- 是否保留 AI 审计。
- 勿扰客户是否被排除。
- C 级线索是否保留合规复核节点。
- 客服/销售是否能理解当前交付物。

## 7. 推荐执行方式

### 7.1 当前推荐

第一阶段建议使用 Inline Execution 逐步推进 PoC 文档和脚本，因为需要业务负责人频繁确认字段、关键词、话术和风险边界。

进入 MVP 开发后，建议使用 Subagent-Driven：

- 后端数据底座一个 agent。
- 移动端工作台一个 agent。
- 管理后台一个 agent。
- QA/审计一个 agent。

### 7.2 每次提交给 Codex 的工作方式

每次只提交一个 Prompt，并要求 Codex：

- 读取对应 Story。
- 先列实施计划。
- 再编辑文件或代码。
- 最后输出验收结果和未完成风险。
- 不跨阶段做未授权扩展。

### 7.3 Go/No-Go 门

PoC 不通过，不进入 MVP 开发。

进入 MVP 的最低条件：

- 至少 3 个可持续低/中风险渠道。
- 100-200 条线索中 30%-50% 达到 B 级。
- 小样本人工触达有真实回复或明确负反馈。
- 无账号封禁、平台投诉、明显违规采集或批量骚扰。
- 客服/销售愿意承接。

## 8. 当前下一步

建议下一条提交给 Codex 的提示词使用 **Prompt 1：PoC 飞书五张表与字段字典**。

原因：

- 它是 PoC 的最小执行底座。
- 不依赖真实代码环境。
- 能快速把业务字段、同步字段和后续 CRM 数据模型对齐。
- 可以尽早暴露字段缺失、角色分工和合规边界问题。
