**Architecture:** 后端继续以 FastAPI + SQLAlchemy + Alembic + PostgreSQL 为事实来源，Redis/现有调度和 `agent_task_runs` 继续作为任务控制与审计基础。第三阶段新增 `apps/agents` 或 `apps/agent-runtime` 作为独立 Agent 项目，局部试点 LangGraph，仅承载 Deep Enrichment Agent 和 Lead Cleanup Agent。移动端以 uni-app(Vue3) 承载线索完善、清洗建议、客户工作台、客户详情和跟进记录。管理后台以 Vue3/Vite 承载第三阶段指标、风控和运行看板。

**Tech Stack:** Python, FastAPI, SQLAlchemy, Alembic, PostgreSQL, Redis, APScheduler, DeepSeek/OpenAI-compatible LLM client, LangGraph, uni-app Vue3, Vue3/Vite, Node v22.22.0.

## 2. 强制业务与合规边界

第三阶段所有 Story 都必须遵守：

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- High 来源未审核不得进入自动触达。
- Forbidden 来源不得作为客户晋级关键来源。
- Invalid 和 Watch 不进入触达队列。
- 勿扰客户不得再次进入触达队列。
- C 级客户报价、合同、付款、物流、清关、交付周期等动作前必须合规复核。
- LLM 缺失字段必须输出 `Unknown`、`null` 或空数组，不得编造。
- 所有 AI 输出必须保存来源证据、prompt 版本、模型和审计记录。
- Agent 不得自动晋级客户、自动归并客户、自动恢复 Invalid、自动触达客户。
- 手动补录、字段采纳、客户晋级、清洗恢复、重复归并都必须审计。

## 3. 第三阶段范围边界

### 3.1 本阶段做

- 线索完善区。
- AI 深挖补全人工触发。
- 人工补录。
- 字段级采纳、修改、拒绝。
- 手动准入客户。
- `staging_leads -> customers / lead_sources / contact_methods` 晋级闭环。
- 客户工作台、客户详情、跟进记录。
- 客户意向车型独立建模。
- Watch/Invalid 清洗建议和人工确认执行。
- 新建独立 Agent 项目局部试点 LangGraph。
- 有效客户承接率、补全成功率、清洗采纳率、风险事件等指标。
- 最小角色权限和合规硬阻断。

### 3.2 本阶段不做

- 不做完整 CRM 商机阶段管理。
- 不做报价单、合同、收款、物流履约。
- 不做自动邮件、短信、社媒发送。
- 不做自动加好友或自动私信。
- 不做销售自动化漏斗全套系统。
- 不做多国家大规模推广。
- 不对 High 风险来源做自动触达。
- 不引入完整复杂 RBAC。
- 不重构现有 Source Discovery Agent 和 Lead Extraction Agent。

## 8. 标准、规范和流程

### 8.1 BMAD 流程标准

每个 Story 必须按以下流程执行：

1. 读取本推进计划、第三阶段方案、当前 Story 和相关代码。
2. 明确当前 Story 的目标、非目标、验收标准和风控边界。
3. 代码 Story 先写失败测试，再做最小实现，再运行测试。
4. 涉及异常排查时，先复现、定位根因、形成假设，再修复。
5. 完成后执行对应测试、关联测试和必要联调。
6. 回写 Story 状态和执行记录。
7. 写入 `_bmad-output/implementation-artifacts/codex-【story-id】-执行结果.md`。
8. 执行两轮独立评审并记录结论、发现项和修正结果。
9. 两轮评审均无新增实质阻塞问题后，当前 Story 才视为完成。
10. 不自动执行下一个 Story。

### 8.2 技术标准

后端：

- 使用 `conda activate booking-room`。
- 使用 FastAPI + SQLAlchemy + Alembic + PostgreSQL。
- Service 层执行业务规则和合规硬门禁，前端按钮禁用只能作为体验优化。
- 数据写入必须保持 raw / staging / core / audit 分层。
- AI、Agent、人工采纳、客户晋级、清洗执行不得直接绕过 Service 层写 core 表。
- 新增 migration 必须可被 Alembic 发现，模型必须在 `app/models/__init__.py` 或既有注册机制中导入。

前端：

- 使用 `nvm use v22.22.0`。
- 移动端使用 uni-app(Vue3)，不得只保留 seed 静态数据作为验收。
- 管理后台使用 Vue3/Vite。
- 页面应参考 `prototypes/mvp-mobile-agent` 已设计原型，保持移动端优先、信息密度可控、底部导航不遮挡核心操作。
- 涉及 API 的页面必须处理 loading、empty、error、权限不足和合规阻断提示。

Agent：

- 第三阶段新增独立 Agent 项目局部试点 LangGraph。
- Agent 只生成候选、建议、证据和结构化输出。
- Agent 不得自动晋级客户、自动归并客户、自动恢复 Invalid、自动触达客户。
- Agent 输出必须由 `apps/api` Service 层校验后写入 PostgreSQL。
- LLM 缺失字段必须输出 `Unknown`、`null` 或空数组，不得编造。

数据库：

- PostgreSQL 是正式验收事实来源。
- 当前沙箱网络或系统权限导致真实 PostgreSQL 连接失败时，必须记录实际错误和待外部环境复跑项。
- 允许用单元测试和 mock session 验证业务逻辑，但不得把 SQLite 内存测试标记为正式数据库验收。

### 8.3 合规与安全标准

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- Forbidden 来源不得作为客户晋级关键来源。
- High 风险来源只允许在人工审核后做只读研究，不得自动触达。
- Watch/Invalid 不进入触达队列。
- 勿扰客户不得再次进入触达队列。
- C 级客户在报价、合同、付款、物流、清关、交付周期等动作前必须合规复核。
- 手动补录、字段采纳、客户晋级、清洗恢复、重复归并、勿扰变更必须审计。

### 8.4 两轮独立评审标准

每个 Story 的执行结果文件必须包含两轮评审：

第一轮：需求一致性与业务边界评审。

- 是否覆盖当前 Story 验收标准。
- 是否符合第三阶段方案和原型设计。
- 是否保持每次只执行一个 Story。
- 是否有超出非目标范围的实现。
- 是否违反触达、采集、勿扰、Watch/Invalid、C 级合规复核等边界。

第二轮：技术落地、测试和回归风险评审。

- 是否有测试先行记录。
- 是否运行当前测试和关联测试。
- 是否记录真实 PostgreSQL / 前后端联调结果。
- 是否存在 schema、枚举、路由、异步 session、JSON 字段、字符串长度、CORS、OPTIONS 等已知高发风险。
- 是否有新增实质阻塞问题；如果有，必须修复后再复核。

## 9. 验收门禁

每个 Story 完成前必须满足：

- 当前 Story 指定测试通过。
- 与当前 Story 强相关的回归测试通过。
- `python -m compileall` 或对应前端构建/类型检查通过。
- 涉及 migration 的 Story 必须尝试真实 PostgreSQL migration 或连接验证，并记录结果。
- 涉及移动端或后台的 Story 必须完成真实 API 联调，无法联调时记录阻断原因、复跑命令和待复跑环境。
- 涉及 Agent 的 Story 必须验证 Agent 只写候选或建议，不自动执行人工决策动作。
- 涉及客户晋级的 Story 必须验证最低准入、勿扰、Forbidden、来源证据和审计。
- 涉及 C 级客户销售动作的 Story 必须验证报价、合同、付款、物流、清关、交付周期前合规复核阻断。
- Story 文件状态必须更新为“实现完成”或“实现完成，真实 PostgreSQL/联调待复跑”等准确状态。
- 执行结果必须写入 `_bmad-output/implementation-artifacts/codex-【story-id】-执行结果.md`。
- 两轮独立评审必须连续无新增实质阻塞问题。
