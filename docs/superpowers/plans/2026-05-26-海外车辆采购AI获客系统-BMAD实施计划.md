# 海外车辆采购客户 AI 获客系统 BMAD 实施计划

创建日期：2026-05-26

## 1. 实施目标

用 **2 周 PoC + 4 周 MVP** 的方式，验证并产品化俄罗斯车辆采购客户 AI 获客闭环。

核心目标不是“抓取更多数据”，而是稳定产出可交付、可跟进、可复盘的 B/C 级客户线索。

## 2. 总体路线

| 阶段 | 周期 | 目标 | 核心产出 |
|---|---:|---|---|
| PoC | 2 周 | 验证俄罗斯车商线索是否可持续获取、清洗、人工触达和交付 | 飞书五张表、100-200 条线索、AI 清洗流程、小样本触达、复盘报告 |
| MVP | 4 周 | 将验证有效的流程产品化为轻量 CRM | Vue3/uni-app/FastAPI/PostgreSQL CRM、飞书同步、AI 建议、SLA、仪表盘、合规配置 |

进入 MVP 的前提：PoC 复盘结论为 **继续进入 MVP**。

## 3. 角色分工

| 角色 | 职责 | PoC 每日建议投入 |
|---|---|---:|
| 业务负责人 | 确认目标市场、客户画像、车源品类、PoC 是否继续 | 30 分钟 |
| 线索运营负责人 | 管理飞书表格、去重、补全、复核、分配 | 2 小时 |
| AI/Agent 操作负责人 | 运行搜索、抽取、清洗、话术生成流程，记录日志 | 2 小时 |
| 客服负责人 | 承接 B 级线索，小样本人工触达，反馈话术效果 | 1 小时 |
| 出口销售负责人 | 承接 C 级线索，判断需求、车型匹配、报价可行性 | 1 小时 |
| 合规/风控负责人 | 审核平台风险、话术边界、贸易红线、勿扰机制 | 30 分钟 |

## 4. PoC 实施计划

### 第 0 天：启动准备

- [ ] 确认 6 类角色负责人和备用人。
- [ ] 确认每日人力投入。
- [ ] 确认 PoC 成本上限：时间、人力、工具/API、渠道试错。
- [ ] 确认第一批轻量车源/报价样例。
- [ ] 确认俄罗斯贸易合规红线初版。

验收：负责人、投入、成本上限、车源样例和合规红线都有明确记录。

### 第 1-2 天：搭建 PoC 数据底座

- [ ] 创建飞书五张表：客户线索、渠道来源、车源/报价、触达记录、话术库。
- [ ] 建立字段规则：A/B/C 级线索、线索状态、联系方式、来源证据、AI 建议、负责人。
- [ ] 创建渠道风险登记表。
- [ ] 创建关键词库初版。
- [ ] 创建客户 FAQ/异议处理库初版。

验收：飞书表格可录入、可筛选、可分配；渠道和关键词可开始执行。

### 第 3-5 天：首批 50 条线索试采

- [ ] 只使用低/中风险渠道：搜索、官网、公开目录、公开地图结果、中国出口同行公开渠道。
- [ ] 每条线索记录客户名称、国家/城市、客户类型、来源链接、联系方式、证据备注。
- [ ] AI 辅助抽取、去重、补全、分类。
- [ ] 人工复核 AI 判断。
- [ ] 记录无效线索和失败分类案例。

验收：完成首批 50 条候选线索，其中至少能形成一批 A/B 级线索；发现明显无效渠道时及时暂停。

### 第 6-9 天：扩展到 100-200 条线索

- [ ] 只扩大前 50 条试采中表现较好的渠道。
- [ ] 某渠道采集 30-50 条仍没有 B 级线索时暂停或调整。
- [ ] 每日导出线索等级、渠道、状态和无效率统计。
- [ ] 沉淀有效关键词和排除关键词。
- [ ] 更新失败案例库。

验收：累计 100-200 条候选线索，B 级增强线索比例目标为 30%-50%。

### 第 10-12 天：人工触达小样本

- [ ] 从 B/C 级线索中选 10-20 条做人工触达。
- [ ] 只允许人工邮件、官网表单，或经合规负责人批准的公开联系方式人工触达。
- [ ] AI 生成俄语话术草稿。
- [ ] 人工审核后发送。
- [ ] 记录触达渠道、话术版本、发送人、发送时间、回复状态、下一步动作。
- [ ] 客户拒绝后立即标记“拒绝联系/勿扰”。

验收：产生真实回复或明确负反馈；没有平台投诉、账号限制或违规触达。

### 第 13-14 天：PoC 复盘

- [ ] 汇总渠道有效率。
- [ ] 汇总 A/B/C 线索比例。
- [ ] 汇总触达回复和负反馈。
- [ ] 汇总平台风险和合规问题。
- [ ] 汇总客服/销售承接反馈。
- [ ] 计算每条有效线索成本和每条回复成本。
- [ ] 输出 PoC 复盘报告。

验收：明确选择三种结论之一：

1. 继续进入 4 周 MVP。
2. 调整渠道/画像后重复 PoC。
3. 暂停俄罗斯方向并重评市场。

## 5. PoC 验收标准

| 验收项 | 通过标准 |
|---|---|
| 渠道可行性 | 至少 3 个可持续低/中风险渠道 |
| 线索质量 | 100-200 条线索中 30%-50% 达到 B 级 |
| AI 效率 | 能稳定完成去重、补全、分类、摘要、话术建议 |
| 触达反馈 | 小样本有真实回复或明确负反馈 |
| 风险控制 | 无账号封禁、平台投诉、明显违规采集或批量骚扰 |
| 团队承接 | 客服/销售认为字段、话术和交付方式可用 |

## 6. MVP 实施计划

MVP 只有在 PoC 通过后启动。

### 第 1 周：后端与数据底座

- [ ] 搭建 FastAPI 项目。
- [ ] 搭建 PostgreSQL 数据库。
- [ ] 建立核心模型：客户主体、来源记录、联系方式、触达记录、AI 审计日志、合规配置。
- [ ] 实现 A/B/C 线索规则。
- [ ] 实现飞书 API 单向同步映射。
- [ ] 实现基础健康检查和后端测试。

验收：飞书数据能进入 PostgreSQL；线索规则测试通过；勿扰状态可以保留。

### 第 2 周：CRM Web 看板

- [ ] 实现线索列表和状态看板。
- [ ] 实现线索详情页。
- [ ] 实现 AI 建议面板。
- [ ] 实现合规风险标签。
- [ ] 实现负责人和团队分配。
- [ ] 实现 B/C 级 SLA 显示。

验收：运营、客服、销售能分别看到自己需要处理的线索。

### 第 3 周：业务闭环与移动端

- [ ] 实现触达记录。
- [ ] 实现车源/报价表与线索匹配。
- [ ] 实现 uni-app 移动端线索列表。
- [ ] 实现移动端线索详情。
- [ ] 实现勿扰状态和拒绝联系流程。
- [ ] 实现 C 级线索合规复核节点。

验收：客服可处理 B 级线索，销售可处理 C 级线索，拒绝联系客户不会再进入触达任务。

### 第 4 周：仪表盘、合规配置与上线 QA

- [ ] 实现渠道指标：采集数、B/C 级线索数、无效率。
- [ ] 实现触达指标：已触达数、回复数、回复率。
- [ ] 实现 SLA 指标：超时数、待处理队列。
- [ ] 实现业务机会指标：询价数、车型需求数、销售推进数。
- [ ] 实现渠道风险等级、允许动作、禁止动作配置。
- [ ] 完成端到端 QA。
- [ ] 输出 MVP 试运行报告。

验收：管理者能看到渠道、触达、SLA、业务机会四组指标；系统通过 QA 清单。

## 8. 风险控制

- 平台风险：只从公开低/中风险渠道启动，高风险渠道先做政策研究。
- 账号风险：PoC 不登录平台，不使用公司官方账号做自动化实验。
- 数据风险：最小必要保存，联系方式和沟通记录做权限控制。
- 话术风险：AI 只能基于审核模板和 FAQ 生成草稿，人工确认后发送。
- 贸易风险：C 级线索报价/合同前必须人工合规复核。
- 承接风险：A 级只入库补全，B 级给客服，C 级给出口销售。

## 9. 暂停或转向条件

- 100-200 条线索中，B 级线索低于 20%。
- 小样本触达几乎无人回复，或多数明确拒绝。
- 出现账号限制、投诉、访问异常或明显违反条款风险。
- 销售团队不愿承接。
- 客户有兴趣但贸易、支付、物流或清关不可行。

## 10. 下一步执行选择

建议先执行 **PoC 阶段任务 0-9**，暂不启动 MVP 开发。

执行方式有两种：

1. Subagent-Driven：每个任务分配独立 agent 执行，我负责整合和复核。
2. Inline Execution：在当前会话中按任务顺序执行，适合小步推进和人工确认。

推荐选择：**Inline Execution 先跑 PoC 准备任务，等真实数据开始采集后再考虑多 agent 并行。**
