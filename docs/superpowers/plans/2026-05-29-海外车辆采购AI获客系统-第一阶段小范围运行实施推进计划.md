# 海外车辆采购 AI 获客系统第一阶段小范围运行实施推进计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 基于已完成的 BMAD 产物、移动端优先原型和既定技术栈，推进第一阶段小范围运行，实现“公开线索发现 -> AI 抽取分级 -> PostgreSQL 入库 -> staging 人工复核 -> core 交付 -> 指标治理”的可审计闭环。

**Architecture:** 系统采用 raw / staging / core / audit / knowledge 五层数据架构，Low/Medium 渠道进入受控自动发现，High 渠道仅做只读公开发现并要求二次复核。Agent 只负责发现、读取、抽取、分级和入库建议，不做自动私信、自动加好友、登录后批量采集或反爬规避；所有关键准入规则由后端强制执行，前端只做展示和操作入口。

**Tech Stack:** 后端 FastAPI + SQLAlchemy + Alembic + PostgreSQL + Redis；知识库 PostgreSQL + pgvector；移动端 uni-app(Vue3)；管理后台 Vue3 + Vite；AI 接入采用可替换 LLM service + prompt version + ai_audit_logs；验证使用真实 PostgreSQL/Redis 环境，不使用内存 SQLite 作为后续正式测试环境。

## 2. 第一阶段业务规则

必须遵守的硬边界：

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- 不让 AI 编造客户名称、联系方式、意向、经营状态或来源证据。
- 缺失字段必须输出 `Unknown`、`null` 或空数组。
- High 风险渠道只做只读公开发现，不得直接进入 core 或触达队列。
- High 来源必须通过 Low/Medium 来源完成二次复核后，才可进入正式交付链路。
- Invalid 和 Watch 不进入触达队列。
- 勿扰客户不得再次进入触达队列。
- C 级线索报价或合同前必须合规复核。
- 所有 AI 输出必须保存来源证据、输入引用、输出 JSON、模型、prompt 版本和校验结果。

第一阶段目标：

- 每日约 100 条候选线索。
- 每日目标 20-50 条 core 有效线索。
- A/B/C 渠道并行拓展更多来源。
- PostgreSQL 直入库，不再使用飞书表格作为主数据承载。
- 后台和移动端对齐 `prototypes/mvp-mobile-agent/` 的信息架构。

## 3. 产品落地范围

### 3.1 移动端优先能力

移动端服务线索运营、客服和销售：

- 首页：展示今日候选、staging、core、风险事件和关键待办。
- 线索池：展示 staging、B/C 线索、High 二次复核、Watch/Invalid 状态。
- 线索详情：展示来源证据、AI 推荐原因、准入闸门、人工复核动作。
- Agent 任务中心：展示发现、读取、抽取、分级、入库任务状态。
- 触达助手：只提供人工审核后的话术草稿、勿扰提示、拒绝联系路径和触达记录，不提供自动发送。
- 车源匹配：支持业务人员查看车辆匹配信号。
- 数据洞察：查看有效线索、渠道质量、风险和复核效率。
- 设置与合规：展示风险规则、知识库版本和操作边界。

### 3.2 管理后台能力

后台服务管理者、运营和合规：

- 渠道计划：配置国家、城市、渠道、关键词、风险等级、配额和负责人。
- 风险规则：配置允许动作、禁止动作、High 只读边界和暂停规则。
- staging 复核：查看候选线索、证据、AI 分级、重复建议和晋级操作。
- AI 审计：查看 LLM 输入、输出、prompt 版本、知识引用和校验结果。
- 知识库：管理 SOP、FAQ、关键词、话术、车辆知识、合规规则和失败案例。
- 指标看板：展示候选漏斗、渠道有效率、B/C 比例、风险事件、AI 成本和人工复核效率。

### 4.2 Story 锁命令

推荐 owner 命名：

```bash
session_name="codex-p1-e1-s1-$(date +%Y%m%d%H%M)"
```

获取 Story 锁：

```bash
source ~/.zshrc >/dev/null 2>&1 || true
nvm use v22.22.0
node scripts/story-lock.js acquire --story <story_key> --owner <session_name> --add _bmad-output/implementation-artifacts/<session_name>-执行结果.md
```

续租 Story 锁：

```bash
source ~/.zshrc >/dev/null 2>&1 || true
nvm use v22.22.0
node scripts/story-lock.js renew --story <story_key> --owner <session_name>
```

释放 Story 锁：

```bash
source ~/.zshrc >/dev/null 2>&1 || true
nvm use v22.22.0
node scripts/story-lock.js release --story <story_key> --owner <session_name>
```

写入 `sprint-status.yaml` 或共享核查表时，必须先持有覆盖该路径的 Story 锁，再短暂获取共享短锁，写完立即释放：

```bash
source ~/.zshrc >/dev/null 2>&1 || true
nvm use v22.22.0
node scripts/story-lock.js path-acquire --path _bmad-output/implementation-artifacts/sprint-status.yaml --owner <session_name> --ttl-seconds 30
node scripts/story-lock.js path-release --path _bmad-output/implementation-artifacts/sprint-status.yaml --owner <session_name>
```

- 页面必须对齐后台原型的信息架构。

AI / Agent Story：

- Prompt 输出必须通过 JSON schema 校验。
- schema 校验失败不得写入 staging。

## 5. Sprint 执行路线

### Sprint 1：数据底座与渠道规则

目标：建立 PostgreSQL raw/staging/core/audit/knowledge 基线，支持渠道计划和动作校验。

Story 顺序：

1. `P1-E1-S1-data-layer-migration-baseline.md`
2. `P1-E1-S2-raw-collection-task-candidate-url.md`
3. `P1-E1-S3-page-snapshots-source-evidence.md`
4. `P1-E1-S4-staging-leads.md`
5. `P1-E1-S5-audit-risk-logs.md`
6. `P1-E2-S1-channel-plans.md`
7. `P1-E2-S2-channel-action-policy-validator.md`
8. `P1-E2-S3-high-risk-public-discovery-isolation.md`

Sprint 1 完成标准：

- Alembic migration 可在真实 PostgreSQL 执行。
- `collection_tasks`、`candidate_urls`、`page_snapshots`、`staging_leads`、audit/risk 基础日志可用。
- `channel_plans` 可配置并校验风险等级。
- High 默认 `public_discovery_only` 且 `queue_eligible=false`。

### Sprint 2：staging 复核闭环

目标：支持 staging 列表、详情、晋级 core、重复检测和渠道暂停。

Story 顺序：

1. `P1-E2-S4-channel-pause-risk-events.md`
2. `P1-E3-S1-staging-review-list-filters.md`
3. `P1-E3-S2-staging-lead-detail-evidence.md`
4. `P1-E3-S3-promote-staging-to-core.md`
5. `P1-E3-S4-dedupe-merge-suggestions.md`

Sprint 2 完成标准：

- 无来源链接、无证据、High 未二次复核、Invalid/Watch、勿扰不得进入待触达。
- 晋级 core 可写入客户、联系方式、来源和 review logs。
- 风险事件可暂停渠道。
- 复核页面与移动端/后台原型中的线索池、详情和风险提示一致。

### Sprint 3：Agent 任务流与知识库

目标：实现受控发现、公开页面读取、LLM 抽取分级、失败案例、pgvector RAG。

Story 顺序：

1. `P1-E4-S1-channel-discovery-agent.md`
2. `P1-E4-S2-public-page-read-agent.md`
3. `P1-E4-S3-llm-lead-extraction.md`
4. `P1-E4-S4-llm-grading-rule-validation.md`
5. `P1-E4-S5-agent-failed-case-library.md`
6. `P1-E5-S1-pgvector-knowledge-schema.md`
7. `P1-E5-S2-import-phase-one-knowledge.md`
8. `P1-E5-S3-knowledge-search-api.md`
9. `P1-E5-S4-rag-in-llm-prompts.md`

Sprint 3 完成标准：

- Agent 不登录、不绕过、不自动触达。
- LLM 输出 schema 校验失败不得写入 staging。
- RAG 只检索 approved 知识。
- RAG 引用写入 `ai_audit_logs`。
- 失败案例可被后续分级和质量改进使用。

### Sprint 4：指标看板与治理

目标：实现每日漏斗、渠道质量、风险看板、LLM 成本和复核效率。

Story 顺序：

1. `P1-E6-S1-daily-funnel-dashboard.md`
2. `P1-E6-S2-channel-quality-metrics.md`
3. `P1-E6-S3-risk-event-dashboard.md`
4. `P1-E6-S4-llm-cost-review-efficiency.md`

Sprint 4 完成标准：

- High 只读结果不得计入可触达有效线索。
- 风险事件不可物理删除。
- 每条 core 有效线索 AI 成本可统计。
- 管理后台可查看渠道产出、B/C 比例、风险事件、复核效率和 ROI 信号。

## 9. 阶段门禁

### 9.1 Sprint 进入门禁

进入每个 Sprint 前确认：

- 上一 Sprint 的 P0 Story 已 Done。
- 未释放的 Story 锁为 0。
- 真实 PostgreSQL 和 Redis 可连接。
- 当前 Sprint 依赖的 migration 已执行。
- 原型或产品文档未发生未冻结变更。

### 9.2 Story 完成门禁

单个 Story 完成必须满足：

- Story 文件状态已更新。
- 验收标准逐条记录。
- 风控检查逐条记录。
- 相关测试已运行并记录结果。
- 失败或跳过的测试说明原因。
- 涉及 AI 输出的 Story 已保存审计。
- 涉及触达或准入的 Story 已验证勿扰、Invalid/Watch、High、C 级合规复核规则。
- Story 锁已释放。

### 9.3 阶段 Go/No-Go 门禁

第一阶段小范围运行进入持续运行前，必须满足：

- 每日可稳定生成约 100 条候选线索。
- 每日可稳定沉淀 20-50 条 core 有效线索，或有明确渠道优化路径。
- High 误入触达队列次数为 0。
- 勿扰误触达次数为 0。
- AI 严重编造事件为 0。
- 投诉、封禁、违规风险为 0。
- 客服和出口销售愿意承接 B/C 线索。
- 指标看板可支持每日复盘。

## 10. 暂停条件

任一情况出现，应暂停对应渠道、任务或阶段推进：

- 平台出现验证码、警告、访问异常、账号限制。
- High 渠道越界进入自动任务或触达队列。
- 出现投诉或疑似骚扰。
- 勿扰客户进入触达队列。
- AI 输出缺少来源证据或疑似编造。
- 销售/客服反馈线索完全不可承接。
- 真实 PostgreSQL migration 存在数据破坏风险。
- Story 锁冲突或共享文档并发写入风险未解除。

## 11. 推荐推进节奏

- 第 1-2 天：完成 Sprint 1 数据底座前 5 个 Story。
- 第 3-4 天：完成 Sprint 1 渠道计划与风险规则 3 个 Story。
- 第 5-7 天：完成 Sprint 2 staging 复核闭环。
- 第 8-12 天：完成 Sprint 3 Agent 任务流和知识库。
- 第 13-15 天：完成 Sprint 4 指标看板与治理。
- 第 16-20 天：小范围运行、复盘、修正规则、决定 Go/重跑/暂停。

节奏可根据真实数据库、AI API、前端联调和风控复核情况调整，但不得跳过数据底座、风险规则和 staging 复核闭环。

## 12. 执行记录要求

每个 Story 建议输出：

- `_bmad-output/implementation-artifacts/<session_name>-执行结果.md`

执行结果必须包含：

- Story 名称与文件路径。
- owner/session 名称。
- 开始和完成时间。
- 修改文件清单。
- 测试命令与结果。
- 验收标准逐条结论。
- 风控检查逐条结论。
- 两轮评审记录，若适用。
- 残留风险。
- 下一 Story 的接力提示。

## 13. 本计划维护规则

- 新增 Story 时，先更新 Epic/Story 拆解和 Sprint 规划，再更新本计划。
- 修改业务边界时，先进入 BMAD correct-course 或新一轮 brainstorm，不直接改代码。
- 修改 UI/UX 页面结构时，先更新 `prototypes/mvp-mobile-agent/` 和设计说明，再进入 Story。
- 修改风控规则时，同步更新后端规则、测试、知识库和文档。
- 修改 AI prompt 时，同步更新 prompt version、schema、审计字段和失败案例。
