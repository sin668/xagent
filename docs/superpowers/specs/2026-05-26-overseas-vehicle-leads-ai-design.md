# 海外车辆采购客户 AI 获客系统设计规格

创建日期：2026-05-26

## 1. 背景与目标

本项目面向海外车辆采购客户，第一阶段聚焦俄罗斯市场，通过 AI 与 Agent 工具辅助完成客户线索发现、信息补全、去重、初筛、打标签、话术生成、CRM 登记和线索交付。

第一阶段不追求最大化抓取数量，而追求稳定产出可交付、可跟进、可沉淀的有效线索。

## 2. 第一阶段战略选择

- 市场：俄罗斯。
- 主攻客户：当地车商/二级经销商。
- 拓展渠道：个人买家、KOL、汽车博主引流。
- 主打车辆品类：二手车、准新车、库存车。
- 成功指标：有效线索为主，触达回复为辅，成交机会作为观察指标。
- AI 定位：MVP 阶段为 AI 线索研究员 + AI 销售助理，长期目标为 AI + 人工混合 SDR 系统。
- 实施节奏：2 周 PoC + 4 周 MVP。

## 3. 第一阶段不做事项

- 不做大规模自动私信或自动加好友。
- 不承诺全平台覆盖。
- 不做复杂黑箱意向评分模型，先采用规则标签和人工复核。
- 不直接把未验证线索推给销售团队。
- 不绕过平台规则做规避式自动化。
- 不做自动社交触达、复杂 BI、多国家完整扩展。

## 4. 业务流程

AI 先负责线索去重、信息补全、初筛和打标签，再分配给客服或出口销售。

- A 级基础线索：客户名称、国家/城市、来源链接、客户类型、至少一个联系方式。先入库补全，不急于交付。
- B 级增强线索：在 A 级基础上增加主营车型、是否卖二手/进口车、社交/官网活跃度、初步匹配原因。交付客服团队。
- C 级高意向线索：在 B 级基础上增加已回复、表达采购兴趣、询价、愿意添加微信/WhatsApp/Telegram 或提供需求。交付出口销售团队。

客服处理低/中意向线索，出口销售处理高意向线索，线下/当地合作团队作为后续能力接入。

## 5. 联系与触达策略

第一阶段不把注册微信作为硬性指标，而把可持续联系作为硬性指标。

支持 WhatsApp、Telegram、Email、电话、VK、微信等多种联系方式。对 C 级高意向客户再引导微信，便于深入报价、发送车源和成交跟进。

触达自动化边界：

- 2 周 PoC：完全人工发送，AI 只生成草稿和建议。
- 4 周 MVP：可尝试人工确认后的半自动发送，但仅限邮件或官网表单等低风险渠道。
- 社交平台私信、加好友、入群互动不纳入第一阶段自动化范围。

## 6. 产品形态

PoC 阶段采用表格 + Agent 工作流。MVP 阶段升级为网页看板 + 表格同步。

PoC 至少建立 5 张表：

- 客户线索表。
- 渠道来源表。
- 车源/报价表。
- 触达记录表。
- 话术库表。

MVP 核心能力：

- 轻量 CRM Web 看板。
- 线索列表、状态流转、分配、筛选和详情页。
- AI 建议面板。
- Agent/AI 执行日志。
- 车源/报价表与线索匹配。
- 触达记录与 SLA。
- 轻量仪表盘。
- 安全合规配置。

## 7. 线索状态流转

- 新采集。
- 待补全。
- 待复核。
- 可交付客服。
- 客服跟进中。
- 可交付销售。
- 销售跟进中。
- 无效/暂不匹配。
- 沉淀观察。
- 拒绝联系/勿扰。

B 级线索需在 48 小时内首次跟进，C 级线索需在 24 小时内首次跟进。

## 8. AI 产品设计

AI 在产品中以可解释助手呈现，而不是黑箱自动代理。

优先能力：

- AI 建议面板：推荐等级、推荐原因、缺失信息、下一步建议、推荐话术、风险提示。
- AI/Agent 证据日志：搜索、来源、证据、输入、输出、模型、执行时间、执行结果。
- AI 批处理能力：去重、补全、俄语触达话术、分级，但所有结果进入人工复核。

AI 生成客户回复时必须优先引用经过审核的 FAQ/异议处理库，不允许自由承诺价格、物流、清关或付款条件。

## 9. 技术栈

PoC：

- 飞书表格。
- CSV 导入导出。
- 简单脚本。
- 大模型 API。
- 公开网页/搜索采集。

MVP：

- 移动端：uni-app + Vue3。
- Web 前端：Vue3。
- 后端：Python + FastAPI。
- 数据库：PostgreSQL。
- 同步：飞书 API 单向同步到 PostgreSQL。
- 任务执行：轻量任务队列，例如 Celery、RQ 或 Arq。
- AI：大模型 API + 轻量 Agent 编排。

采集技术优先级：

1. 搜索引擎 + 公开网页抓取。
2. 平台官方 API。
3. 浏览器辅助采集。

第一阶段明确不做模拟登录、批量操作、反爬规避或绕过平台限制。

## 10. 安全与合规

总原则：

- 平台规则优先。
- 数据保护优先。
- 贸易与制裁合规前置。

每个渠道需配置风险等级、允许动作和禁止动作。Agent 执行前必须读取渠道规则，不允许自行越界。

平台风险分级：

- 低风险：官网公开联系方式、公开企业目录、地图标注、公开搜索结果。
- 中风险：YouTube、公开 Telegram 频道、公开 VK 页面、公开论坛。
- 高风险：需要登录的平台、私域群、批量私信、加好友、频繁访问、模拟人工操作。
- 禁止行为：绕过反爬、盗用账号、购买异常账号、批量骚扰、规避平台限制、抓取非公开数据。

俄罗斯贸易与制裁风险：

- 建立贸易合规红线清单。
- C 级高意向线索在报价、合同或实质交易推进前必须进入人工合规复核。
- AI 只做风险提示，不替代法务、贸易合规或业务负责人判断。

## 11. 平台政策初步核查

初步官方资料核查显示，YouTube、X、Meta、Telegram、WhatsApp、Google Maps 等平台均不适合第一阶段做未经授权的自动化抓取或批量触达。PoC 应继续坚持低风险路径：公开搜索、官网、地图结果人工核验、官方 API、人工确认触达。

已初步核查来源：

- YouTube Terms: https://tv.youtube.com/tv/terms
- X Developer Guidelines: https://docs.x.com/developer-guidelines
- X Automation Rules: https://help.x.com/articles/20174732
- Meta Automated Data Collection Terms: https://www.facebook.com/legal/automated_data_collection_terms
- Telegram Terms: https://telegram.org/tos/eu
- Telegram Spam FAQ: https://www.telegram.org/faq_spam/
- WhatsApp Business Policy: https://whatsappbusiness.com/policy/
- Google Maps Platform Terms: https://cloud.google.com/maps-platform/terms
- Google Places API Policies: https://developers.google.com/maps/documentation/places/web-service/policies
- Auto.ru Terms: https://yandex.ru/legal/autoru_terms_of_service/ru/index.html

Avito、VK、Yandex 等仍需进一步核查最新官方条款、API 文档、robots 和账号规则；暂按中高风险渠道处理，不进入 PoC 自动化采集范围。

## 12. 可扩展性设计

扩展顺序：

1. 国家/区域。
2. 渠道。
3. 车辆品类。
4. 自动化程度。

第一阶段架构中应优先将国家、渠道、话术和风险规则做成配置，避免系统被俄罗斯市场写死。

MVP 数据库采用客户主体 + 多来源记录 + 多联系方式的结构。第二阶段可扩展为客户主体 + 联系人 + 渠道账号 + 门店/分支机构。

## 13. 知识库与 SOP

PoC 需沉淀：

- 渠道 SOP。
- 线索审核 SOP。
- 触达 SOP。
- 销售交付 SOP。
- 合规 SOP。
- 关键词库。
- 失败案例库。
- 客户 FAQ/异议处理库。

PoC 阶段使用飞书文档 + 表格备注维护。MVP 阶段将关键 SOP 嵌入 CRM 页面。研发侧同步维护 Markdown 文档。

## 14. 成本与 ROI

PoC 记录基础成本，不做复杂财务模型。

关注指标：

- 每条有效线索成本。
- 每条回复成本。
- 每个销售机会成本。
- 长期每个成交成本和复购价值。

PoC 设置四类成本上限：

- 时间上限：2 周内完成。
- 人力上限：限定每天投入时间。
- 工具/API 成本上限。
- 渠道试错上限：某渠道采集 30-50 条仍没有 B 级线索时暂停或调整。

## 15. PoC 交付物

2 周 PoC 交付：

- 俄罗斯渠道与竞品初版地图。
- 飞书五张表。
- 100-200 条初步线索。
- AI 清洗与补全脚本/流程。
- 人工触达小样本。
- PoC 复盘报告。
- 知识库/SOP 初版。

PoC 不做网页 CRM 开发，重点验证业务闭环和渠道可行性。

## 16. MVP 交付物

4 周 MVP 交付：

- 轻量 CRM Web 看板。
- 飞书 API 单向同步 PostgreSQL。
- AI 建议面板。
- Agent/AI 执行日志。
- 车源/报价表与线索匹配。
- 触达记录与 SLA。
- 轻量仪表盘。
- 安全合规配置。

## 17. PoC 验收标准

- 至少找到 3 个可持续获取俄罗斯车商线索的低/中风险渠道。
- 100-200 条线索中，至少 30%-50% 可达到 B 级增强线索。
- AI 能稳定完成去重、补全、分类、摘要和话术建议。
- 小样本人工触达后，有真实回复或明确负反馈。
- 没有账号封禁、平台投诉、明显违规采集或批量骚扰行为。
- 客服/销售认为线索字段、话术和交付方式可用。

PoC 通过标准：渠道可持续 + B 级线索比例达标 + 小样本触达有反馈 + 风险可控 + 团队愿意接。

## 18. 暂停或转向条件

- 100-200 条线索中，B 级线索低于 20%。
- 小样本触达几乎无人回复，或多数明确拒绝。
- 出现账号限制、投诉、访问异常或明显违反条款风险。
- 销售团队不愿承接。
- 客户有兴趣但贸易、支付、物流或清关不可行。

暂停或转向不是失败机制，而是在证据基础上调整渠道、关键词、地区、话术或客户画像。
