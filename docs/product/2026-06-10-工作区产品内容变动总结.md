# 2026-06-10 工作区产品内容变动总结

## 对比范围

本总结基于当前工作区与 GitHub 远端 `origin/main` 的差异整理，覆盖最近几天尚未提交的产品能力变化。范围包含 `apps/mobile`、`apps/admin`、`apps/api`、`apps/agents` 以及配套测试、配置和迁移文件。

本次变动不是单点页面修补，而是围绕“来源发现 -> 线索池 -> 清洗/晋级 -> 客户工作台 -> 人工触达 -> 后台治理”的小范围运行闭环做了一轮产品化补齐。

## 移动端产品体验

### 首页指标改为真实业务口径

移动端首页的线索指标从概览型候选数，调整为更贴近运营动作的三类统计：

- `A/B/C级线索`：按 `pages/leads/index` 线索池中的 A/B/C 级 staging 线索统计。
- `线索来源`：按 `pages/sources/index` 来源候选池总数统计。
- `被清洗线索`：按 `pages/leads/cleaned` 已清洗线索统计。

同时首页新增客户侧指标，区分线索池和客户工作台，避免已晋级客户继续被误读为普通线索池存量。

### 线索池和线索详情修正

线索池列表改为以 staging lead 为主，不再把已晋级客户混入线索池展示。列表顶部筛选改为可点击的业务统计卡片，支持按邮箱、社媒联系方式、A/B/C 级、D/E 级进行快速筛选。

线索详情页补齐了真实详情映射，减少 `Unknown` 展示；调整了线索名称、经营状况判断等样式，使页面风格与移动端其他详情页保持一致。勿扰、调整等级、晋级客户等操作改为使用真实 staging lead ID 和后端动作接口，修复了刷新后状态回退、调整等级后未持久化等问题。

### 清洗线索页面

新增 `pages/leads/cleaned`，用于承载被清洗线索列表。清洗建议映射增强了线索展示名称、来源域名、联系方式和证据字段，避免清洗列表只显示弱信息或 `Unknown`。

### 客户工作台和客户详情

客户工作台的指标从 B/C 客户调整为 A/B/C 级客户，并支持点击统计卡片筛选。客户卡片对联系方式、车型意向、来源证据、下一步动作做了截断和布局优化，提升移动端密集信息阅读体验。

客户详情页新增“待补全字段”的多项人工补录入口。每个待补字段旁边都有“人工补录”按钮，提交后写入客户跟进记录。联系方式和来源证据区块调整为标签在首行右侧展示，避免信息较多时标签挤压正文或换行失控。底部操作区也调整为跟进记录页面同款安全区按钮布局。

### 来源审核和洞察入口

来源审核页从复杂筛选控件改为顶部业务统计卡片，支持快速查看待审 High、可抽取来源、重复候选和阻断来源。页面增加返回入口，底部导航归属到洞察入口。

新增 `pages/insights/index`，用于承接来源、运行、质量等洞察类入口，减少首页和线索池的导航负担。

### 触达助手和邮件发送

触达助手页移除了最底部导航，底部按钮改为参考 `pages/customers/followups` 的安全区操作栏。点击“编辑”会进入新的 `pages/outreach/send` 邮件发送页。

邮件发送页按普通邮件编辑体验设计，包含：

- 收件人：从客户联系方式或 staging lead 联系方式中提取邮箱。
- 主题：带入 AI 俄语触达草稿主题。
- 正文：带入 AI 俄语触达草稿正文。
- 操作：支持取消和发送。

发送动作调用后端真实邮件发送接口 `POST /outreach-drafts/{customer_id}/send-email`，走统一 `EmailSender` 服务，并要求人工确认，不做自动触达。

## 后台管理端产品变化

### 后台从单页大看板拆成多页面工作台

Admin 应用从一个超长 `App.vue` 重构为侧边栏页面工作台，新增独立页面模块：

- 运行总览
- 线索渠道
- 同步与 AI 审计
- Agents 运行看板
- 客户指标与风控
- LLM 大模型治理
- 知识库
- 邮件审核
- 真实 API 联调

这样后台从“全部指标堆在一个页面”调整为“按治理任务进入对应工作台”，更适合持续小范围运行和问题排查。

### LLM 和 Prompt 治理增强

LLM 治理页增加 Prompt 工作台视角，可以查看选中模板的 System Prompt、User Prompt Template 和输出 Schema。邮件审核、邮件质量指标改为适配新的 `/email-replies` API，并兼容新的自动发送决策、人工审核和硬拦截原因字段。

### 真实运行总览

新增运行总览服务和页面，用于聚合真实 API 运行状态，而不是依赖静态 seed 数据作为验收依据。后台页面拆分后保留 hash 导航，方便直接定位到具体治理页面。

## Agents 和后端运行链路

### Source Discovery 从静态默认 Prompt 调整为运行时查询规划

默认 Source Discovery Prompt 改为 `source_discovery_query_planner_default`，输出结构从固定来源字段扩展为 `queries + candidates + blocked_candidates`。产品含义是：Agent 先生成公开搜索查询，再提出可人工核验的来源候选，仍严格禁止登录绕过、验证码绕过、反爬绕过和自动触达。

### 外部 Agent 调度扩展为批处理闭环

外部 Agent 调度链路新增批量 Lead Extraction/Grading 输入准备和结果收口能力：

- 来源候选会按批次读取公开页面内容并准备输入。
- 每个来源候选记录准备成功、准备失败、处理成功、处理失败。
- 处理完成后更新来源候选的 extraction status。
- 任务运行摘要记录 processed items、created count、updated count、失败原因等信息。

同时新增 Deep Enrichment 和 Lead Cleanup 的外部 Agent 定时任务配置，为后续自动深挖补全和清洗治理提供调度基础。

### Prompt 模板进入数据库运行时

Agents 侧新增 LLM Prompt 模板模型和仓储服务，API 侧新增迁移用于种子运行时 Prompt 模板。产品上意味着 Prompt 不再只是代码内静态文本，而可以进入后台治理、版本管理和运行时选择。

### 线索消费、勿扰和晋级逻辑修正

后端 staging lead 详情新增勿扰匹配信息和对应客户 ID，移动端可据此正确显示“勿扰”状态并执行取消勿扰或后续动作。

客户晋级服务新增通过名称或联系方式查找勿扰客户的能力。线索等级调整中，Watch/Invalid 不再直接进入 rejected，而是保持在可复核的 pending 状态，避免清洗类线索从工作台中不可见。

## 邮件发送服务

后端新增首次触达邮件发送接口：

- `POST /outreach-drafts/{customer_id}/send-email`
- 请求包含收件人、主题、正文、发送人和人工确认标记。
- 发送前校验 `human_confirmed=true` 和发件邮箱配置。
- 使用统一 `EmailSender.from_settings(settings).send(...)`。
- 返回 provider、provider message id、收件人、主题和 `auto_send=false`。

该能力把“AI 生成触达草稿”和“人工确认后真实发送邮件”串成闭环，同时保留自动触达边界。

## 测试和验收覆盖

本次变动补充了移动端、Admin、API、Agents 多层测试：

- 移动端页面契约、线索池、客户详情、清洗线索、触达助手、邮件发送、首页指标等测试。
- Admin 页面拆分、运行总览、LLM 治理、邮件审核、真实 API 联调等测试。
- API 外部 Agent 调度、批处理消费、staging lead 动作、邮件发送、勿扰审计等测试。
- Agents Prompt 运行时、Source Discovery、Lead Extraction/Grading、Deep Enrichment、Lead Cleanup 等测试。

已在本地完成的关键验证包括：

- `npm --prefix apps/mobile test`
- `npm --prefix apps/mobile run check:syntax`
- `npm --prefix apps/mobile run build:h5`
- `pytest apps/api/tests/test_outreach_draft_api.py -q`

## 产品影响总结

这批改动把系统从“可展示的获客 CRM 原型”推进到“可运营的小范围运行工作台”：

- 运营人员可以在移动端更清楚地区分来源、线索、清洗线索和客户。
- 线索的勿扰、等级调整、晋级客户、触达邮件发送开始走真实后端链路。
- 管理者可以在 Admin 端按运行、风控、LLM、知识库、邮件审核等任务分区查看真实状态。
- Agents 链路开始具备批处理、运行时 Prompt、结果消费和失败审计能力。
- 产品边界继续保持人工确认优先，AI 不自动触达、不自动绕过风控。
