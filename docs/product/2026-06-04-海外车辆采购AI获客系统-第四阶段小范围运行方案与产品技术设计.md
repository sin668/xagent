## 1. 阶段目标

第四阶段的核心目标是把现有分散在 `apps/api` 中的 LLM Agent 能力，平行迁移到独立运行的 `apps/agents` 服务中，并使用 LangGraph 重构 Agent 编排。

本阶段强调 **局部小范围重构、平行运行、服务边界清晰、可对照验证**，不是立即替换 `apps/api` 中已运行的生产链路。

本阶段要证明：

- `apps/agents` 可以作为独立服务运行，通过 HTTP API 被 `apps/api` 调用。
- `apps/agents` 可以承载 Deep Enrichment、Lead Cleanup、Source Discovery、Lead Extraction/Grading 的 LangGraph 平行版本。
- Deep Enrichment 和 Lead Cleanup 可以进入受控 active_run，由 `apps/api` 继续执行业务写库、合规硬规则和人工审核。
- Source Discovery 和 Lead Extraction/Grading 可以进入 shadow_run，不污染现有来源池和 `staging_leads`。
- Agent 执行事实源逐步从 `apps/api.agent_task_runs` 迁移到 `apps/agents` 自己的运行表。
- `apps/api` 中已有 LLM Agent、调度、审计、业务接口保持不变，第四阶段只新增兼容调用与摘要记录。

## 2. 继承规则与强制边界

第四阶段必须继承并严格执行以下规则：

- `apps/api` 中现有 LLM Agent 保持不变，不在第四阶段直接替换生产入口。
- `apps/agents` 独立服务运行，`apps/api` 通过 HTTP API 调用，不做本地包注入。
- `apps/api` 与 `apps/agents` 同机同环境运行，独立端口：`apps/api:8000`，`apps/agents:8010`。
- `apps/agents` 不直接写 `customers`、`lead_sources`、`contact_methods`、`staging_leads` 等业务 core 表。
- `apps/agents` 只写自己的 Agent 运行状态表，例如 `agent_service_runs`。
- 所有业务表写入、合规硬规则、人工确认、客户晋级、字段采纳、清洗执行仍由 `apps/api` 负责。
- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- 不自动晋级客户。
- 不自动归并客户。
- 不自动恢复 Invalid。
- 不自动触达客户。
- High/Forbidden、勿扰、C 级合规复核、证据校验等硬规则不得被 LangGraph 绕过。
- LLM 缺失字段必须输出 `Unknown`、`null` 或空数组，不得编造。
- 所有 Agent 输出必须包含 schema 版本、执行状态、节点摘要、风险 flags、来源证据和审计信息。

## 3. 本阶段产品范围

### 3.1 本阶段要做

- 新增 `apps/agents` FastAPI 服务入口。
- 新增 `apps/agents` `/health` 健康检查。
- 新增 `apps/agents` 内部 API Key 鉴权。
- 新增统一 Agent Run request / response envelope。
- 新增 `apps/agents` 专属运行状态表，作为 Agent 执行事实源。
- 新增 Agent 失败重试能力，由 `apps/agents` 自己负责。
- 新增 Deep Enrichment LangGraph HTTP API。
- 新增 Lead Cleanup LangGraph HTTP API。
- 新增 Source Discovery LangGraph shadow API。
- 新增 Lead Extraction/Grading LangGraph 组合 shadow API。
- 新增 `apps/api` 到 `apps/agents` 的 HTTP Agent client。
- `apps/api.agent_task_runs` 不改表结构，只在 `output_summary_json` 保存 `external_agent_run_id` 等兼容摘要。
- 对每类 Agent 进行 20-50 条小样本运行。
- 输出对照指标、失败分类、Go/No-Go 报告。

### 3.2 本阶段不做

- 不删除 `apps/api` 现有 LLM Agent。
- 不删除 `apps/api.agent_task_runs`。
- 不删除 `apps/api` retry worker 或 scheduler。
- 不把 `apps/agents` 作为 Python 本地包注入到 `apps/api`。
- 不引入队列 worker。
- 不引入 Docker Compose 或容器编排作为必需项。
- 不让 Source Discovery 的 LangGraph 输出直接污染 `lead_source_candidates`。
- 不让 Lead Extraction/Grading 的 LangGraph 输出直接写入 `staging_leads`。
- 不做完整多租户权限、JWT 或复杂 RBAC。
- 不把 `apps/agents` 暴露为公网服务。

## 4. 总体架构

### 4.1 服务拓扑

```text
Mobile / Admin / Operator
        |
        v
apps/api FastAPI :8000
  - 业务 API
  - 权限与人工操作
  - 业务表写入
  - 合规硬规则
  - 兼容 agent_task_runs 摘要
        |
        | HTTP + X-Agents-Api-Key
        v
apps/agents FastAPI :8010
  - LangGraph Agent API
  - Agent run 状态
  - Agent retry
  - Agent 节点 trace
  - LLM / Tool 编排
        |
        v
PostgreSQL
  - apps/api 业务表
  - apps/agents 专属运行表
```

### 4.2 职责边界

| 服务 | 职责 | 不负责 |
|---|---|---|
| `apps/api` | 业务入口、权限、人工操作、业务表写入、合规硬规则、客户晋级、字段采纳、清洗执行、兼容摘要 | LangGraph 节点编排、Agent 自主重试、Agent 节点状态事实源 |
| `apps/agents` | LangGraph 编排、LLM 调用、工具调用、节点状态、Agent 重试、Agent 运行审计、结构化输出 | 写 core 业务表、执行人工确认、触达客户、客户晋级、绕过风险规则 |

### 4.3 当前代码迁移关系

当前 `apps/api` 已有：

- Source Discovery Agent
- Lead Extraction / Lead Grading
- Deep Enrichment service 调用 runtime 的接口
- Lead Cleanup service 调用 runtime 的接口
- `agent_task_runs`
- retry worker / scheduler
- LLMClient / prompt templates / audit logs

第四阶段处理方式：

```text
apps/api 现有能力保持
  -> 不删除
  -> 不替换生产入口
  -> 新增 HttpAgentRuntimeClient

apps/agents 新增独立服务能力
  -> 实现 LangGraph 平行版本
  -> 自己管理 Agent run 状态和重试
  -> 返回结构化输出给 apps/api
```

## 5. Agent 范围与运行模式

第四阶段迁移范围选择 C：全部 LLM Agent 都做 LangGraph 平行版本。

| Agent | 第四阶段目标 | 默认运行模式 | 是否允许 active_run | 业务写入 |
|---|---|---|---:|---|
| Deep Enrichment | LangGraph HTTP API + active 接入 | active 可开启 | 是 | `apps/api` 写 `lead_enrichment_results` / `lead_enrichment_field_candidates` |
| Lead Cleanup | LangGraph HTTP API + active 接入 | active 可开启 | 是 | `apps/api` 写 `lead_cleanup_runs` / `lead_cleanup_suggestions` |
| Source Discovery | LangGraph shadow API | shadow | 否 | 不直接写 `lead_source_candidates`，只做对照 |
| Lead Extraction/Grading | LangGraph 组合 shadow API | shadow | 否 | 不直接写 `staging_leads`，只做对照 |

建议配置：

```text
AGENT_LANGGRAPH_DEEP_ENRICHMENT_MODE=active
AGENT_LANGGRAPH_LEAD_CLEANUP_MODE=active
AGENT_LANGGRAPH_SOURCE_DISCOVERY_MODE=shadow
AGENT_LANGGRAPH_LEAD_EXTRACTION_GRADING_MODE=shadow
```

运行模式定义：

| 模式 | 说明 | 是否影响业务表 |
|---|---|---:|
| `dry_run` | 只运行 Agent 并返回结果，不进入业务流程 | 否 |
| `shadow_run` | 与现有链路并行运行，用于对照，不影响原业务结果 | 否 |
| `active_run` | `apps/api` 使用 `apps/agents` 输出进入现有校验和写库流程 | 是，但只由 `apps/api` 写 |

## 6. apps/agents HTTP API 设计

### 6.1 基础接口

```text
GET  /health
POST /agent-runs/deep-enrichment
POST /agent-runs/lead-cleanup
POST /agent-runs/source-discovery
POST /agent-runs/lead-extraction-grading
GET  /agent-runs/{agent_service_run_id}
```

第四阶段优先实现组合 API：

```text
POST /agent-runs/lead-extraction-grading
```

内部保留 extraction 子图和 grading 子图边界，后续可扩展：

```text
POST /agent-runs/lead-extraction
POST /agent-runs/lead-grading
```

### 6.2 请求 Envelope

```json
{
  "request_id": "uuid",
  "agent_task_run_id": "uuid-or-null",
  "trigger_source": "manual_api|shadow_run|scheduler|test",
  "agent_mode": "active|shadow|dry_run",
  "input": {},
  "options": {
    "max_retries": 2,
    "timeout_seconds": 120,
    "dry_run": false,
    "shadow_mode": false
  }
}
```

### 6.3 响应 Envelope

```json
{
  "schema_version": "phase4.agent.run.v1",
  "agent_service_run_id": "uuid",
  "request_id": "uuid",
  "status": "succeeded|failed|blocked|retrying",
  "agent_type": "deep_enrichment|lead_cleanup|source_discovery|lead_extraction_grading",
  "agent_mode": "active|shadow|dry_run",
  "output": {},
  "audit": {
    "writes_core_tables": false,
    "executed_nodes": [],
    "failed_node": null,
    "risk_flags": [],
    "source_urls": [],
    "llm_provider": "deepseek",
    "llm_model": "deepseek-chat"
  },
  "error": null
}
```

### 6.4 鉴权

`apps/agents` 使用内部 API Key：

```text
Header: X-Agents-Api-Key: <AGENTS_API_KEY>
apps/api env: AGENTS_API_KEY
apps/agents env: AGENTS_API_KEY
```

第四阶段不做用户级 JWT / RBAC。用户权限仍由 `apps/api` 控制。

### 6.5 同步与异步调用策略

由于第四阶段确认 Agent 失败重试由 `apps/agents` 自己负责，Agent Run API 必须支持“提交任务”和“查询任务结果”两种语义。

建议第四阶段采用混合策略：

| Agent | 推荐调用方式 | 说明 |
|---|---|---|
| Deep Enrichment | 同步优先，超时后转查询 | 用户点击深挖后希望较快看到结果；若 LLM/搜索超时，返回 run id，由前端或 `apps/api` 查询 |
| Lead Cleanup | 异步优先 | 清洗可能扫描多条线索，不应阻塞 API 请求 |
| Source Discovery | 异步优先 | shadow 样本和搜索链路耗时不可控 |
| Lead Extraction/Grading | 异步优先 | 页面读取、LLM 抽取、分级和硬规则校验链路较长 |

建议响应分两类：

1. 已完成：

```json
{
  "schema_version": "phase4.agent.run.v1",
  "agent_service_run_id": "uuid",
  "status": "succeeded",
  "output": {},
  "audit": {},
  "error": null
}
```

2. 已受理但仍在运行或等待重试：

```json
{
  "schema_version": "phase4.agent.run.v1",
  "agent_service_run_id": "uuid",
  "status": "running",
  "output": null,
  "audit": {
    "accepted": true
  },
  "error": null
}
```

`apps/api` 处理策略：

- 对 active_run 的 Deep Enrichment / Lead Cleanup，如果返回 `succeeded`，立即走现有 schema 校验和写库流程。
- 如果返回 `running` / `retrying`，`apps/api` 只保存 `external_agent_run_id`，不写业务结果。
- 后续由轮询、人工刷新或后台轻量同步任务查询 `GET /agent-runs/{agent_service_run_id}`，拿到最终结果后再进入 `apps/api` 业务写入流程。
- 第四阶段不要求实现 webhook callback，避免新增反向调用鉴权和幂等复杂度。

专业判断：

- 该策略与“`apps/agents` 是 Agent 执行事实源”一致。
- `apps/api` 不再主控 retry，只负责发起、查询和消费最终结构化结果。
- 第四阶段先用轮询/查询，后续如果需要再引入 webhook 或队列。

## 8. LangGraph 图设计

### 8.1 Deep Enrichment Graph

目标：为待完善线索寻找公开证据，输出字段级候选，不直接写 core 表。

```text
load_lead
  -> validate_trigger_boundary
  -> build_search_keywords
  -> search_public_sources
  -> read_public_pages
  -> extract_field_candidates
  -> validate_evidence
  -> score_confidence
  -> output_field_candidates
  -> wait_human_review
```

关键边界：

- Watch / Invalid 不允许深挖。
- 勿扰匹配不允许深挖。
- Forbidden 来源不允许深挖。
- 无证据字段不得输出为可采纳候选。
- 输出写入由 `apps/api` 负责。

### 8.2 Lead Cleanup Graph

目标：生成重复、合并、恢复、确认无效等清洗建议，不自动执行。

```text
load_cleanup_scope
  -> detect_duplicates
  -> classify_invalid_reason
  -> find_restore_candidates
  -> validate_no_auto_execution
  -> output_cleanup_suggestions
  -> wait_human_review
```

关键边界：

- 不自动删除线索。
- 不自动恢复 Invalid。
- 不自动合并客户。
- 所有清洗建议必须人工确认。

### 8.3 Source Discovery Graph

目标：平行发现候选来源，用于 shadow 对照，不直接污染来源候选池。

```text
load_channel_strategy
  -> build_discovery_queries
  -> search_public_sources
  -> normalize_source_candidates
  -> classify_channel_risk
  -> dedupe_candidates
  -> validate_source_evidence
  -> output_shadow_candidates
```

关键边界：

- Forbidden 不得输出为可抽取候选。
- High 只可作为待人工审核候选。
- 无来源 URL、无证据摘要、登录墙/验证码来源不得进入可抽取候选。
- 第四阶段只做 shadow_run。

### 8.4 Lead Extraction / Grading Graph

目标：组合执行线索抽取和分级，用于 shadow 对照，不直接写 `staging_leads`。

```text
load_source_candidate
  -> validate_source_allowed_for_shadow
  -> read_public_page
  -> extract_lead_json
  -> validate_extraction_schema
  -> validate_evidence_and_contacts
  -> grade_lead
  -> apply_hard_rules
  -> output_shadow_staging_lead
  -> compare_with_api_result
```

关键边界：

- 抽取和分级内部拆图，外部优先组合 API。
- 联系方式必须在公开页面文本中出现。
- High / Forbidden / 勿扰 / C 级合规规则必须与 `apps/api` 硬规则一致。
- 第四阶段只做 shadow_run。

## 9. Agent 重试与错误分类

### 9.1 重试事实源

Agent 失败重试由 `apps/agents` 自己负责。

`apps/api` 的 `agent_task_runs`、retry worker、scheduler 在第四阶段只是临时保留，后续逐步去掉 Agent 执行相关职责。

### 9.2 错误分类

| error_type | 是否可重试 | 说明 |
|---|---:|---|
| `timeout_error` | 是 | HTTP 超时、LLM 超时 |
| `provider_rate_limited` | 是 | LLM 限流 |
| `transient_network_error` | 是 | 网络短暂失败 |
| `schema_validation_error` | 否 | 输出 schema 不合格 |
| `evidence_validation_error` | 否 | 证据缺失或联系方式反编造失败 |
| `risk_blocked` | 否 | High/Forbidden/勿扰/策略阻断 |
| `contract_mismatch` | 否 | `apps/api` 与 `apps/agents` contract 不一致 |

### 9.3 重试策略

建议第四阶段采用：

- 每个 run 默认最多 2 次重试。
- 只对可重试错误重试。
- 指数退避或固定间隔均可，第四阶段优先简单固定间隔。
- 所有重试必须记录 `retry_count`、`error_type`、`next_retry_at`。
- 如果最终失败，`apps/agents` 返回结构化失败结果，`apps/api` 记录兼容摘要。

## 10. apps/api HTTP Agent Client

### 10.1 配置

建议新增：

```text
AGENTS_BASE_URL=http://127.0.0.1:8010
AGENTS_API_KEY=change_me
AGENTS_TIMEOUT_SECONDS=120
AGENT_LANGGRAPH_DEEP_ENRICHMENT_MODE=active
AGENT_LANGGRAPH_LEAD_CLEANUP_MODE=active
AGENT_LANGGRAPH_SOURCE_DISCOVERY_MODE=shadow
AGENT_LANGGRAPH_LEAD_EXTRACTION_GRADING_MODE=shadow
```

### 10.2 Client 职责

`apps/api` 新增 `AgentRuntimeClient` 或 `HttpAgentRuntime`：

- 调用 `apps/agents /health`。
- 调用各 Agent Run API。
- 注入 `X-Agents-Api-Key`。
- 传递 `request_id`、`agent_task_run_id`、`trigger_source`。
- 处理超时和 HTTP 错误。
- 校验响应 envelope。
- 将 `external_agent_run_id`、status、schema_version、mode 写入兼容摘要。

### 10.3 不迁移的职责

`apps/api` client 不负责：

- LangGraph 节点执行。
- Agent 内部重试。
- Agent 节点状态存储。
- 直接修正 `apps/agents` 输出。

## 11. 小范围样本与对照指标

每类 Agent 采用 20-50 条样本。

| Agent | 样本量 | 样本构成建议 |
|---|---:|---|
| Source Discovery | 30-50 条 | Low/Medium 官网与目录、High 公开社媒入口、重复 URL、无效 URL、Forbidden 边界 |
| Lead Extraction/Grading | 30-50 条 | A/B/C/Watch/Invalid、联系方式完整/缺失、证据充分/不足、High/Forbidden、疑似编造边界 |
| Deep Enrichment | 20-30 条 | 缺城市、缺联系方式、缺经营信号、缺意向车型、已有勿扰匹配、Forbidden 来源 |
| Lead Cleanup | 20-30 条 | 强重复、弱重复、联系方式冲突、来源证据冲突、Watch 可恢复、Invalid 待确认 |

对照指标：

| Agent | 指标 | Go/No-Go 建议 |
|---|---|---|
| Source Discovery | URL 有效率、重复率、风险分级一致率、证据完整率 | 证据完整率 >= 90%，Forbidden 误放为 0 |
| Lead Extraction | schema 通过率、证据命中率、联系方式反编造通过率、字段完整度 | schema 通过率 >= 95%，联系方式反编造失败不得高于现有链路 |
| Lead Grading | 等级一致率、硬规则一致率、C/Invalid/Watch 分流准确性 | 硬规则一致率 100%，等级差异必须可解释 |
| Deep Enrichment | 字段候选有效率、人工接受率、无证据候选率 | 无证据候选率为 0 |
| Lead Cleanup | 重复建议准确率、错误合并建议数、人工拒绝率 | 不得出现自动执行，错误合并建议必须可人工拦截 |

## 14. 风险与缓解

| 风险 | 表现 | 缓解 |
|---|---|---|
| 范围膨胀 | 同时迁移四类 Agent，开发周期变长 | 拆分 P4-E1 到 P4-E7，小步验收 |
| 双状态源混乱 | `apps/api.agent_task_runs` 与 `apps/agents.agent_service_runs` 表达不一致 | 明确 `apps/agents` 是执行事实源，`apps/api` 只是兼容摘要 |
| Source Discovery 污染来源池 | shadow 输出被误写入 `lead_source_candidates` | 第四阶段禁止 Source Discovery active_run |
| Extraction/Grading 污染 staging | LangGraph 输出直接写 `staging_leads` | 第四阶段禁止 Lead Extraction/Grading active_run |
| 鉴权缺失 | 同机端口被误调用 | 内部 API Key + 默认绑定本机/内网 |
| LLM 输出编造 | 字段、联系方式、证据不可靠 | schema + evidence + contact 反编造校验 |
| core 表误写 | `apps/agents` 绕过 `apps/api` 写客户表 | API contract 和 DB 权限边界限制 |
| 重试风暴 | LLM 限流后持续重试 | max_retries + retryable error 分类 + backoff |

## 15. 后续阶段建议
