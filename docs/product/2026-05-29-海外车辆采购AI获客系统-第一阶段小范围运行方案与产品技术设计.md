## 1. 阶段目标

第一阶段从 PoC 表格验证进入小范围运行，目标不是追求最大自动化，而是验证“可控自动发现 + AI 抽取分级 + PostgreSQL 直入库 + 人工复核交付”的稳定闭环。

本阶段要证明：

- 每日可稳定发现约 100 条候选线索。
- 有效线索可从候选数据中通过规则、AI 和人工复核进入正式客户库。
- A/B/C 三类渠道可并行运行，并能区分风险、证据和准入状态。
- High 风险渠道仅作为只读公开发现入口，不直接进入触达队列。
- 数据不再依赖飞书表格，统一进入 PostgreSQL。
- AI 处理结果可审计、可复核、可追溯。

## 2. 核心结论

| 维度 | 结论 |
|---|---|
| 渠道策略 | A/B/C 并重，拓展更多线索来源 |
| High 风险渠道 | AI/浏览器 Agent 可只读公开页面，不登录、不互动、不触达，结果必须二次复核 |
| 智能体策略 | 受控自动发现，不做完全自主 Agent |
| 大模型策略 | 规则 + LLM 混合，LLM 负责理解生成，规则负责硬约束 |
| 知识库策略 | PostgreSQL + pgvector 统一承载 |
| 入库策略 | 宽进严出，raw/staging/core/audit 四层 |
| Go/No-Go 主指标 | 有效线索数量优先 |
| 初始规模 | 每日约 100 条候选线索，目标 20-50 条 core 有效线索 |

## 3. 产品范围

### 3.1 本阶段要做

- 渠道计划配置：国家、城市、渠道、关键词、风险等级、任务上限。
- 受控自动发现：自动生成搜索任务和候选 URL。
- 公开页面读取：读取允许范围内的公开页面文本。
- AI 抽取：抽取客户名称、城市、联系方式、经营信号和证据。
- AI 分级：输出 A/B/C/Invalid/Watch、原因、缺失字段和下一步动作。
- PostgreSQL 直入库：raw/staging/core/audit 分层保存。
- 人工复核队列：运营人员确认有效性、二次复核、晋级 core。
- 渠道风险闸门：按 Low/Medium/High/Forbidden 执行动作约束。
- 知识库 RAG：使用 PostgreSQL + pgvector 提供 SOP、FAQ、关键词和话术上下文。
- 指标看板：展示候选、有效线索、渠道产出、风险事件和复核效率。

### 3.2 本阶段不做

- 不做自动社交私信。
- 不做自动加好友。
- 不做登录后批量采集。
- 不做反爬规避。
- 不抓取非公开数据。
- 不将 High 来源线索直接进入触达队列。
- 不让 AI 编造联系方式、客户意向或来源证据。
- 不在报价、合同前跳过 C 级合规复核。
- 不做完整销售 CRM、ERP 或库存系统深度集成。
- 不做复杂知识图谱，先做可用的结构化规则 + RAG。

## 4. 渠道矩阵

### 4.1 A 类低风险渠道

| 渠道 | 典型来源 | 允许动作 | 输出目标 |
|---|---|---|---|
| 官网 | 车商官网、联系页、关于我们 | 读取公开页面、提取公开联系方式、保存来源链接 | 稳定有效线索 |
| 公开企业目录 | 黄页、工商目录、本地企业目录 | 读取公开字段、跳转官网复核 | 客户名称、官网、邮箱、电话 |
| 搜索引擎结果 | Google/Yandex/Bing 搜索结果指向的公开网页 | 通过搜索任务发现候选 URL，读取目标公开页 | 候选客户池 |
| 品牌/经销商公开列表 | 品牌经销商页、汽车城公开列表 | 读取公开商户信息 | 经销商线索 |

A 类渠道可进入智能体任务队列，是 PostgreSQL core 有效线索的主要来源。

### 4.2 B 类中风险渠道

| 渠道 | 典型来源 | 允许动作 | 限制 |
|---|---|---|---|
| Google/Yandex 地图 | 公开商户卡片、官网链接 | 小批量公开核验、优先跳转官网 | 不批量抓取地图数据库 |
| Drom 公开页 | 商户页、公开帖 | 读取公开商户信息、保留原始链接 | 不登录、不自动联系 |
| 公开论坛 | 公开商户帖、行业讨论 | 只提取企业名、官网、公开商务联系方式 | 不抓评论关系链 |
| 公开视频描述 | YouTube 等公开描述区 | 读取频道简介/描述中的官网或邮箱 | 不自动评论、不抓粉丝/评论 |

B 类渠道负责扩大覆盖面，必须配置样本上限和异常暂停机制。

### 4.3 C 类行业链路渠道

| 渠道 | 典型来源 | 价值 | 运行方式 |
|---|---|---|---|
| 展会名录 | 汽车展、配件展、贸易展参展商 | 客户质量高 | 少量、高质量、强复核 |
| 协会会员 | 经销商协会、商会、进口车协会 | 身份可信 | 人工复核优先 |
| B2B 平台公开企业页 | 企业主页、产品页 | 可发现批发/进口商 | 只读公开页 |
| 物流/清关伙伴 | 公开案例、合作客户 | 接近交易链路 | 不采集私人信息 |
| 检测/维修/二手车服务商 | 公开客户案例、服务网络 | 可发现车商生态 | 作为线索发现入口 |

C 类渠道与 A/B 并行运行，但不走粗放批量采集，应作为质量观察池和高质量来源池。

### 4.4 High 风险渠道

High 渠道包括 VK、Telegram、Facebook/Instagram、X、Avito、Auto.ru、TikTok、Odnoklassniki、Max 等。

本阶段允许：

- AI/浏览器 Agent 读取公开页面。
- 提取公开页面中的客户名称、官网、公开邮箱、公开电话、WhatsApp、Telegram。
- 保存有限证据和来源链接。
- 将结果标记为 `Watch` 或 `Needs Low/Medium Verification`。

本阶段禁止：

- 登录。
- 私信。
- 加好友。
- 关注。
- 入群。
- 自动互动。
- 抓取评论、粉丝、好友、关系链。
- 绕过验证码、访问限制、频率限制或反爬机制。
- 直接进入触达队列。

High 渠道结果必须通过官网、地图、公开目录或搜索结果完成二次复核，才能进入正式流程。

## 5. Agent 产品设计

### 5.1 Agent 分工

| Agent | 输入 | 输出 | 关键限制 |
|---|---|---|---|
| 渠道发现 Agent | 国家、城市、渠道、关键词、配额 | 搜索任务、候选 URL、发现理由 | 只发现，不触达 |
| 公开页面读取 Agent | 候选 URL、渠道规则 | 页面标题、公开文本摘要、证据摘录、读取状态 | 不登录、不绕过、不高频 |
| 线索抽取 Agent | 公开文本、来源 URL、渠道元数据 | 客户名称、联系方式、城市、经营信号 | 缺失输出 Unknown/null/[] |
| 分级风控 Agent | 抽取结果、规则、证据 | A/B/C/Invalid/Watch、原因、下一步动作 | Invalid/Watch 不触达 |
| 入库审计 Agent | 结构化线索、AI 输出、任务日志 | raw/staging/core/audit 记录 | 无证据不得进 core |
| 复核队列 Agent | staging 线索、风险标记 | 人工复核任务 | 不替代人工最终确认 |

### 5.2 标准任务流

```text
channel_plan
  -> search_task
  -> candidate_url
  -> page_snapshot
  -> llm_extraction
  -> rule_validation
  -> llm_grading
  -> dedupe
  -> postgres_upsert
  -> human_review_queue
  -> core_customer
```

### 5.3 Agent 执行前检查

每个任务执行前必须检查：

- 渠道是否登记。
- 渠道风险等级。
- 允许动作是否覆盖当前动作。
- 禁止动作是否被触发。
- 每日渠道配额是否超限。
- 是否存在验证码、登录墙、访问异常、平台警告。
- 是否需要 High 二次复核。

任何检查失败，任务进入 `blocked` 或 `needs_manual_review`。

## 6. 大模型接入设计

### 6.1 LLM 使用场景

- 公开文本结构化抽取。
- 俄语/英语/中文识别与翻译。
- 客户类型判断。
- 活跃度、规模信号、进口/二手相关性判断。
- 推荐等级与原因生成。
- 缺失信息识别。
- 俄语触达草稿生成。
- 人工复核建议。

### 6.2 规则与 LLM 的边界

| 类型 | 执行方 | 原则 |
|---|---|---|
| 渠道是否允许采集 | 规则 | 硬约束 |
| High 是否可进触达队列 | 规则 | 默认不可 |
| 勿扰是否阻断触达 | 规则 | 必须阻断 |
| C 级是否需合规复核 | 规则 | 必须复核 |
| 客户画像摘要 | LLM | 可生成，但需证据 |
| 推荐等级原因 | LLM + 规则 | LLM 解释，规则校验 |
| 缺失字段 | LLM | 不得编造 |
| 话术草稿 | LLM + 知识库 | 人工审核后使用 |

### 6.3 AI 审计要求

每次 LLM 调用必须保存：

- `task_id`
- `prompt_version`
- `model_name`
- `input_ref`
- `input_text_excerpt`
- `output_json`
- `knowledge_item_refs`
- `source_urls`
- `evidence_notes`
- `validation_result`
- `created_at`
- `created_by_agent`

### 7.3 入库闸门

- 没有来源链接：不得进入 core。
- 没有证据备注：不得进入 core。
- 没有联系方式：可进入 staging，但不得进入待触达队列。
- Invalid/Watch：不得进入触达队列。
- High 来源：必须二次复核为 Low/Medium 来源后才可进入 core。
- 勿扰客户：不得再次进入触达队列。
- C 级线索：报价/合同前必须合规复核。

### 7.4 幂等与去重

建议使用多层去重：

- 强重复：标准化客户名称 + 联系方式 hash。
- 疑似重复：标准化客户名称 + 城市 + 来源域名。
- 来源重复：source URL hash。
- High 转 Low/Medium 二次复核：保留原始 High 发现来源和二次复核来源，不覆盖证据链。

## 8. 知识库与 RAG 设计

### 8.1 知识集合

| 集合 | 内容 | 用途 |
|---|---|---|
| `channel_sop` | 渠道允许/禁止动作、采集 SOP | Agent 执行前检索 |
| `faq` | 价格、车况、物流、付款、合作模式 FAQ | 话术生成 |
| `script_template` | 俄语触达模板 | 人工审核话术 |
| `keyword_library` | 俄语/英语/中文关键词 | 任务生成 |
| `vehicle_knowledge` | 车型、车况、配置、报价说明 | 客户匹配 |
| `compliance_rules` | 贸易、制裁、触达合规红线 | 风控提示 |
| `failed_cases` | 失败案例和误判样本 | 质量改进 |

### 8.2 知识状态

- `draft`
- `reviewing`
- `approved`
- `deprecated`

只有 `approved` 知识允许进入生产 Agent 检索。

### 8.3 RAG 使用规则

- 抽取任务检索关键词库、渠道 SOP。
- 分级任务检索分级规则、失败案例、渠道 SOP。
- 话术任务检索 FAQ、触达模板、合规规则。
- 合规任务检索贸易红线和渠道限制。
- LLM 输出必须引用知识条目 ID 或来源。

## 9. 后台与移动端产品页面

### 9.1 管理后台优先页面

第一阶段后台优先级：

1. 渠道计划与任务页。
2. staging 复核队列。
3. 线索详情与证据页。
4. AI 审计日志页。
5. 渠道风险配置页。
6. 指标看板。
7. 知识库管理页。

### 9.2 移动端优先页面

移动端继续服务客服和销售承接：

1. 今日待复核/待跟进。
2. B/C 线索池。
3. 线索详情。
4. 触达助手。
5. 触达记录。
6. 勿扰标记。

### 9.3 人工复核操作

复核人员需要能做：

- 确认有效/无效。
- 修正客户名称、城市、客户类型、联系方式。
- 确认来源证据是否充分。
- 标记重复。
- 标记 High 是否已二次复核。
- 确认 A/B/C/Invalid/Watch。
- 分配客服或销售。
- 标记勿扰。

## 10. 运行 SOP

### 10.1 每日启动

1. 查看昨日风险事件。
2. 检查暂停渠道。
3. 确认今日渠道计划和配额。
4. 确认知识库 approved 状态。
5. 启动 Low/Medium 自动发现任务。
6. 启动 High 只读公开发现小样本任务。

### 10.2 每日复核

1. 查看 staging 队列。
2. 优先复核带联系方式和来源证据完整的线索。
3. 对 High 来源检查二次复核。
4. 对 C 级检查合规复核标记。
5. 对 Invalid/Watch 保持不进入触达队列。
6. 对拒绝或敏感客户标记勿扰。

### 10.3 每日收工

1. 统计候选 URL、staging 线索、core 有效线索。
2. 统计渠道有效率和重复率。
3. 统计 AI 错误、缺证据、疑似编造。
4. 统计风险事件。
5. 输出次日关键词和渠道调整建议。

## 11. 指标与暂停条件

### 11.1 Go/No-Go 主指标

主指标：有效线索数量。

建议目标：

- 每日候选线索约 100 条。
- 每日 core 有效线索目标 20-50 条。
- 连续 1-2 周稳定产出，进入 Go。

### 11.2 辅助硬闸门

- 来源证据完整率达标。
- High 误入自动触达队列次数为 0。
- 勿扰误触达次数为 0。
- 投诉、封禁、违规风险为 0。
- AI 编造严重事件为 0。
- 客服/销售愿意承接。

### 11.3 暂停条件

任一情况出现，应暂停相关渠道或任务：

- 平台出现验证码、警告、访问异常、账号限制。
- High 渠道越界进入自动任务。
- 出现投诉或疑似骚扰。
- 勿扰客户进入触达队列。
- AI 输出缺少来源证据或疑似编造。
- 销售/客服反馈线索完全不可承接。

## 12. 风险与缓解

| 风险 | 表现 | 缓解 |
|---|---|---|
| 平台风控 | 验证码、封禁、警告 | 渠道配额、只读公开页、异常暂停 |
| 数据污染 | 大量重复、无联系方式、无证据 | raw/staging/core 分层，严进 core |
| AI 编造 | 补出不存在联系方式或意向 | Unknown/null 规则、审计、人工复核 |
| High 越界 | 自动读登录页、互动、触达 | 任务类型隔离、禁止动作校验 |
| 销售不承接 | 线索质量低 | C 类渠道质量池、复核标准、反馈闭环 |
| 成本失控 | LLM 和人工复核成本高 | 每日配额、缓存、批量摘要、成本看板 |
