## 1. 阶段目标

第五阶段的核心目标是把前几阶段已经跑通的“线索发现、线索抽取、线索完善、客户承接、Agent 服务化”继续推进到 **可治理的邮件自动回复闭环**。

本阶段要证明：

- 当前以文件形式存储的 Prompt 可以全部入库，后台可查看、草稿编辑、发布新版本、切换默认版本和回滚。
- Q&A、邮件回复模板、合规话术、车型/流程说明可以进入统一知识库底座，并按内容类型、业务场景、语言、风险等级和自动回复权限治理。
- 已发布知识可以异步生成 pgvector embedding，并在 `embedding_ready` 后进入 RAG 召回。
- `apps/admin` 可以管理 Prompt、邮件回复内容和 Q&A 知识库，不再依赖静态 seed 页面。
- `apps/agents` 新增独立 `EMAIL_REPLY` Agent/任务流，通过 `apps/api` 内部接口获取业务上下文、检索知识、生成建议回复、执行自动发送判断和提交发送请求。
- 白名单客户、固定 FAQ、首次邮件触达、低风险场景在通过硬拦截后可以自动发送邮件。
- 其他场景由 AI 生成建议回复，运营/销售人工确认、编辑后发送或标记已发送。
- AI 建议、最终发送内容、知识命中、Prompt 版本、模型、自动发送判断、人工编辑和发送结果均可审计、可复盘。

## 2. 继承规则与强制边界

第五阶段必须继承并严格执行以下规则：

- `apps/api` 是唯一业务数据权威，负责业务表写入、权限、审计、DNC/勿扰、自动发送准入、硬拦截和邮件发送提交。
- `apps/agents` 只负责 LangGraph/Agent 编排、LLM 调用、节点追踪、失败重试和结果回传，不直接写 core 业务表。
- `EMAIL_REPLY` Agent 不得绕过 `apps/api` 的自动发送规则直接调用邮件发送通道。
- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- 不对 DNC/勿扰客户自动发送。
- 不对 Watch/Invalid（对外显示 D/E 级）客户自动发送。
- 不在缺少已发布知识证据、语言不确定、知识召回不足、客户风险不明时自动发送。
- 不自动承诺价格、库存、付款、合同、发票、税务、物流、清关、交付周期、法律合规和出口管制事项。
- 所有 LLM 输出必须结构化，缺失字段输出 `Unknown`、`null` 或空数组，不得编造。
- 所有 AI 回复必须保留 Prompt 版本、模型、输入上下文摘要、知识命中、风险判断和审计记录。
- 后台页面必须接真实 API，不使用 seed 静态数据替代。
- 每个 Story 实施完成后必须执行两轮独立多维度评审，并记录结论、发现项和修正结果。

## 3. 本阶段产品范围

### 3.1 本阶段要做

- 扩展 `llm_prompt_templates` 治理能力，支持来源文件 hash、迁移批次、草稿编辑、发布、默认版本切换和回滚。
- 创建 Prompt 文件入库迁移脚本，扫描 `prompts/*.md` 并幂等写入数据库。
- 扩展知识库内容类型和业务字段，支持 Q&A、邮件回复模板、合规话术、车型说明、流程 SOP。
- 支持知识库草稿、审核、发布、下线、blocked、embedding 状态和重试。
- 支持发布后异步生成 pgvector embedding。
- 新增待回复邮件数据模型，支持后台手动新增和 API 导入，预留邮箱同步适配层。
- 新增 AI 回复建议草稿和最终发送内容分离的落库模型。
- 新增自动发送准入规则和硬拦截服务。
- 新增真实邮件发送适配层，支持预览、人工确认发送、白名单自动发送、失败重试、退信和线程关联。
- 在 `apps/admin` 新增 Prompt 模板管理、邮件回复内容管理、Q&A 知识库管理、邮件回复审核页面和质量指标页面。
- 在 `apps/agents` 新增 `EMAIL_REPLY` Agent/任务流。
- 新增邮件自动回复质量指标和 Go/No-Go 报告。

### 3.2 本阶段不做

- 不做完整多邮箱 IMAP/Gmail/Outlook 同步。
- 不做批量营销邮件。
- 不做复杂邮件营销活动管理。
- 不做全量自动发送。
- 不做自动报价、自动合同、自动付款指引。
- 不做库存和价格自动承诺。
- 不做社媒自动回复。
- 不做完整多租户复杂 RBAC。
- 不让 `apps/agents` 直接读写 `customers`、`outreach_records`、`knowledge_items` 等业务表。
- 不删除或废弃 `prompts/` 文件，文件保留为基线和灾备。

## 4. 总体业务流程

### 4.1 Prompt 入库治理链

```text
prompts/*.md
  -> 迁移脚本解析和计算 hash
  -> 写入 llm_prompt_templates 草稿或 active 默认版本
  -> apps/admin 查看 Prompt
  -> 编辑草稿
  -> 校验变量 / schema / 风险提示
  -> 发布新版本
  -> 切换默认版本
  -> Agent/API 运行时读取 active default
  -> 审计与回滚
```

### 4.2 知识库发布召回链

```text
运营创建 Q&A / 邮件回复模板 / 合规话术
  -> draft
  -> in_review
  -> published
  -> pending_embedding
  -> 异步生成 pgvector embedding
  -> embedding_ready
  -> active_for_retrieval
  -> EMAIL_REPLY RAG 检索
  -> 记录命中、采纳、编辑、风险和发送结果
```

### 4.3 邮件自动回复链

```text
email_messages 待回复邮件
  -> apps/api 获取客户上下文
  -> EMAIL_REPLY Agent 通过内部 API 检索知识
  -> LLM 生成建议回复
  -> apps/api 执行自动发送准入和硬拦截
  -> 可自动发送：调用邮件发送适配层
  -> 不可自动发送：进入人工确认队列
  -> outreach_records / email_reply_drafts / ai_audit_logs / metrics
```

### 4.4 人工确认链

```text
AI 建议回复
  -> apps/admin 邮件回复审核页面
  -> 运营/销售查看知识证据、风险原因和客户上下文
  -> 编辑回复正文
  -> 人工确认发送或标记已发送
  -> 保存最终发送内容
  -> 更新触达历史和知识质量指标
```

## 6. Prompt 入库与版本治理

### 6.1 迁移策略

迁移脚本扫描 `prompts/*.md`：

- 计算文件 hash。
- 根据文件名和 frontmatter 或约定推断 `task_type`。
- 将 system/user/schema 分段解析为数据库字段。
- 没有明确 schema 的 Prompt 需要补充默认输出契约或标记为 `validation_failed`。
- 同一 `source_file_path + source_file_hash + version` 不重复写入。
- 文件 hash 变化但已有 active 默认版本时，创建新草稿，不覆盖 active。

### 6.2 后台治理流程

```text
查看 Prompt
  -> 复制为草稿
  -> 编辑 system_prompt / user_prompt_template
  -> 运行变量和 schema 校验
  -> 测试样例预览
  -> 发布新版本
  -> 可选设为默认版本
  -> 审计记录
  -> 支持回滚
```

## 7. Q&A 与邮件回复知识库设计

### 7.1 内容类型

| 类型 | 用途 | 是否可自动发送 |
|---|---|---:|
| `qa_entry` | 常见问答 | 视 `auto_reply_allowed` |
| `email_reply_template` | 邮件回复模板 | 是，但需通过规则 |
| `compliance_phrase` | 合规话术 | 通常作为约束和补充 |
| `vehicle_product_note` | 车型/产品说明 | 默认需人工确认 |
| `process_sop` | 流程说明 | 低风险可自动 |

### 7.2 审核发布流程

```text
draft
  -> in_review
  -> published
  -> pending_embedding
  -> embedding_ready
  -> active_for_retrieval
```

禁用流程：

```text
active_for_retrieval
  -> archived / blocked
```

### 7.3 RAG 召回过滤

自动回复 RAG 必须至少过滤：

- `status = published/active_for_retrieval`
- `embedding_status = embedding_ready`
- `language = reply_language`
- `content_type in allowed_types`
- `business_scene matches`
- `risk_level != blocked`
- 自动发送场景要求 `auto_reply_allowed = true`

召回结果必须记录：

- `knowledge_item_id`
- `version`
- `similarity_score`
- `content_type`
- `business_scene`
- `filter_conditions`

## 8. 自动回复规则设计

### 8.1 自动发送准入

满足以下任一入口，才允许进入自动发送候选：

- 白名单客户。
- 固定 FAQ 命中。
- 首次邮件触达。
- 低风险场景。

低风险场景包括：

- 基础业务介绍。
- 合作流程说明。
- 车型资料说明，但不得承诺库存和价格。
- 联系信息确认。
- 下一步沟通预约。

### 8.2 硬拦截

以下任一命中，必须转人工：

- 投诉、威胁举报、要求停止联系、DNC/勿扰。
- 平台封禁、违规风险、隐私风险、来源合规不清。
- 付款收款、价格承诺、合同条款、发票、税务、法律合规。
- 制裁、出口管制、禁运、贸易路径不可行。
- 客户要求删除数据、拒绝沟通或强烈负面情绪。
- 涉及敏感个人信息或无法确认公开来源合法性。
- LLM 置信度低、知识库召回不足、缺少可引用知识证据。
- 客户或线索等级为 Watch/Invalid，对外显示 D/E 级。
- 语言识别不确定或缺少同语言 `embedding_ready` 内容。

## 14. Go / No-Go 指标

### 14.1 Go 条件

- Prompt 入库覆盖率 100%。
- 已发布知识 embedding ready 率 ≥ 95%。
- AI 回复建议生成成功率 ≥ 90%。
- 人工采纳率 ≥ 50%。
- 自动发送硬拦截准确执行率 100%。
- DNC/勿扰客户不自动发送 100%。
- D/E 客户不自动发送 100%。
- 投诉、封禁、违规事件为 0。
- 语言不确定、缺少同语言知识、知识召回不足时不自动发送 100%。

### 14.2 重跑小范围

- 功能跑通但采纳率低。
- 人工修改幅度高。
- embedding ready 不稳定。
- 邮件发送失败率偏高但无风险事件。
- 知识库召回质量不足。

### 14.3 暂停自动发送

发生以下任一情况，暂停自动发送：

- 投诉、封禁、违规事件。
- DNC/勿扰误发。
- D/E 客户误发。
- 硬拦截失效。
- 语言误判外发。
- 缺少知识证据仍自动发送。
- 自动回复承诺价格、库存、付款、合同或交付事项。

## 17. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| Prompt 后台编辑破坏输出契约 | Agent 失败或误发 | 草稿校验、Schema 校验、测试样例、发布新版本、回滚 |
| 未审核知识进入自动回复 | 合规风险 | 只有 published + embedding_ready + auto_reply_allowed 可自动召回 |
| LLM 编造回复内容 | 对外误导 | 必须引用知识证据，无知识命中不得自动发送 |
| 语言误判 | 外发质量差 | 语言不确定转人工，同语言知识缺失转人工 |
| DNC/勿扰误发 | 高风险事故 | 发送前硬拦截，DNC 100% 阻断 |
| Agent 绕过业务规则 | 数据和合规失控 | Agent 只通过 `apps/api` 内部接口行动 |
| 邮件发送失败或退信 | 影响客户体验 | 发送适配层、失败重试、退信记录和人工处理 |
| 知识库质量低 | 回复采纳率低 | 质量指标、人工编辑差异、建议下线队列 |

## 19. 两轮独立评审记录

### 19.1 第一轮评审：需求覆盖、架构边界与现有模型兼容

**结论：** 主体方案覆盖第五阶段头脑风暴 19 项决策，继承第四阶段 `apps/api` 与 `apps/agents` 的服务边界，产品流程、数据模型、后台页面、Agent、邮件发送、审计和 Go/No-Go 指标完整。

**发现项：**

- 文档中使用了 `published`、`active_for_retrieval`、`embedding_ready`、`stale_embedding` 等产品语义状态，但现有代码中 `KnowledgeItemStatus`、`KnowledgeReviewStatus`、`KnowledgeEmbeddingStatus` 已有枚举值，若后续 Story 直接扩展或替换枚举，可能破坏现有迁移和测试。

**修正结果：**

- 已新增“5.4 现有枚举兼容映射”小节，明确第五阶段产品语义如何映射到当前枚举。
- 已明确 `stale_embedding` 第五阶段优先通过新版本和 `metadata_json.embedding_stale=true` 实现，不强制扩展现有 embedding 枚举。
- 已明确新增 `EMAIL_REPLY_*` 任务类型必须通过独立 migration 扩展枚举，并补充兼容测试。

### 19.2 第二轮评审：端到端闭环、风险边界与实施可执行性

**结论：** 第一轮修正后，方案在端到端闭环、自动发送边界、人工确认、DNC/Watch/Invalid 拦截、多语言策略、真实 API 联调和实施优先级上保持一致，未发现新的实质阻塞问题。

**发现项：**

- 未发现新增实质阻塞问题。
- 仍需在后续 Epic/Story 拆分阶段，把邮件发送服务商配置、内部 API Key、Prompt 迁移脚本幂等性、embedding 异步任务和后台权限控制拆成可测试验收点。

**修正结果：**

- 无需追加实质修正。
- 已在验收清单和开发标准中保留真实 API、PostgreSQL、两轮评审、Agent 不直接写业务表等硬约束，供后续 Story 继承。
