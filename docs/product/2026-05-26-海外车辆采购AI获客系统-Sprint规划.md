# 海外车辆采购客户 AI 获客系统 Sprint 规划

创建日期：2026-05-26

## 1. Sprint 规划目标

本规划将已拆解的 Epic 和 Story 安排到可执行 Sprint 中，形成从 PoC 到 MVP 的交付节奏。

总体节奏：

- **PoC：2 周**
- **MVP：4 周**
- 总周期：**6 周**

核心原则：

1. PoC 先验证业务闭环，不提前开发 CRM。
2. MVP 只有在 PoC 复盘结论为“继续进入 MVP”后启动。
3. 每个 Sprint 必须有可验收产出。
4. 高风险渠道不进入自动化任务。
5. 所有 AI 判断保留证据和审计。

## 2. Sprint 总览

| Sprint | 周期 | 阶段 | 目标 | 关键产出 |
|---|---:|---|---|---|
| Sprint 0 | 2-3 天 | PoC 准备 | 搭建运营底座 | 飞书五张表、渠道风险表、关键词库、FAQ 初版 |
| Sprint 1 | 5-7 天 | PoC 采集 | 完成首批采集与 AI 清洗 | 50 条试采、AI 抽取、去重、分级、失败案例 |
| Sprint 2 | 4-5 天 | PoC 触达与复盘 | 完成人工触达并做阶段决策 | 100-200 条线索、10-20 条触达、PoC 复盘 |
| Sprint 3 | 1 周 | MVP 数据底座 | 搭建后端、数据库和同步 | FastAPI、PostgreSQL、飞书同步、规则模型 |
| Sprint 4 | 1 周 | MVP 移动工作台 | 实现移动端线索处理主流程 | 首页、线索池、详情页、AI 面板 |
| Sprint 5 | 1 周 | MVP 业务闭环 | 实现触达、车源、合规复核 | 触达记录、车源匹配、勿扰、C 级复核 |
| Sprint 6 | 1 周 | MVP 管理后台与试运行 | 实现后台、仪表盘和 QA | 管理后台、渠道风险、同步审计、上线 QA |

## 3. 团队角色与容量假设

| 角色 | Sprint 0-2 PoC | Sprint 3-6 MVP |
|---|---:|---:|
| 业务负责人 | 每日 30 分钟 | 每周 2-3 次评审 |
| 线索运营 | 每日 2 小时 | 每日 1 小时试用反馈 |
| AI/Agent 操作 | 每日 2 小时 | 每日 1 小时任务调试 |
| 客服 | 每日 1 小时 | 每日 1 小时试运行 |
| 出口销售 | 每日 1 小时 | 每日 1 小时试运行 |
| 合规/风控 | 每日 30 分钟 | 每周 2 次规则复核 |
| 后端工程 | 不参与或轻参与 | 全程参与 |
| 前端工程 | 不参与或轻参与 | 全程参与 |
| 测试/QA | 复盘阶段参与 | 每 Sprint 验收参与 |

## 4. Sprint 0：PoC 准备

**周期：** 2-3 天  
**目标：** 搭建 PoC 可以运行的最低运营底座。

### Story 范围

| Story | 名称 | 优先级 |
|---|---|---|
| E0-S1 | 创建飞书五张表 | P0 |
| E0-S2 | 创建 PoC 渠道风险登记表 | P0 |
| E0-S3 | 创建俄罗斯关键词库 | P0 |
| E9-S3 | FAQ 与异议处理库初版 | P0 |

### 任务清单

- [ ] 确认 PoC 负责人、备用人和每日投入时间。
- [ ] 创建飞书客户线索表。
- [ ] 创建飞书渠道来源表。
- [ ] 创建飞书车源/报价表。
- [ ] 创建飞书触达记录表。
- [ ] 创建飞书话术库表。
- [ ] 创建渠道风险登记表。
- [ ] 创建俄语/英语/中文关键词库初版。
- [ ] 创建客户 FAQ/异议处理库初版。
- [ ] 确认第一批轻量车源/报价样例。
- [ ] 确认俄罗斯贸易合规红线初版。

### 验收标准

- 飞书五张表可录入、可筛选、可分配。
- 渠道风险表覆盖低、中、高、禁止四类。
- 关键词库至少覆盖客户、车辆、渠道、排除、地区五类关键词。
- FAQ 覆盖价格、车况、物流、付款信任、合作模式。
- PoC 可以进入真实线索试采。

### 风险

- 如果负责人和每日投入未确认，不进入 Sprint 1。
- 如果合规红线未确认，C 级触达和报价相关动作暂不执行。

## 5. Sprint 1：PoC 采集与 AI 清洗

**周期：** 5-7 天  
**目标：** 完成首批 50 条低/中风险候选线索试采，并验证 AI 抽取、去重、分级是否可用。

### Story 范围

| Story | 名称 | 优先级 |
|---|---|---|
| E2-S1 | AI 公开网页信息抽取 | P0 |
| E2-S2 | AI 线索分级建议 | P0 |
| E1-S3 | 实现基础去重规则 | P0 |
| E9-S2 | 失败案例库 | P1 |

### 任务清单

- [ ] 只选择 Low/Medium 风险渠道采集。
- [ ] 完成首批 50 条候选线索。
- [ ] 每条线索记录来源链接和证据备注。
- [ ] AI 抽取客户名称、城市、联系方式、经营信号。
- [ ] AI 输出 A/B/C/Invalid/Watch 建议。
- [ ] 人工复核 AI 建议。
- [ ] 标记重复线索。
- [ ] 记录失败案例和无效原因。
- [ ] 计算首批线索的 B 级比例。
- [ ] 暂停明显无效渠道。

### 验收标准

- 至少完成 50 条候选线索。
- 每条线索有来源链接。
- AI 可输出分级建议和推荐原因。
- 人工可复核并覆盖 AI 判断。
- 形成第一版失败案例库。

### Sprint 1 通过条件

- 首批 50 条中存在可继续扩展的有效渠道。
- 没有出现平台投诉、账号限制或违规采集。
- AI 抽取结果可被线索运营接受。

## 6. Sprint 2：PoC 扩展、触达与复盘

**周期：** 4-5 天  
**目标：** 扩展到 100-200 条线索，完成人工触达小样本，并做 Go/No-Go 决策。

### Story 范围

| Story | 名称 | 优先级 |
|---|---|---|
| E4-S1 | 生成俄语触达草稿 | P0 |
| E4-S2 | 记录触达结果 | P0 |
| E0-S4 | 建立 PoC 复盘模板 | P0 |
| E7-S1 | 渠道与线索仪表盘手工版 | P0 |

### 任务清单

- [ ] 扩展表现较好的渠道。
- [ ] 累计完成 100-200 条候选线索。
- [ ] 对 B/C 级线索生成俄语触达草稿。
- [ ] 人工审核触达草稿。
- [ ] 选择 10-20 条 B/C 级线索人工触达。
- [ ] 记录触达渠道、发送人、话术版本、回复状态。
- [ ] 客户拒绝后标记勿扰。
- [ ] 统计 B 级比例、回复率、无效率。
- [ ] 统计每条有效线索成本和每条回复成本。
- [ ] 输出 PoC 复盘报告。

### 验收标准

- 100-200 条线索中，B 级比例目标为 30%-50%。
- 至少完成 10-20 条人工触达。
- 有真实回复或明确负反馈。
- 没有账号封禁、平台投诉、明显违规采集或批量骚扰。
- 客服/销售认可线索字段、话术和交付方式。

### Go/No-Go 决策

PoC 结束时选择以下结论之一：

1. **进入 MVP**：渠道可持续、B 级比例达标、触达有反馈、风险可控。
2. **重跑 PoC**：渠道或画像需调整，但方向仍有价值。
3. **暂停俄罗斯方向**：线索质量、触达反馈或交易路径不可行。

|---|---|---|
| E1-S1 | 定义客户主体模型 | P0 |
| E1-S2 | 实现飞书到 PostgreSQL 单向同步 | P0 |
| E6-S1 | 渠道风险等级配置 | P0 |
| E6-S2 | 勿扰机制 | P0 |

### 任务清单

- [ ] 初始化 FastAPI 项目。
- [ ] 初始化 PostgreSQL 数据库。
- [ ] 创建客户主体表。
- [ ] 创建来源记录表。
- [ ] 创建联系方式表。
- [ ] 创建触达记录表。
- [ ] 创建 AI 审计日志表。
- [ ] 创建合规配置表。
- [ ] 实现飞书 API 单向同步。
- [ ] 实现 A/B/C 线索规则。
- [ ] 实现勿扰状态。
- [ ] 实现渠道风险配置读取。
- [ ] 编写后端基础测试。

### 验收标准

- 飞书线索能同步进入 PostgreSQL。
- 客户主体可挂多个来源和多个联系方式。
- 勿扰状态同步后不丢失。
- High/Forbidden 渠道不能被自动任务选中。
- 后端健康检查通过。

### 风险

- 如果 Feishu API 权限未准备，先用 CSV 导入替代，但必须保留同步接口设计。

## 9. Sprint 5：MVP 业务闭环

**周期：** 1 周  
**目标：** 实现触达记录、车源匹配和 C 级合规复核，让线索真正可交付。

### Story 范围

| Story | 名称 | 优先级 |
|---|---|---|
| E4-S2 | 记录触达结果 | P0 |
| E5-S1 | 维护轻量车源/报价表 | P0 |
| E5-S2 | 线索与车源匹配 | P1 |
| E6-S3 | C 级线索合规复核 | P0 |

### 任务清单

- [ ] 实现触达记录创建。
- [ ] 支持回复状态：已发送、已回复、拒绝、无回复、错误联系方式。
- [ ] 客户拒绝后自动标记勿扰。
- [ ] 实现车源/报价表。
- [ ] 实现线索详情中的车源推荐。
- [ ] 实现报价前合规复核状态。
- [ ] 未复核 C 级线索不能标记为已报价。
- [ ] 销售可记录下一步动作。

### 验收标准

- B 级线索可由客服触达并记录结果。
- C 级线索可由销售查看需求和车源匹配。
- 拒绝联系客户不会再次进入触达队列。
- 报价前合规复核节点生效。

## 11. 跨 Sprint 质量门

每个 Sprint 结束必须检查：

- [ ] 是否新增了平台风险。
- [ ] 是否触碰自动社交私信/加好友/登录平台批量采集边界。
- [ ] 是否保留来源证据。
- [ ] 是否保留 AI 输入、输出、模型、时间和执行结果。
- [ ] 勿扰客户是否被排除。
- [ ] C 级线索是否保留合规复核节点。
- [ ] 客服/销售是否能理解当前交付物。

## 12. 发布节奏

### PoC 发布

PoC 不是系统发布，而是运营验证发布。

PoC 发布内容：

- 飞书表格。
- 关键词库。
- 风险登记表。
- AI prompts。
- 小样本触达记录。
- PoC 复盘报告。

### MVP 内测发布

MVP 内测发布对象：

- 线索运营。
- 客服。
- 出口销售。
- 管理者。
- 合规/风控。

MVP 内测发布条件：

- B 级线索可以交付客服。
- C 级线索可以交付销售。
- 勿扰客户不会再次触达。
- 高风险渠道被阻断。
- AI 建议有审计记录。
- 管理后台有可用指标。

## 13. 暂不进入本轮 Sprint 的事项

- 自动社交私信。
- 自动加好友。
- 平台登录后批量采集。
- 反爬规避。
- 多国家完整扩展。
- 复杂黑箱评分模型。
- ERP/库存系统深度集成。
- 多模型路由和评测平台。
- 自动平台政策监控。
