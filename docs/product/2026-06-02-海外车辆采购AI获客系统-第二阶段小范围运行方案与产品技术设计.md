## 1. 阶段目标

第二阶段的核心目标是把第一阶段“LLM 输出 JSON 校验和入库”的间接模式，升级为“系统内真实调用 LLM Provider、自动定时发现来源、人工审核来源风险、自动抽取消费、入库审计和移动端管理”的闭环。

本阶段要证明：

- 系统可以稳定连接真实 LLM Provider，默认 DeepSeek，预留多 Provider 切换。
- Source Discovery Agent 可以按计划持续新增来源候选。
- 来源候选先进入 `lead_source_candidates`，不直接污染正式 `lead_sources`。
- Low/Medium 来源可按规则自动进入抽取，High 来源必须人工审核后只读抽取。
- `LEAD_EXTRACTION` 从审核通过的来源池消费来源，自动进行公开文本读取、LLM 抽取、分级、入库和审计。
- 移动端新增来源审核、来源详情、Agent 手动调用和任务运行状态页面。
- 所有 AI 输出、来源证据、prompt 版本、模型、输入输出和任务状态可追溯。

## 2. 继承规则与强制边界

本阶段必须继承并严格执行以下规则：

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- 不自动触达 High 风险来源。
- Forbidden 来源不得进入自动抽取。
- Invalid 和 Watch 不进入触达队列。
- 勿扰客户不得再次进入触达队列。
- C 级线索在报价、合同或实质交易前必须合规复核。
- LLM 缺失字段必须输出 `Unknown`、`null` 或空数组，不得编造。
- 所有 AI 输出必须保留来源证据和审计记录。

## 3. 第二阶段产品范围

### 3.1 本阶段要做

- 修正并实现真实 LLM Provider 连接。
- 新增 `LLMClient` 统一抽象，默认 DeepSeek。
- 新增 `llm_prompt_templates` 管理 prompt/schema。
- 新增 `agent_task_runs` 作为 Agent 任务运行审计事实表。
- 新增 `lead_source_candidates` 来源候选池。
- 新增 Source Discovery Agent，持续自动发现来源。
- 新增 APScheduler 定时任务，结合 Redis lock 和 PostgreSQL 审计。
- 改造 `LEAD_EXTRACTION`，从审核通过的来源候选池自动消费。
- 移动端新增来源审核队列、来源详情、审核动作和 Agent 手动调用页面。
- 管理后台新增 LLM Provider、Prompt、任务、成本和风险监控入口。

### 3.2 本阶段不做

- 不做自动私信、自动评论、自动加好友、自动关注。
- 不做登录态采集或需要绕过验证码的动作。
- 不做完整 CRM 替换。
- 不做复杂销售自动化。
- 不把 prompt/schema 编辑权限开放给普通运营。
- 不让 High 风险来源绕过人工审核。

## 4. 总体业务流程

```text
渠道计划/关键词/行业链路
  -> Source Discovery Agent 定时发现来源
  -> lead_source_candidates 来源候选池
  -> Low/Medium 自动通过抽取准入，High 进入移动端人工审核
  -> approved 来源进入 LEAD_EXTRACTION 队列
  -> 公开页面读取和证据快照
  -> LLM 抽取客户信息
  -> LLM 分级 + 规则硬校验
  -> staging_leads
  -> 人工复核晋级 core customers / lead_sources
  -> 客服或出口销售交付
```

## 5. 渠道与风险策略

第二阶段来源发现采用 A + B + C 并行：

| 类型 | 来源 | 默认策略 |
|---|---|---|
| A | 官网、公开目录、搜索引擎、地图公开结果 | Low/Medium 可自动进入抽取 |
| B | 公开社媒、公开视频页、平台公开商户页 | High 默认待审，只读，不触达 |
| C | 协会、展会、B2B、物流/清关、检测维修、进口车生态 | 作为高质量来源池，按风险分层 |

风险准入规则：

- Low：`review_status=auto_approved`，`approved_for_extraction=true`。
- Medium：`review_status=auto_approved`，`approved_for_extraction=true`，但受配额和暂停规则控制。
- High：`review_status=high_risk_review`，`approved_for_extraction=false`。
- Forbidden：`review_status=rejected`，`approved_for_extraction=false`。

## 6. 数据模型设计

### 6.1 `lead_source_candidates`

用途：承载 LLM Source Discovery Agent 自动发现的来源候选。该表不绑定客户，不等于正式 `lead_sources`。

字段分组：

- 基础来源：`id`、`source_url`、`normalized_domain`、`platform`、`channel_name`、`country`、`city`
- 风险审核：`risk_level`、`review_status`、`approved_for_extraction`、`reviewer_id`、`review_note`、`reviewed_at`
- 发现证据：`discovery_method`、`discovery_query`、`discovery_reason`、`evidence_note`、`evidence_links`
- LLM 输出：`llm_provider`、`llm_model`、`llm_output_json`、`confidence_score`
- 队列状态：`extraction_status`、`last_extracted_at`、`next_retry_at`、`retry_count`
- 去重归并：`dedupe_key`、`duplicate_of_id`、`is_duplicate`
- 审计时间：`created_at`、`updated_at`、`created_by_task_run_id`

### 6.2 `llm_prompt_templates`

用途：管理 Source Discovery、Lead Extraction、Lead Grading 等任务的 prompt 和 output schema。

核心字段：

- `id`
- `name`
- `task_type`
- `provider`
- `model`
- `system_prompt`
- `user_prompt_template`
- `output_schema_json`
- `version`
- `status`
- `is_default`
- `created_by`
- `created_at`
- `updated_at`

状态：`draft`、`active`、`paused`、`archived`。

### 6.3 `agent_task_runs`

用途：作为所有 Agent 任务运行的审计事实表。

核心字段：

- `id`
- `task_type`
- `status`
- `trigger_source`
- `input_json`
- `output_summary_json`
- `llm_provider`
- `llm_model`
- `prompt_template_id`
- `prompt_version`
- `token_usage_json`
- `latency_ms`
- `error_message`
- `retry_count`
- `started_at`
- `finished_at`
- `created_at`
- `updated_at`

## 7. LLM Provider 接入方案

第二阶段新增统一 `LLMClient`：

- 默认 Provider：DeepSeek。
- 接口风格：OpenAI-compatible。
- 预留 Provider：OpenAI、Claude、Qwen、Gemini。
- 业务服务只调用 `LLMClient`，不直接依赖某家 SDK。

推荐配置：

```env
LLM_PROVIDER=deepseek
LLM_API_KEY=sk-xxxxx
LLM_BASE_URL=https://api.deepseek.com/v1
LLM_DEFAULT_MODEL=deepseek-chat
LLM_SOURCE_DISCOVERY_MODEL=deepseek-chat
LLM_EXTRACTION_MODEL=deepseek-chat
LLM_GRADING_MODEL=deepseek-chat
AGENT_SCHEDULER_ENABLED=false
```

Fallback 规则：

- 网络、超时、限流可 fallback。
- schema 校验失败不 fallback，进入人工复核或 prompt 修正。
- 输出命中风险规则不 fallback，进入风控处理。
- 输出疑似编造不 fallback，阻断并审计。

## 8. Source Discovery Agent 方案

职责边界：

- 只发现来源。
- 不抽取客户。
- 不触达。
- 不生成私信。
- 不绕过平台限制。

输出必须符合结构化 schema：

```json
{
  "task_type": "SOURCE_DISCOVERY",
  "country": "Russia",
  "city": "Moscow",
  "channel_strategy": "official_website_public_directory_search_engine",
  "candidates": [
    {
      "source_url": "https://example.com/dealers",
      "platform": "official_website",
      "channel_name": "dealer_directory",
      "country": "Russia",
      "city": "Moscow",
      "risk_level": "Low",
      "discovery_method": "keyword_search",
      "discovery_query": "автосалон импорт авто Москва",
      "discovery_reason": "页面公开展示车商目录，可能包含二级经销商线索",
      "evidence_note": "公开页面包含 dealer / auto sales / contact 相关信息",
      "evidence_links": ["https://example.com/dealers"],
      "confidence_score": 0.72,
      "recommended_review_status": "auto_approved",
      "approved_for_extraction": true
    }
  ],
  "blocked_candidates": [
    {
      "source_url": "https://blocked.example.com",
      "risk_level": "Forbidden",
      "blocked_reason": "需要登录或违反渠道规则"
    }
  ]
}
```

## 9. 自动调度方案

第二阶段采用：

```text
APScheduler + Redis 锁/状态 + PostgreSQL agent_task_runs
```

推荐任务：

- `source_discovery_hourly`：每小时发现来源。
- `lead_extraction_interval`：每 15-30 分钟消费已审核来源。
- `retry_failed_tasks`：每 10 分钟处理可重试失败任务。
- `channel_health_check_daily`：每天复盘渠道失败率和风险事件。
- `llm_cost_rollup_daily`：每天汇总 LLM 成本。

初始配额：

- 来源发现任务每小时 1 次。
- 每次新增候选来源 20-50 条。
- 每日新增候选来源上限 300-500 条。
- 每日进入自动抽取来源 100-200 条。
- High 风险待审来源每日 20-50 条。
- 单渠道连续失败 3 次自动暂停。
- 单渠道出现 1 次风险事件进入人工复核。

## 10. 移动端页面设计

原型基础沿用 `prototypes/mvp-mobile-agent/` 的移动端优先设计方向。

新增页面：

- 来源候选队列：按风险、状态、国家、城市、平台、渠道筛选候选来源。
- 来源详情页：展示 URL、domain、平台、风险、证据、LLM 原始输出摘要和任务审计。
- 审核动作区：通过并进入抽取、驳回、标记高风险、暂停渠道、送人工二次复核、添加备注。
- Agent 手动调用页：选择任务类型、国家、城市、渠道策略、prompt 版本、运行上限并启动任务。
- 任务运行详情：展示状态、输出数量、失败原因、重试次数、成本和审计引用。

移动端原则：

- 通过来源只表示允许抽取，不表示允许触达。
- 风险等级和证据摘要必须在列表和详情中可见。
- 审核动作必须记录操作者、备注和时间。
- Agent 手动调用必须走同一套 `agent_task_runs` 审计链路。

## 11. 后端 API 设计

建议新增 API 模块：

- `GET /lead-source-candidates`
- `GET /lead-source-candidates/{id}`
- `POST /lead-source-candidates/{id}/approve-for-extraction`
- `POST /lead-source-candidates/{id}/reject`
- `POST /lead-source-candidates/{id}/mark-high-risk`
- `POST /lead-source-candidates/{id}/pause-channel`
- `GET /llm-prompt-templates`
- `POST /agent-tasks/source-discovery/run`
- `POST /agent-tasks/lead-extraction/run`
- `GET /agent-task-runs`
- `GET /agent-task-runs/{id}`

所有接口必须：

- 返回审计 ID。
- 校验风险状态。
- 禁止 Forbidden 来源进入抽取。
- 禁止未审核 High 来源进入自动抽取。
- 不提供自动触达动作。

## 15. 推进计划摘要

推荐按以下顺序实施：

1. 数据底座：`llm_prompt_templates`、`agent_task_runs`、`lead_source_candidates`。

9. 指标看板：来源发现、审核、抽取、失败、成本和风险。
10. 联调验收：真实 PostgreSQL、Redis、LLM、移动端 H5 和后端 API。
