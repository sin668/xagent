## 1. 阶段目标

第三阶段的核心目标是把第二阶段已经形成的“来源发现 -> 线索抽取 -> staging 入库”能力，继续推进到“线索完善 -> 客户晋级 -> 客户承接 -> 跟进记录 -> 质量清洗”的业务闭环。

本阶段要证明：

- 线索池中的线索可以通过 AI 深挖补全和人工补录变成可承接客户。
- 客户只有满足最低准入条件后，才允许晋级到 `customers`、`lead_sources`、`contact_methods`。
- 客户管理工作台可以展示线索完善后的客户资料、联系方式、来源证据、触达历史、意向车型和跟进记录。
- Watch/Invalid 线索可以通过清洗 Agent 生成去重、归并、恢复、确认无效和放弃建议，避免线索池长期堆积。
- 第三阶段新增独立 Agent 项目，局部试点 LangGraph，只承载深挖线索 Agent 和清洗 Agent，不重构第二阶段已运行链路。
- 有效客户承接率可以作为第三阶段核心成功指标。

## 2. 继承规则与强制边界

本阶段必须继承并严格执行以下规则：

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- 不自动触达 High 风险来源。
- Forbidden 来源不得进入自动抽取，不得作为客户晋级关键来源。
- Invalid 和 Watch 不进入触达队列。
- 勿扰客户不得再次进入触达队列。
- C 级客户在报价、合同、付款、物流、清关、交付周期等动作前必须合规复核。
- LLM 缺失字段必须输出 `Unknown`、`null` 或空数组，不得编造。
- 所有 AI 输出必须保留来源证据和审计记录。
- 客户触达仍为人工确认模式，系统可生成草稿和记录历史，但不得自动发送。
- 手动补录、字段采纳、客户晋级、清洗恢复、重复归并都必须审计。

## 3. 第三阶段产品范围

### 3.1 本阶段要做

- 新增线索完善区，承载 AI 深挖和人工补录的字段级候选。
- 新增“深挖线索”人工触发动作。
- 新增字段级采纳、修改、拒绝能力。
- 新增手动补录和手动准入能力。
- 新增客户晋级服务，将满足准入条件的线索晋级到 `customers`、`lead_sources`、`contact_methods`。
- 新增客户管理工作台、客户详情、客户跟进记录。
- 新增客户意向车型独立建模。
- 新增客户跟进记录独立建模。
- 新增 Watch/Invalid 线索清洗 Agent，定时生成清洗建议，人工确认执行。
- 新增独立 Agent 项目，局部试点 LangGraph。
- 新增有效客户承接率、补全成功率、清洗采纳率、风险事件等指标。

### 3.2 本阶段不做

- 不做完整 CRM 商机阶段管理。
- 不做报价单、合同、收款、物流履约。
- 不做自动邮件、短信、社媒发送。
- 不做自动加好友或自动私信。
- 不做销售自动化漏斗全套系统。
- 不做多国家大规模推广。
- 不对 High 风险来源做自动触达。
- 不引入完整复杂 RBAC，只做最小角色权限边界。
- 不重构现有 Source Discovery Agent 和 Lead Extraction Agent。

## 4. 总体业务流程

### 4.1 线索完善晋级链

```text
staging_leads
  -> 线索详情 / 线索完善队列
  -> 人工点击“深挖线索”或手动补录
  -> AI/人工补全字段候选
  -> 字段级采纳、修改、拒绝
  -> 最低准入校验
  -> 建议晋级客户
  -> 人工确认晋级
  -> customers + lead_sources + contact_methods
  -> 客户工作台
  -> 分配客服/销售
  -> 记录首次跟进
```

### 4.2 Watch/Invalid 清洗治理链

```text
Watch / Invalid / 疑似重复 / 低质量 staging leads
  -> Lead Cleanup Agent 定时扫描
  -> 生成清洗建议
  -> 人工确认归并 / 恢复 / 确认无效 / 放弃
  -> 更新主线索或客户字段
  -> 保留重复与清洗审计
  -> 清洗指标统计
```

### 4.3 Agent 边界

```text
Source Discovery Agent -> lead_source_candidates
Lead Extraction Agent  -> staging_leads
Deep Enrichment Agent  -> lead_enrichment_results / lead_enrichment_field_candidates
Cleanup Agent          -> lead_cleanup_runs / lead_cleanup_suggestions
Promotion Service      -> customers / lead_sources / contact_methods
Customer Workbench     -> customers + intents + followups + outreach history
```

## 5. 客户晋级准入规则

第三阶段采用 **平衡准入**。

线索晋级到 `customers` 的最低条件：

- 客户名称。
- 国家/城市。
- 至少一个联系方式。
- 来源证据。
- 勿扰校验通过。

客户类型、经营信号、意向车型可以后续继续补全，但必须在客户工作台中显示为“待补全”或“待确认”，不能伪装为已确认事实。

手动准入规则：

- 运营或管理员可以执行手动准入。
- 手动准入必须满足最低准入条件。
- 手动准入不得绕过勿扰、Forbidden、High 风险、C 级合规等硬规则。
- 手动准入必须记录操作人、时间、字段来源、备注和审计日志。

## 6. 线索完善设计

### 6.1 AI 深挖补全

触发方式：

- 必须由人工在线索详情或线索完善队列点击 **“深挖线索”**。
- 不做后台无差别自动深挖。

输入：

- 当前客户名称、别名、国家、城市。
- 当前联系方式。
- 当前来源链接和证据摘要。
- 当前 AI 分级、缺失字段、客户类型。
- 人工备注和人工指定关键词。
- 允许搜索渠道范围和风险边界。

处理：

```text
load_staging_lead
  -> build_search_keywords
  -> filter_allowed_channels
  -> search_public_sources
  -> read_public_pages
  -> extract_field_candidates
  -> validate_schema_and_evidence
  -> write_enrichment_candidates
  -> recommend_action
```

输出：

- 搜索关键词和查询语句。
- 命中的公开来源链接。
- 字段级候选值。
- 字段级证据备注。
- 置信度。
- 缺失字段。
- 是否满足最低晋级条件。
- 推荐动作：建议晋级、继续补全、降级、放弃、人工复核。
- LLM prompt 版本、模型、token、耗时和任务审计。

配额：

- 每条线索每天最多 1-2 次深挖。
- 每次深挖最多生成 3-5 个搜索查询。
- 每次深挖最多读取 5-10 个公开链接。
- Watch / Invalid / 勿扰 / Forbidden 线索不允许深挖。
- 深挖失败或无有效补全结果时，必须记录失败原因。

### 6.2 人工补录

人工补录来源类型：

- `manual_public_info`：人工从公开资料补录。
- `manual_customer_reply`：客户回复产生的信息。
- `manual_business_note`：业务人员线下或沟通备注。
- `unknown`：未知来源，不建议作为晋级关键证据。

人工补录要求：

- 必须记录补录人、补录时间、字段来源和备注。
- 关键字段建议带来源链接或证据说明。
- 人工补录不得覆盖审计历史。
- 人工补录不得绕过合规硬门禁。

### 6.3 字段级采纳

字段级候选支持：

- 采纳。
- 修改后采纳。
- 拒绝。
- 标记待确认。

只有被采纳的字段才能参与客户晋级。未采纳字段保留在补全结果中，用于审计和失败案例复盘，但不得污染 `customers` core 数据。

## 7. Watch/Invalid 线索清洗 Agent

### 7.1 目标

新增 Lead Cleanup Agent，对线索池中的 Watch / Invalid / 疑似重复 / 低质量线索进行清洗，减少低质量数据堆积，并发现可恢复线索。

### 7.2 处理范围

- 强重复线索。
- 疑似重复线索。
- 可恢复 Watch 线索。
- 确认无效线索。
- 重复线索补充主记录。

### 7.3 输出建议类型

`suggestion_type` 建议包括：

- `strong_duplicate`
- `possible_duplicate`
- `merge_contact_method`
- `merge_source_evidence`
- `restore_from_watch`
- `confirm_invalid`
- `mark_abandoned`
- `needs_manual_review`

### 7.4 执行方式

采用 **定时生成建议 + 人工确认执行**。

- Agent 可定时扫描 Watch、Invalid、疑似重复、低质量线索。
- Agent 自动生成去重、归并、恢复、确认无效、放弃等清洗建议。
- 关键动作必须由人工确认后执行。
- Agent 不得自动删除线索。
- Invalid 恢复必须人工确认。
- 勿扰状态在归并时必须传播到主记录。
- 所有建议、确认和执行结果必须审计。

## 9. 客户状态与团队交付

建议客户状态：

```text
待分配
  -> 待首次触达
  -> 已触达待回复
  -> 有回复待判定
  -> 有意向待销售
  -> 销售跟进中
  -> 成交 / 长期跟进 / 暂停 / 勿扰 / 无效
```

交付策略：

- A 级客户：进入客服队列，优先补充需求和完成首次触达。
- B 级客户：进入客服队列，重点确认车型、采购频率、预算和合作意向。
- C 级客户：进入销售候选队列，但报价、合同或实质交易前必须完成合规复核。
- Watch / Invalid 不进入客户工作台和触达队列。
- 勿扰客户不得再次进入触达队列。

团队队列：

- 客服队列：A/B 级、待首次触达、待回复、待补充需求。
- 销售队列：C 级、有明确车型/数量/预算/采购周期的客户。
- 合规队列：C 级报价前、High 来源边界、客户要求价格/物流/清关/交付承诺时。
- 运营队列：待补全、疑似重复、来源证据不足、深挖失败。

## 12. Agent 项目与 LangGraph 局部试点

### 12.1 项目定位

第三阶段新建独立 Agent 项目，建议目录：

```text
apps/agents
  -> pyproject.toml
  -> app/
     -> graphs/
        -> deep_enrichment.py
        -> lead_cleanup.py
     -> schemas/
     -> tools/
     -> adapters/
     -> tests/
```

或：

```text
apps/agent-runtime
```

### 12.2 试点范围

优先承载：

- `Lead Deep Enrichment Agent`
- `Lead Cleanup Agent`

暂不迁移：

- `Source Discovery Agent`
- `Lead Extraction Agent`
- APScheduler 调度。
- Redis lock。
- `agent_task_runs` 审计。

### 12.3 主控边界

- 外层调度、任务审计、数据库写入、人工确认、风险硬门禁仍由现有 `apps/api` 主控。
- LangGraph 只负责新增 Agent 内部多步骤流程表达、分支、状态流转和人工等待点。
- Agent 项目不直接绕过后端写 core 表。
- Agent 项目输出结构化结果，由 `apps/api` Service 层校验后写入 PostgreSQL。
- Agent 项目不得自动晋级客户、自动归并客户、自动恢复 Invalid、自动触达客户。

### 12.4 Deep Enrichment Graph

建议图：

```text
load_lead
  -> build_keywords
  -> search_public_sources
  -> read_public_pages
  -> extract_candidates
  -> validate_evidence
  -> write_enrichment_candidates
  -> recommend_action
```

### 12.5 Cleanup Graph

建议图：

```text
load_watch_invalid
  -> detect_duplicates
  -> classify_invalid_reason
  -> find_restore_candidates
  -> write_cleanup_suggestions
  -> wait_human_review
```

### 12.6 与现有 LLM 编排选型的关系

该方案落实 `docs/architecture/2026-06-03-LLM-Agent服务编排选型分析.md` 的建议：不是全面重构，而是在新增复杂 Agent 中局部试点 LangGraph。

好处：

- 风险隔离，不影响第二阶段已跑通的来源发现和线索抽取。
- 深挖与清洗天然多步骤、多分支、有人审等待，适合图流程。
- 保留现有 FastAPI、PostgreSQL、Redis lock、APScheduler 和审计体系。
- 如果试点稳定，再评估是否迁移 Lead Extraction 或 Source Discovery。

## 13. 权限模型

第三阶段采用最小角色权限模型。

### 13.1 运营

- 可触发深挖线索。
- 可人工补录。
- 可字段级采纳/拒绝。
- 可提交或执行符合条件的手动准入。
- 可处理普通清洗建议。

### 13.2 客服

- 可查看已分配客户。
- 可记录人工触达和跟进。
- 可补充客户反馈和初步意向车型。
- 可标记勿扰。
- 不可绕过 C 级合规复核。

### 13.3 销售

- 可查看 C 级和销售候选客户。
- 可维护意向车型、采购数量、预算、交付关注点。
- 可记录销售跟进。
- 报价/合同前必须触发并通过合规复核。

### 13.4 合规/管理员

- 可审核 C 级复核。
- 可处理 High/Forbidden 边界争议。
- 可恢复 Invalid。
- 可确认高风险重复归并。
- 可取消勿扰。
- 可查看完整审计。

关键边界：

- 恢复 Invalid：必须合规/管理员确认。
- 标记勿扰：运营、客服、销售均可做，立即生效。
- 取消勿扰：必须合规/管理员确认。
- C 级报价/合同前复核：必须合规/管理员通过。
- 强重复归并：运营可确认；疑似重复和客户级归并建议管理员确认。

## 14. 合规与审计

### 14.1 硬阻断门禁

- 勿扰客户不得生成触达草稿、不得记录新的主动触达、不得进入客服/销售触达队列。
- Forbidden 来源不得作为客户晋级关键来源。
- C 级客户在报价、合同、付款、物流、清关、交付周期等动作前必须完成合规复核。
- 系统不得自动发送邮件、短信、社媒消息或自动加好友。
- 客户晋级必须有来源证据。
- AI 深挖补全字段必须有字段级证据或人工来源说明。
- 手动补录、字段采纳、字段修改、客户晋级、客户放弃必须记录操作人和时间。

### 14.2 审计事件

建议审计事件：

- `lead_deep_enrichment_started`
- `lead_enrichment_field_accepted`
- `lead_enrichment_field_rejected`
- `lead_promoted_to_customer`
- `lead_marked_watch_invalid_abandoned`
- `lead_cleanup_suggestion_created`
- `lead_cleanup_suggestion_approved`
- `lead_cleanup_suggestion_executed`
- `customer_assigned`
- `customer_outreach_draft_generated`
- `customer_manual_outreach_recorded`
- `customer_do_not_contact_marked`
- `customer_compliance_review_requested`
- `customer_compliance_review_passed_or_rejected`

## 15. 指标与验收

### 15.1 核心指标

第三阶段核心成功指标为：

```text
有效客户承接率 = 已被客服/销售接收并完成首次跟进的晋级客户数 / 晋级客户总数
```

### 15.2 辅助指标

线索完善效率：

- 深挖触发线索数。
- 深挖成功补全率。
- 字段级采纳率。
- 人工补录完成率。
- 平均补全耗时。

客户晋级质量：

- `staging_leads -> customers` 晋级率。
- 晋级客户中联系方式完整率。
- 晋级客户中有意向车型比例。
- 晋级客户重复率/误晋级率。
- 放弃/Watch/Invalid 原因分布。

团队承接效率：

- 客户分配及时率。
- 首次人工触达及时率。
- 客户回复率。
- C 级销售承接率。
- 跟进记录完整率。

清洗治理：

- 清洗建议生成数。
- 清洗建议采纳率。
- 重复线索归并率。
- Watch 恢复率。
- Invalid 确认率。

风险与合规：

- 勿扰误入队列次数，目标 0。
- 自动触达违规次数，目标 0。
- C 级未复核报价/合同次数，目标 0。
- 无证据字段采纳次数，目标 0。
- High/Forbidden 违规晋级次数，目标 0。

### 15.3 Go / 重跑 / 暂停

Go：

- 深挖补全有效。
- 客户晋级质量可接受。
- 客服/销售愿意承接。
- 有效客户承接率达标。
- 风险事件为 0。

重跑：

- 深挖关键词或渠道质量不足，但流程和数据模型可行。
- 清洗建议准确率不足，但人工确认机制有效。

暂停：

- 客户晋级污染严重。
- 销售不愿承接。
- 风险门禁失效。
- LLM 成本不可控。
- 勿扰、Forbidden、C 级复核等硬规则出现严重违规。
