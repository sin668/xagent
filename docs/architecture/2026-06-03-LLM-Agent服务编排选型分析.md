# LLM Agent 服务编排选型分析

创建日期：2026-06-03  
适用阶段：第二阶段小范围运行及后续 MVP 增强  
候选方案：n8n、LangChain / LangGraph、AutoGen、CrewAI、OpenAI Agents SDK、Pydantic AI、Temporal、Microsoft Agent Framework  
结论等级：建议采用

## 1. 结论摘要

当前项目如果要引入 LLM Agent 服务编排，推荐选择：

**主推荐：LangChain 生态，优先使用 LangGraph 作为有状态 Agent 编排层。**

如果必须在题目给出的四个选项中选择一个，则归类为：

**选择 LangChain。**

配套建议：

- 核心 LLM Agent 编排：LangChain / LangGraph，嵌入现有 Python FastAPI 后端。
- 定时调度：短期继续使用现有 APScheduler + Redis lock + `agent_task_runs` 审计。
- 外围流程自动化：n8n 可作为后续通知、运营审批、飞书/邮件/Slack/企业微信集成的辅助工具，不作为核心抽取与风控编排引擎。
- AutoGen、CrewAI：暂不作为当前阶段主编排框架，后续只有在出现真正多 Agent 协作、专家辩论、复杂任务分工和长链路自治规划时再评估。

推荐路线不是“立即重构所有 Agent”，而是：

```text
保留当前 Agent Runtime
  -> 抽象统一 Agent Step / Agent State / Tool 调用接口
  -> 在 LEAD_EXTRACTION 链路试点 LangGraph
  -> 再扩展到 SOURCE_DISCOVERY / RAG / 分级 / 审计
```

## 2. 当前项目事实

当前系统不是泛用聊天机器人，而是一个强约束的海外车辆采购 AI 获客系统。第二阶段核心流程为：

```text
渠道计划/关键词/行业链路
  -> Source Discovery Agent 定时发现来源
  -> lead_source_candidates 来源候选池
  -> Low/Medium 自动通过抽取准入，High 进入移动端人工审核
  -> approved 来源进入 LEAD_EXTRACTION 队列
  -> 公开页面读取和证据快照
  -> LLM 抽取客户信息
  -> LLM 分级 + 规则硬校验
  -> staging_leads
  -> 人工复核晋级 customers / lead_sources / contact_methods
  -> 客服或出口销售交付
```

当前后端已经具备以下基础：

- FastAPI 后端：`apps/api`
- PostgreSQL 数据底座
- Redis lock
- APScheduler 定时任务
- `LLMClient` 统一调用层
- `AgentSchedulerService` 定时调度
- `AgentThreadRunner` 后台线程运行
- `agent_task_runs` 任务运行审计
- `llm_prompt_templates` prompt/schema 管理
- `lead_source_candidates` 来源候选池
- `staging_leads` 线索暂存池
- 移动端来源审核、线索池、触达助手等页面已逐步接入 API

因此，选型必须优先满足：

- 可嵌入现有 Python 服务，而不是另起一套黑盒系统。
- 可保留 PostgreSQL 审计事实表。
- 可做 schema 强校验。
- 可做风险准入硬闸门。
- 可测试、可回放、可定位失败原因。
- 不降低合规边界。

## 3. 强制业务与合规边界

服务编排框架不得改变以下规则：

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- Forbidden 来源不得进入自动抽取。
- High 风险来源必须人工审核。
- Invalid / Watch 不进入触达队列。
- 勿扰客户不得再次进入触达队列。
- C 级线索报价、合同或实质交易前必须合规复核。
- LLM 缺失字段必须输出 `Unknown`、`null` 或空数组，不得编造。
- 所有 AI 输出必须保留来源证据和审计记录。

这些规则要求核心编排层必须是“可控任务流”，而不是“自由自治 Agent”。

## 4. 评估维度

本次选型使用以下维度：

| 维度 | 说明 |
|---|---|
| 与现有技术栈匹配度 | 是否适配 Python + FastAPI + PostgreSQL + Redis |
| 可审计性 | 是否容易把输入、输出、步骤、错误、token、证据写入现有审计表 |
| 合规可控性 | 是否能把风险规则作为硬闸门，而不是提示词软约束 |
| Schema 与结构化输出 | 是否适合 JSON schema、Pydantic、枚举、规则校验 |
| 工作流表达能力 | 是否支持分支、重试、人工复核、状态持久化 |
| RAG 与工具调用 | 是否适合接入知识库、搜索、页面读取、数据库工具 |
| 测试友好度 | 是否容易写单元测试、集成测试和回放测试 |
| 运维复杂度 | 是否增加额外服务、部署、权限和监控成本 |
| 团队可维护性 | 是否适合长期迭代，而不是依赖低代码界面里的隐性逻辑 |

## 5. 四个方案对比

| 方案 | 定位 | 优点 | 主要问题 | 当前适配结论 |
|---|---|---|---|---|
| n8n | 低代码工作流自动化和集成平台，也提供 AI Agent、LLM Chain、信息抽取、向量库等节点 | 可视化编排强；外部系统集成丰富；适合通知、审批、表单、邮件、Webhook；运营人员理解成本较低 | 核心风控逻辑容易分散在节点配置里；代码级测试和版本治理弱于后端代码；与现有 `agent_task_runs`、schema、风险闸门耦合成本较高 | 不作为核心 Agent 编排；适合作为外围自动化辅助 |
| LangChain / LangGraph | Python LLM 应用与 Agent 编排生态；LangGraph 面向有状态、可持久化、可人工介入的工作流 | 与 Python/FastAPI 匹配；支持多模型抽象、工具调用、结构化输出、RAG；LangGraph 适合状态机、分支、重试、human-in-the-loop；可嵌入现有审计体系 | 需要工程化封装，不能直接把业务规则交给 Agent 自由运行；版本升级要谨慎 | 主推荐 |
| AutoGen | 面向单 Agent / 多 Agent 会话和事件驱动多 Agent 系统 | 适合多 Agent 协作、研究型任务、专家角色讨论、动态对话式工作流 | 当前项目不是多 Agent 对话协作主导；引入后容易增加不可预测性；审计和硬规则仍需大量自研 | 当前不推荐作为主框架 |
| CrewAI | 角色型 Agent、Crew、Flow、任务流程编排平台 | 角色/任务表达清晰；多 Agent 协作上手快；提供 Flow、guardrails、memory、knowledge、observability 等能力 | 当前核心诉求是受控抽取、风控准入、证据审计，不是角色协作；框架抽象可能与现有数据模型重复 | 暂不主选，可后续专项评估 |

## 6. 与当前现状相比，LangChain / LangGraph 能带来的增量收益

当前系统已经不是从零开始：`LLMClient`、`AgentSchedulerService`、`AgentThreadRunner`、`agent_task_runs`、`llm_prompt_templates`、`lead_source_candidates`、`staging_leads` 和真实 PostgreSQL/Redis 联调约束都已经存在。因此，引入 LangChain / LangGraph 的价值不在于“替代现有后端”，而在于补齐现有 Agent Runtime 的编排表达、状态管理和可观测能力。

### 6.1 当前现状的主要短板

| 当前能力 | 已有基础 | 主要短板 |
|---|---|---|
| 定时触发 | APScheduler + Redis lock 已具备 | 只回答“什么时候跑”，不表达“每一步如何分支、暂停、恢复” |
| 后台执行 | `AgentThreadRunner` 已具备 | 更偏线程执行器，不是可视化/可持久化的 Agent 状态流 |
| LLM 调用 | `LLMClient` 已统一 provider/model | 调用前后步骤、工具调用和重试路径仍散落在业务 service 中 |
| 审计 | `agent_task_runs` 已具备 | 单个 run 内部 step 级状态、边界输入输出、恢复点还不够结构化 |
| 风控 | 已有风险准入和人工审核规则 | 如果链路继续膨胀，硬规则容易分散在多个 service 分支里 |
| 测试 | 已有 mock LLM、schema、真实库联调测试 | 复杂分支的回放测试和 step 级失败定位成本会升高 |

### 6.2 LangChain / LangGraph 的直接收益

| 收益 | 对当前项目的具体意义 | 不应误用的地方 |
|---|---|---|
| 显式状态机 | 把 `LEAD_EXTRACTION` 从“长 service 方法”拆成 `load_source -> read_page -> extract -> validate -> grade -> write_staging` 的可读图 | 不把业务裁决权交给 LLM 自由决定 |
| 分支和中断 | High 来源、证据不足、schema 错误、C 级合规复核可变成明确边 | 不绕过现有移动端人工审核队列 |
| 持久化与恢复 | 长任务或失败任务可从已完成 step 恢复，减少重复抓取、重复 LLM 调用和重复入库风险 | 不替代 PostgreSQL 事实表，仍以现有库为事实来源 |
| Human-in-the-loop | 与 High 来源审核、C 级复核、失败案例修正天然匹配 | 不允许人审状态只存在框架内存里 |
| 工具调用抽象 | 把 `PublicPageReader`、`RiskPolicyValidator`、`KnowledgeRetriever`、Repository 包成受控工具 | 工具必须最小权限，不给 Agent 任意数据库写权限 |
| 模型与 RAG 生态 | 保留 DeepSeek/OpenAI-compatible 接入，同时便于扩展 RAG、retriever、structured output | 不因生态丰富而引入不必要依赖 |
| 可观测性 | LangSmith 或自建 trace 可补充 step 级轨迹；现有 `agent_task_runs` 继续记录业务审计 | 不把外部 trace 平台作为唯一审计来源 |

### 6.3 与继续纯自研编排相比

继续纯自研的优点是依赖少、边界完全可控，短期也能满足第二阶段小范围运行。但随着链路增加，会出现三个成本：

- 状态流成本：每条链路都要重复实现 step、branch、retry、resume、manual review pending。
- 观测成本：每个 service 都要自己定义 step 日志、耗时、错误分类和回放输入。
- 演进成本：SOURCE_DISCOVERY、LEAD_EXTRACTION、LEAD_GRADING、RAG、失败案例库很容易形成多套相似但不一致的编排代码。

LangGraph 的价值是把这些共性沉到图编排层；业务规则仍保留在 Python service、Pydantic/schema 和 PostgreSQL 事务里。

### 6.4 与 n8n 等外部工作流相比

LangGraph 更适合当前核心链路，因为它作为 Python 库嵌入后端，不需要把核心风控迁移到外部低代码平台。n8n 更适合外围运营自动化，例如通知、审批提醒、日报、CRM/飞书/邮件集成。

核心判断：

- 只要涉及线索是否入库、是否可触达、是否绕过人工审核，就必须留在后端代码和数据库审计中。
- 只要是通知、提醒、报表、轻量跨系统流转，可以交给 n8n。

## 7. 为什么推荐 LangChain / LangGraph

### 7.1 与现有 Python 后端最匹配

当前后端已经有 `LLMClient`、`AgentSchedulerService`、`AgentThreadRunner`、PostgreSQL 审计表和大量 Python 测试。LangChain/LangGraph 可以作为库嵌入现有服务，不需要把核心任务迁移到独立工作流平台。

这意味着：

- API 边界不用重做。
- 数据库模型不用重做。
- prompt/schema 入库策略可以保留。
- `agent_task_runs` 可以继续作为审计事实表。
- 当前测试体系可以继续演进。

### 7.2 更适合“受控 Agent”，不是“自治 Agent”

项目中的 Agent 应该按固定流程运行：

```text
读取候选来源
  -> 风险准入校验
  -> 抓取公开页面文本
  -> 证据快照
  -> LLM 抽取
  -> schema 校验
  -> LLM 分级
  -> 规则硬校验
  -> 写入 staging
  -> 记录审计
```

LangGraph 更适合把这个流程建成显式状态机，每个节点只做一个动作，每个边界都能加硬校验。这样比让 Agent 自由决定下一步更可靠。

### 7.3 支持人审和状态持久化

第二阶段有明确的人工复核需求：

- High 来源进入移动端人工审核。
- C 级线索报价/合同前合规复核。
- 来源证据不足时进入待补全。
- LLM schema 错误进入失败案例库。

LangChain 官方文档说明其 Agent 基于 LangGraph，可利用 LangGraph 的 durable execution、human-in-the-loop、persistence 等能力。这个方向与当前“来源审核 + staging 复核 + 任务审计”的结构匹配。

### 7.4 更利于 RAG 和知识库演进

当前系统后续需要知识库：

- 渠道风险政策库
- 国家/地区规则库
- 车型与报价知识
- FAQ 与触达话术
- 失败案例库
- Prompt/schema 版本库

LangChain 生态对 RAG、Retriever、工具调用和多模型适配更成熟，便于把 `pgvector` 知识库、PostgreSQL 查询工具、公开页面读取工具统一接入。

### 7.5 保留代码级测试和合规闸门

当前项目必须避免以下问题：

- LLM 编造来源。
- LLM 输出没有证据。
- High/Forbidden 误入自动抽取。
- Watch/Invalid 误入触达。
- 勿扰客户被再次触达。

这些不能只依赖 prompt。LangChain/LangGraph 作为代码内编排层时，可以把规则做成普通 Python 函数、Pydantic 校验器、数据库事务和测试用例，失败即阻断。

## 8. 为什么 n8n 不适合作为核心

n8n 的优势是流程可视化和外部系统集成。官方文档展示了丰富的 workflow、nodes、executions、schedule trigger、webhook、AI Agent、Information Extractor、PGVector、Postgres Chat Memory 等能力。

但当前项目的核心不是“把多个 SaaS 串起来”，而是：

- 对公开来源进行合规准入。
- 对 LLM 输出做强 schema 和证据校验。
- 对客户线索进入正式表前做风控。
- 形成可回放、可追责、可测试的审计链。

如果把核心风控逻辑放进 n8n，主要风险是：

- 业务规则分散在节点配置、表达式和低代码流程中。
- 单元测试、代码评审、变更审计不如后端代码直接。
- 与已有 `agent_task_runs`、`lead_source_candidates`、`staging_leads` 的事务边界不自然。
- 运维上多一套服务、凭证、权限和执行日志治理。

因此 n8n 的合理位置是外围：

- Agent 失败告警通知。
- 定期日报/周报。
- 人工审批消息推送。
- CRM、邮件、飞书、企业微信、Slack 等系统集成。
- 运营侧轻量流程自动化。

不建议让 n8n 直接决定线索是否入库、是否可触达、是否绕过风险审查。

## 9. 为什么 AutoGen 不作为当前主选

AutoGen 官方定位是构建 AI agents and applications，其中 AgentChat 用于 conversational single and multi-agent applications，Core 是事件驱动的多 Agent AI 系统框架，适合多 Agent 协作研究、动态业务流程和分布式 Agent。

这些能力有价值，但当前项目的主要问题不是“多个 Agent 如何对话协作”，而是：

- 来源是否允许自动抽取。
- LLM 输出是否有证据。
- JSON schema 是否合规。
- 线索是否可晋级。
- 任务失败后如何审计和重试。

AutoGen 的多 Agent 对话能力会增加动态性和不确定性。除非后续出现以下需求，否则不建议作为主线：

- 多专家 Agent 对同一线索进行辩论式评估。
- 研究员 Agent、风控 Agent、销售 Agent 需要长期协作。
- 复杂市场研究任务需要动态计划和反思。
- 跨语言、跨进程分布式 Agent 网络。

当前阶段这些都不是最优先问题。

## 10. 为什么 CrewAI 不作为当前主选

CrewAI 的角色、crew、flow 表达非常适合多角色协作型 Agent，例如“研究员、分析师、销售策略师、审核员”共同完成任务。官方文档也强调 crews、flows、guardrails、memory、knowledge、observability 等能力。

但当前系统已有清晰的数据表、API 和审核流程，真正需要的是把每个动作做成可审计、可重试、可测试的后端任务。CrewAI 的角色化抽象可能导致：

- 任务边界看起来清晰，但实际风控仍需另写硬规则。
- 与现有 `agent_task_runs`、`llm_prompt_templates`、移动端审核队列有重复抽象。
- 容易把“角色协作”引入到本来可以确定执行的抽取流程中。

因此 CrewAI 可以作为后续探索项，例如：

- 市场研究报告生成。
- 渠道策略头脑风暴。
- 销售话术多角色评审。
- 复杂国家进入策略分析。

但不建议作为 LEAD_EXTRACTION 的主编排框架。

## 11. 更适合现状的其他成熟方案评估

除原始四个候选外，结合当前 Python 后端、强审计、强 schema、真实库联调和小范围运行阶段，还应关注以下方案。

| 方案 | 当前定位 | 适配度 | 建议 |
|---|---|---:|---|
| OpenAI Agents SDK | 轻量 Python Agent runtime，包含 agents、handoffs、guardrails、sessions、human-in-the-loop、tracing、MCP 工具调用 | 中高 | 如果后续主要使用 OpenAI 模型和 Responses API，可作为 LangGraph 之外的轻量候选；当前 DeepSeek/OpenAI-compatible 多 provider 策略下不宜作为主编排 |
| Pydantic AI | Python-first、强类型 Agent 框架，强调 Pydantic 输出、依赖注入、工具、测试、评估和 Pydantic Graph | 高 | 很适合当前“schema 强校验 + Python service + 可测试”的风格；如果团队想要比 LangGraph 更轻、更类型优先的方案，建议做 PoC 对比 |
| Temporal | 通用 durable workflow 引擎，强在长任务、重试、超时、补偿、任务队列和可靠执行 | 中 | 如果后续 Agent 任务变成跨小时/跨天、强 SLA、多 worker、必须精准恢复的生产系统，Temporal 适合做底层任务编排；当前阶段引入偏重 |
| Microsoft Agent Framework | Microsoft 新一代 Agent 框架方向，承接 Semantic Kernel/AutoGen 生态能力 | 中 | 适合 Microsoft/Azure 生态或企业内多 Agent 标准化场景；当前项目不应优先切入 |
| Prefect / Dagster | 数据流和批任务编排 | 中低 | 可用于离线数据同步、报表、批处理；不适合作为核心 LLM 风控 Agent 状态机 |

### 11.1 我更推荐的现实路线

如果只选一个主路线：

**LangGraph 作为核心 Agent 状态编排，保留现有 FastAPI + PostgreSQL + Redis + APScheduler 主控。**

如果允许并行做一个轻量 PoC：

**用 Pydantic AI 对 `LEAD_EXTRACTION` 做同等 PoC，对比 LangGraph 的代码量、类型安全、测试复杂度和恢复能力。**

原因：

- 当前项目的核心风险不是模型能力，而是结构化输出、证据、审计、人审和硬规则。
- Pydantic AI 与现有 Pydantic/schema 风格贴近，学习成本可能低于 LangGraph。
- LangGraph 在复杂状态机、暂停恢复、人审中断、长期运行方面更强。
- 两者都能嵌入 Python 后端，不必迁移核心风控到外部平台。

最终取舍建议：

| 场景 | 更合适方案 |
|---|---|
| 第二阶段小范围运行，先把链路拆清楚 | 继续现有 Runtime + Step 抽象 |
| LEAD_EXTRACTION 出现多分支、暂停、人审恢复 | LangGraph |
| 重点是强类型输出、测试、轻量 Agent 调用 | Pydantic AI |
| 重点是 OpenAI 模型生态、handoff、内置 tracing | OpenAI Agents SDK |
| 任务跨多 worker、跨天运行、强可靠重试 | Temporal + LangGraph/Pydantic AI |
| 运营通知、审批提醒、外部系统连接 | n8n |

### 11.2 不建议的路线

- 不建议把 n8n 作为核心线索准入和写库决策引擎。
- 不建议为“看起来更 Agent”而引入 AutoGen/CrewAI 多角色对话，当前主链路更需要确定性。
- 不建议同时引入 LangGraph、CrewAI、AutoGen、OpenAI Agents SDK 多套 Agent 框架，容易造成依赖、日志、审计和测试碎片化。
- 不建议把 `agent_task_runs` 废弃为框架自带 trace；框架 trace 可以补充，但业务审计必须仍在 PostgreSQL。

## 12. 推荐目标架构

```text
FastAPI API 层
  |
  |-- Agent 控制接口
  |-- 移动端来源审核/线索审核接口
  |
Agent Runtime 层
  |
  |-- APScheduler 定时触发
  |-- Redis lock 防重入
  |-- AgentThreadRunner 后台执行
  |-- agent_task_runs 审计
  |
LangGraph 编排层
  |
  |-- SOURCE_DISCOVERY_GRAPH
  |-- LEAD_EXTRACTION_GRAPH
  |-- LEAD_GRADING_GRAPH
  |-- RAG_CONTEXT_GRAPH
  |
工具层
  |
  |-- LLMClient
  |-- PublicPageReader
  |-- RiskPolicyValidator
  |-- SchemaValidator
  |-- EvidenceRecorder
  |-- PostgresRepository
  |-- KnowledgeRetriever
  |
数据层
  |
  |-- lead_source_candidates
  |-- staging_leads
  |-- customers
  |-- lead_sources
  |-- contact_methods
  |-- llm_prompt_templates
  |-- agent_task_runs
  |-- audit_risk_logs
```

关键原则：

- APScheduler 只负责“什么时候触发”。
- LangGraph 只负责“任务步骤怎么走”。
- Python 规则函数负责“能不能继续走”。
- PostgreSQL 负责“事实、状态、证据、审计”。
- LLM 只负责“语言理解和结构化建议”，不得拥有最终风控裁决权。

## 13. 分阶段落地计划

### 阶段 0：不引入框架前的整理

目标：先把现有 Agent Runtime 抽象稳定。

任务：

- 统一 Agent 输入输出协议。
- 明确 Agent 状态枚举。
- 明确失败原因枚举。
- 将 `SOURCE_DISCOVERY`、`LEAD_EXTRACTION`、`LEAD_GRADING` 的 step 拆开。
- 所有 step 写入 `agent_task_runs.output_summary_json` 或关联审计表。

验收：

- 不引入 LangChain/LangGraph 也能看清每一步状态。
- 每个失败点都有可读错误和审计记录。

### 阶段 1：在 LEAD_EXTRACTION 试点 LangGraph

目标：只改一个链路，避免大范围重构。

建议图节点：

```text
load_approved_sources
  -> read_public_page
  -> record_page_snapshot
  -> extract_lead_json
  -> validate_extraction_schema
  -> grade_lead
  -> validate_grade_and_risk
  -> write_staging_lead
  -> mark_source_extracted
```

每个节点必须：

- 输入输出结构化。
- 失败可定位。
- 失败可写入 `failed_cases` 或 `agent_task_runs`。
- 不允许跳过风险校验。

验收：

- 可抽取真实来源。
- schema 错误不会入库。
- 有联系方式的合格线索可以进入 `staging_leads`。
- High/Forbidden 仍不会自动进入抽取。

### 阶段 2：扩展到 SOURCE_DISCOVERY

目标：让来源发现也进入显式图流程。

建议图节点：

```text
build_discovery_queries
  -> call_llm_source_discovery
  -> normalize_candidate_url
  -> classify_platform_and_risk
  -> validate_evidence
  -> dedupe_candidate
  -> auto_approve_low_medium
  -> queue_for_extraction_or_review
```

验收：

- Low/Medium 自动通过。
- High 自动进入人工审核。
- Forbidden 自动拒绝。
- 来源重复不会污染候选池。

### 阶段 3：接入 RAG 和知识库

目标：把渠道政策、车型、FAQ、失败案例用于 Agent 推理。

任务：

- 将风险政策知识库作为检索工具。
- 将失败案例作为 prompt 反例输入。
- 将 FAQ 和话术库作为触达草稿辅助，不自动发送。
- 将检索命中的知识条目写入审计。

验收：

- 每次使用知识库都有引用记录。
- RAG 内容只影响建议，不绕过硬规则。

### 阶段 4：n8n 外围自动化

目标：只做外围运营效率，不接管核心风控。

可接入场景：

- Agent 失败通知。
- 每日来源发现统计。
- 每日线索抽取统计。
- High 风险来源待审提醒。
- C 级合规复核提醒。
- 销售承接超时提醒。

禁止场景：

- n8n 直接写入正式客户。
- n8n 自动触达客户。
- n8n 绕过 High 风险审核。
- n8n 修改勿扰状态。

### 阶段 5：Pydantic AI 轻量对照 PoC

目标：避免过早锁死在单一框架，用同一条 `LEAD_EXTRACTION` 子链路验证类型安全和测试成本。

PoC 范围：

- 只覆盖 `extract_lead_json -> validate_extraction_schema -> grade_lead`。
- 复用现有 `LLMClient` 或 OpenAI-compatible provider 配置。
- 输出仍走现有 schema、Repository 和 `agent_task_runs`。
- 不新增生产写库路径，只在测试或 feature flag 下运行。

验收：

- 与 LangGraph PoC 使用同一批输入和 mock LLM fixture。
- 对比代码行数、测试复杂度、失败定位、恢复能力。
- 如果没有明显收益，不进入主线。

## 14. 技术规范

### 14.1 编排边界

- 编排框架不得直接操作数据库事务，必须通过 Repository/Service 层。
- 编排节点不得吞异常，必须返回标准失败原因。
- 每个 Agent run 必须有 `agent_task_runs` 记录。
- 每次 LLM 调用必须记录 provider、model、prompt version、token usage、latency、raw output 摘要。

### 14.2 Schema 规范

- 所有 LLM 输出先进入 Pydantic/schema 校验。
- 缺失值只允许 `Unknown`、`null`、`[]`。
- 枚举扩展必须有审计，不允许静默扩张影响风控。
- 有联系方式不等于可触达，仍需勿扰、风险、等级规则判断。

### 14.3 风控规范

- 风险准入必须是代码硬规则。
- LLM 不得直接决定 High 来源自动通过。
- Forbidden 来源不得进入自动任务。
- 触达相关动作必须保留人工确认。

### 14.4 测试规范

必须覆盖：

- Low/Medium 来源自动通过并进入抽取。
- High 来源进入人工审核。
- Forbidden 来源被拒绝。
- LLM 输出缺证据时阻断。
- LLM schema 错误时阻断。
- Watch/Invalid 不进入触达队列。
- 勿扰客户不进入触达队列。
- Agent 重试不重复写入线索。

## 15. 最终选型建议

最终建议如下：

1. 当前阶段不引入 n8n、AutoGen、CrewAI 作为核心编排。
2. 选择 LangChain 生态，具体执行优先采用 LangGraph。
3. 短期保留现有 APScheduler、Redis lock、PostgreSQL 审计、`AgentThreadRunner` 和 `LLMClient`。
4. 先在 `LEAD_EXTRACTION` 单链路试点 LangGraph，验证效果后再扩展。
5. 可并行做 Pydantic AI 小 PoC，但只作为轻量对照，不影响主线交付。
6. n8n 仅作为外围通知、审批、报表和跨系统连接工具。
7. OpenAI Agents SDK、AutoGen、CrewAI、Microsoft Agent Framework 放入技术观察池，等出现明确多 Agent 协作、OpenAI 深度绑定或企业 Agent 平台化需求后再评估。
8. Temporal 暂不引入；只有当 Agent 任务进入跨天长流程、多 worker、强 SLA 和复杂补偿阶段，才作为底层 durable workflow 评估。

一句话决策：

**当前项目最需要的是可审计、可测试、可硬控的 LLM 任务编排；LangChain/LangGraph 与现有 Python 后端和风控边界最匹配。**

## 18. 第二轮独立复核：技术落地与风险

复核维度：

- 是否避免引入额外黑盒导致审计弱化。
- 是否避免把合规规则交给 LLM 自由判断。
- 是否与现有 FastAPI、PostgreSQL、Redis、APScheduler、`AgentThreadRunner`、`LLMClient`、测试体系兼容。
- 是否有渐进式落地路径，避免一次性重构风险。
- 是否避免多框架并行导致审计和测试碎片化。

结论：

- 推荐方案保持现有后端为主控，不把核心风控迁移到低代码平台。
- LLM 只作为语言理解和结构化建议能力，最终准入由代码规则和数据库状态决定。
- LangChain/LangGraph 作为库嵌入，能复用现有服务、审计表和测试。
- Pydantic AI 只建议做小范围对照 PoC，Temporal 只作为未来长任务底层工作流候选，避免当前阶段过度设计。
- 落地计划从 LEAD_EXTRACTION 单链路试点开始，风险可控。

发现项：

- 如果直接引入 LangGraph 并替换所有 Agent，风险过高。
- 如果 n8n 直接写核心表，可能绕过当前审计和测试边界。
- 如果同时引入多套 Agent 框架，会造成 tracing、状态、错误枚举和测试 fixture 分裂。

修正结果：

- 已明确“先试点单链路，不立即重构所有 Agent”。
- 已明确 n8n 禁止直接写正式客户、自动触达或绕过 High 风险审核。
- 已明确不建议多框架并行进入主线；Pydantic AI 只作为受控 PoC，OpenAI Agents SDK/AutoGen/CrewAI/Microsoft Agent Framework 进入观察池。
