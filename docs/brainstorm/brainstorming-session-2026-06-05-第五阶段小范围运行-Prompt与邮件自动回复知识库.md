# 第五阶段小范围运行头脑风暴：Prompt 与邮件自动回复知识库

## Session Overview

**主题：** 第五阶段小范围运行，建设可治理的 Prompt、邮件触达自动回复内容、Q&A 后台知识库和 pgvector 语义存储能力。

**目标：**

1. 将当前以文件存储的 `prompts/` 全部入库，进入 `llm_prompt_templates` 或其扩展治理体系。
2. 新增邮件触达自动回复的邮件内容，可在 `apps/admin` 后台查看和编辑。
3. 新增 Q&A 后台管理知识库，支持运营维护常见问题、标准回复、合规话术和业务解释。
4. 将邮件自动回复内容和 Q&A 知识库语义存储到 PostgreSQL + pgvector，用于后续 RAG 召回。
5. 明确权限、版本、审核、发布、回滚、审计、合规和端到端联调流程。

## 当前系统初步上下文

- 已有 `llm_prompt_templates` 表和 Prompt 查询 API，第二阶段主要是只读治理，后续可扩展到编辑、版本、发布和回滚。
- 已有知识库 pgvector 基础表和 `knowledge_search` / `rag_prompt_context` 服务，适合承接邮件回复和 Q&A 的语义召回。
- 已有 `apps/admin` 管理后台，可新增 Prompt 治理、邮件自动回复模板、Q&A 知识库页面。
- 已有邮件/触达相关记录和客户跟进模型，但“自动回复邮件内容”需要明确是模板、草稿建议、自动发送规则，还是人工确认后发送。
- 第五阶段建议保持小范围运行原则：先做到后台可治理、语义可召回、人工确认可用，不直接放开无人值守自动发送。

## 第一轮讨论记录

### 问题 1：第五阶段的自动回复边界

**决定：选择 C。**

邮件触达自动回复采用“低风险自动发送 + 其他场景人工确认”的边界：

- 对白名单客户、固定 FAQ、首次邮件触达、低风险场景允许自动发送。
- 其他场景由 LLM 根据客户来信、客户资料、Q&A、邮件回复知识库生成建议回复。
- 运营/销售需要在后台或工作台人工确认、编辑后发送，或标记已发送。

**专业建议沉淀：**

- 第五阶段可以验证自动回复闭环，但不能直接放开所有客户邮件的无人值守自动发送。
- 自动发送必须受场景、客户白名单、风险等级、话术模板状态、合规关键词和 DNC/勿扰机制共同约束。
- 自动发送和人工确认都必须写入审计日志，至少记录客户、来源邮件、命中知识、Prompt 版本、模型、回复内容、触发规则、发送状态和操作人。

### 问题 2：Prompt 入库后的治理边界

**决定：选择 B。**

Prompt 入库后采用“后台可编辑草稿，发布生成新版本”的治理方式：

- 后台可以查看 Prompt 原文、任务类型、模型、输出 JSON Schema、状态、默认版本和历史版本。
- 后台编辑只生成草稿版本，不直接覆盖线上生效版本。
- 草稿发布前必须通过基础校验：必填变量、JSON Schema 可解析、任务类型合法、风险规则提示存在、输出契约摘要存在。
- 发布时生成新的版本号，并可标记为该任务类型的默认版本。
- 历史版本保留，可回滚到上一个已发布版本。

**专业建议沉淀：**

- 第五阶段先支持 Prompt 草稿、发布、回滚、默认版本切换，不建议直接编辑线上 active 记录。
- Schema 可先支持查看和校验，不建议普通运营自由编辑；如需编辑 Schema，应限制为管理员或技术角色，并强制通过测试样例。
- 每次发布必须写入审计日志，记录编辑人、发布人、变更摘要、发布原因、影响任务类型和回滚目标。

### 问题 3：Q&A 与邮件自动回复内容的知识库建模方式

**决定：选择 B。**

Q&A 知识库和邮件自动回复内容采用统一知识库底座，按内容类型、业务场景和风险属性区分：

- 底层复用 `knowledge_collections`、`knowledge_items`、`knowledge_embeddings` 或其第五阶段扩展表。
- 通过字段区分用途，例如 `content_type`、`business_scene`、`language`、`status`、`risk_level`、`auto_reply_allowed`。
- 典型内容类型包括：
  - `qa_entry`：Q&A 问答条目。
  - `email_reply_template`：邮件回复模板。
  - `compliance_phrase`：合规话术。
  - `vehicle_product_note`：车型/车源说明。
  - `process_sop`：业务流程说明。
- pgvector embedding 统一生成和存储，但检索时必须按业务场景过滤，避免把不适合发送的知识召回到自动回复。

**专业建议沉淀：**

- 后台 UI 可以按“Q&A 知识库”和“邮件回复内容”分菜单展示，但 API 和数据库底座建议复用，减少重复建设。
- 自动回复 RAG 检索必须带过滤条件：语言、客户市场、场景、状态、是否允许自动回复、风险等级。
- `draft` 内容不得进入自动回复召回；只有 `published/active` 且 `auto_reply_allowed=true` 的内容可以进入自动发送候选。

### 问题 4：自动发送准入规则与硬拦截规则

**决定：选择 C。**

邮件自动回复采用“准入规则 + 硬拦截规则”双层判断：

**自动发送准入条件：**

- 白名单客户。
- 固定 FAQ 命中。
- 首次邮件触达。
- 低风险场景，例如基础业务介绍、合作流程说明、车型资料说明、联系信息确认、下一步沟通预约。

**硬拦截规则：**

以下任一命中时，不允许自动发送，必须转人工确认：

- 投诉、威胁举报、要求停止联系、DNC/勿扰。
- 平台封禁、违规风险、隐私风险、来源合规不清。
- 付款收款、价格承诺、合同条款、发票、税务、法律合规。
- 制裁、出口管制、禁运、贸易路径不可行。
- 客户要求删除数据、拒绝沟通或表达强烈负面情绪。
- 涉及敏感个人信息或无法确认公开来源合法性。
- LLM 置信度低、知识库召回不足、缺少可引用知识证据。
- 客户或线索等级为 Watch/Invalid，对外显示为 D/E 级。

**专业建议沉淀：**

- 自动发送不能只依赖 LLM 判断，必须由规则引擎先做硬拦截。
- 自动发送前应保存“准入原因”和“拦截检查结果”，方便后续复盘和事故追踪。
- 自动回复内容必须引用命中的知识库条目或邮件模板版本；无命中知识时只能生成建议回复，不能自动发送。

### 问题 5：后台管理页面与角色权限

**决定：选择 B。**

后台管理采用“三类页面拆开，权限分层”的方式：

**后台页面：**

- Prompt 模板管理：查看、编辑草稿、版本发布、默认版本切换、回滚、测试样例。
- 邮件回复内容管理：维护首次触达、固定 FAQ 回复、低风险自动回复模板、人工建议回复模板。
- Q&A 知识库管理：维护问答、合规话术、车型/业务说明、流程 SOP，并支持语义召回配置。

**角色权限：**

- `operator`：可创建和编辑 Q&A/邮件回复草稿，不可发布，不可修改 Prompt Schema。
- `sales_manager`：可审核并发布业务内容，如 Q&A、邮件回复模板、固定 FAQ。
- `admin`：可发布 Prompt 模板、切换默认版本、回滚版本。
- `tech_admin`：可编辑 JSON Schema、模型配置、embedding 配置和系统级 Prompt。

**专业建议沉淀：**

- 三类内容生命周期不同，不建议混在一个页面里。
- Prompt 和 Schema 影响 Agent 输出契约，必须比普通知识内容有更高发布权限。
- 所有发布、回滚、默认版本切换和自动发送权限变更都要写入审计日志。

### 问题 6：pgvector embedding 生成与更新策略

**决定：选择 B。**

邮件回复内容和 Q&A 知识库采用“发布后异步生成 embedding”的策略：

- 草稿阶段不进入正式向量召回。
- 内容发布后进入 `pending_embedding` 状态。
- 后台发布接口不阻塞等待 embedding 完成，只返回发布结果和 embedding 待处理状态。
- 异步任务生成 pgvector embedding，成功后更新为 `embedding_ready`。
- 生成失败时更新为 `embedding_failed`，后台可重试。
- 已发布内容被编辑后，新版本重新进入 `pending_embedding`；旧版本保留历史，但不作为默认召回。

**建议状态：**

- `pending_embedding`：等待生成向量。
- `embedding_ready`：可进入 RAG 召回。
- `embedding_failed`：生成失败，需重试或人工处理。
- `stale_embedding`：内容已变化，旧向量不再代表当前内容。

**专业建议沉淀：**

- 自动回复只允许召回 `published/active + embedding_ready + auto_reply_allowed=true` 的内容。
- RAG 检索结果需要记录命中的知识条目 ID、版本、相似度和过滤条件。
- embedding 模型版本必须入库，后续切换模型时可以批量重建。

### 问题 7：Prompt 文件入库迁移方式

**决定：选择 B。**

当前 `prompts/` 文件采用“一次性迁移脚本 + 文件保留为基线”的方式入库：

- 创建迁移/seed 脚本扫描 `prompts/*.md`。
- 将 Prompt 内容解析后写入 `llm_prompt_templates`。
- 文件继续保留，作为开发基线、灾备和差异对照来源。
- 运行时优先读取数据库中该任务类型的 `active + is_default=true` 版本。
- 后台编辑只产生数据库草稿和发布版本，不自动回写文件。

**建议迁移字段：**

- `source_file_path`
- `source_file_hash`
- `migration_batch_id`
- `task_type`
- `version`
- `status`
- `is_default`
- `system_prompt`
- `user_prompt_template`
- `output_schema_json`

**专业建议沉淀：**

- 不建议服务启动时自动同步文件到库，避免覆盖后台编辑结果。
- 迁移脚本应具备幂等能力：同一文件 hash 和同一版本不重复写入。
- 文件 hash 变化但数据库已有发布版本时，应生成新草稿或迁移候选，不直接覆盖 active 版本。

### 问题 8：邮件自动回复与 Q&A 的审核发布流程

**决定：选择 B。**

邮件自动回复内容和 Q&A 知识库采用“草稿 → 审核 → 发布 → embedding → 可召回”的流程：

- `operator` 可创建和编辑草稿。
- 草稿提交后进入 `in_review`。
- `sales_manager` 或具备发布权限的角色审核发布业务内容。
- 发布后进入 `pending_embedding`，由异步任务生成 pgvector embedding。
- embedding 成功后进入 `embedding_ready`，才允许进入 RAG 召回。
- 自动发送还需要额外满足 `auto_reply_allowed=true` 和自动发送准入规则。

**建议状态：**

- `draft`：草稿。
- `in_review`：待审核。
- `published`：已发布。
- `embedding_ready`：向量可召回。
- `active_for_retrieval`：可被指定场景召回。
- `archived`：已下线。
- `blocked`：因合规/风险原因禁用。

**专业建议沉淀：**

- Q&A 虽然不是直接外发模板，但会影响 LLM 回复内容，也必须审核发布后才可召回。
- 邮件自动发送内容必须比普通 Q&A 更严格，需要单独开关 `auto_reply_allowed`。
- 审核、发布、下线、重新生成 embedding 均需写入审计日志。

### 问题 9：自动回复生成时需要注入哪些上下文

**决定：选择 C。**

自动回复生成采用“完整业务上下文 + RAG + 风险检查”的上下文注入方式：

- 客户基础信息。
- 客户最近来信或当前待回复邮件。
- 最近 5 条触达历史。
- 意向车型摘要。
- 客户等级和线索来源风险。
- DNC/勿扰状态。
- 命中的 Q&A 条目。
- 命中的邮件回复模板。
- 命中的合规话术。
- 自动发送准入结果和硬拦截结果。

**专业建议沉淀：**

- 第五阶段暂不把实时车源报价作为自动回复依据，避免产生库存、价格和交付承诺风险。
- 回复生成必须输出结构化结果：建议回复正文、使用知识证据、风险判断、是否允许自动发送、需要人工确认原因。
- RAG 命中内容要记录条目 ID、版本、相似度和过滤条件，便于审计与复盘。

### 问题 10：自动回复结果落库与审计方式

**决定：选择 B。**

自动回复结果采用“AI 建议和最终发送内容分开存”的方式：

- LLM 生成的建议回复作为 AI 回复草稿保存。
- 人工编辑后的最终回复内容单独保存。
- 自动发送场景也要保存“AI 建议版本”和“最终发送版本”，即使两者一致。
- `ai_audit_logs` 记录 LLM 输入、输出、Prompt 版本、模型、token、知识命中、风险判断。
- `outreach_records` 或第五阶段扩展表记录实际发送/标记已发送结果。

**建议记录字段：**

- 客户 ID。
- 来信 ID 或触达线程 ID。
- AI 建议正文。
- 最终正文。
- 命中知识条目 ID、版本、相似度。
- Prompt 模板 ID、Prompt 版本、模型。
- 自动发送判断。
- 拦截原因。
- 人工编辑人。
- 发送状态。
- 发送时间。
- 失败原因。

**专业建议沉淀：**

- 不建议只存最终发送内容，否则无法判断问题来自 AI 生成、人工编辑还是发送通道。
- 自动发送内容也必须可回放：当时使用了哪些客户上下文、哪些知识条目、哪版 Prompt。
- 对外发送前后的内容差异可作为后续 Prompt 优化和知识库质量评估指标。

### 问题 11：邮件通道和发送方式边界

**决定：选择 B。**

第五阶段接入真实邮件发送，但自动发送必须受白名单和硬拦截限制：

- 支持真实邮件发送通道，优先抽象发送接口，便于后续切换 SMTP、SendGrid、Mailgun、企业邮箱等服务。
- 支持发送前预览和人工确认发送。
- 支持白名单客户、固定 FAQ、首次触达、低风险场景自动发送。
- 自动发送前必须经过准入规则和硬拦截规则。
- 支持发送失败、重试、退信、邮件线程关联和发送审计。
- 支持 DNC/勿扰拦截，客户要求停止联系后不得继续自动发送。

**建议邮件通道能力：**

- 发件账号配置。
- 收件人校验。
- 发送预览。
- 人工确认发送。
- 白名单自动发送。
- 失败重试。
- 退信状态。
- 邮件线程关联。
- DNC 拦截。
- 发送审计。

**专业建议沉淀：**

- 第五阶段可以验证真实发送闭环，但不能绕过自动发送准入和硬拦截。
- 发送服务要作为独立适配层，业务侧只依赖统一接口和发送结果。
- 退信、失败和客户回复必须进入客户触达历史，作为后续自动回复和客户管理上下文。

### 问题 12：知识库质量评估与闭环优化

**决定：选择 B。**

Q&A 和邮件回复知识库采用“记录命中与使用效果，形成质量闭环”的方式：

- 每次 AI 回复生成时，记录命中的知识条目、版本、相似度和过滤条件。
- 每次人工确认或编辑时，记录采纳、修改、拒绝和原因。
- 每次自动发送时，记录发送成功、失败、退信、客户回复和风险拦截。
- 后台可按知识条目查看召回次数、采纳率、修改率、自动发送成功率、客户回复率和风险事件。

**建议指标：**

- 知识条目召回次数。
- 回复采纳率。
- 人工编辑幅度。
- 自动发送成功率。
- 退信率。
- 客户回复率。
- 投诉/拦截次数。
- 最后使用时间。
- 建议下线标记。

**专业建议沉淀：**

- 不建议只做内容管理，知识库需要通过真实回复效果不断优化。
- LLM 自动评分只能作为辅助参考，优先采用业务真实反馈和人工编辑差异。
- 经常被大幅修改、经常触发拦截、长期无采纳的知识条目应进入“待优化/建议下线”队列。

### 问题 13：邮件自动回复是否需要新增专门 Agent/任务流

**决定：选择 B，并在 `apps/agents` 中新增 `EMAIL_REPLY` Agent/任务流。**

第五阶段新增独立的邮件自动回复 Agent/任务流：

- 在 `apps/agents` 中新增 `EMAIL_REPLY` Agent。
- 支持手动触发、收件箱事件触发、定时扫描待回复邮件。
- 支持生成建议回复、自动发送准入判断、硬拦截判断、可选自动发送、失败重试和审计。
- 不与 Lead Extraction、Deep Enrichment、Lead Cleanup 混用任务流，避免职责混乱。

**建议任务类型：**

- `EMAIL_REPLY_DRAFT`：生成邮件回复建议。
- `EMAIL_REPLY_KNOWLEDGE_RETRIEVAL`：检索 Q&A、邮件模板、合规话术。
- `EMAIL_REPLY_AUTO_SEND_CHECK`：执行自动发送准入和硬拦截判断。
- `EMAIL_REPLY_SEND`：调用邮件发送通道。

**专业建议沉淀：**

- `EMAIL_REPLY` Agent 应作为第五阶段独立试点，便于单独监控回复质量、自动发送成功率、拦截率、失败重试和 LLM 成本。
- Agent 负责流程编排和 LLM 调用，业务数据的权威写入仍应由 `apps/api` 提供接口完成。
- 自动发送前必须由 `apps/api` 的规则服务返回准入结果，Agent 不应绕过业务规则直接发送。

### 问题 14：`EMAIL_REPLY` Agent 与 `apps/api` 的职责边界

**决定：选择 B。**

`EMAIL_REPLY` Agent 通过 `apps/api` 内部接口读写业务数据，`apps/api` 保持唯一业务权威：

- `apps/agents` 负责流程编排、RAG 调用、LLM 生成、节点追踪、失败重试和结果回传。
- `apps/api` 负责客户上下文读取、知识检索、自动发送准入检查、DNC/勿扰检查、草稿保存、发送提交、触达记录、审计落库和权限判断。
- Agent 不直接读写 PostgreSQL，不绕过 `apps/api` 的业务规则。
- Agent 不直接绕过自动发送规则调用邮件发送通道。

**建议内部接口：**

- 获取待回复邮件和客户上下文。
- 检索 Q&A、邮件回复模板、合规话术。
- 执行自动发送准入和硬拦截检查。
- 保存 AI 回复建议草稿。
- 提交人工确认或自动发送请求。
- 写入 Agent run、节点 trace、AI 审计和发送结果。

**专业建议沉淀：**

- 数据一致性、权限和审计必须收敛在 `apps/api`。
- `apps/agents` 可以使用 LangGraph 编排 `retrieve → draft → validate → send/hold` 流程，但每个业务动作都应通过 API 执行。
- 这种边界便于后续替换 Agent 框架或扩展邮件通道，不影响核心业务数据模型。

### 问题 15：第五阶段最小可交付范围

**决定：选择 B。**

第五阶段采用“一条端到端闭环 MVP”的最小可交付范围：

- Prompt 文件全部入库，并支持后台查看、草稿编辑、发布新版本和回滚。
- Q&A 与邮件回复内容进入统一知识库底座，支持后台草稿、审核、发布。
- 已发布知识异步生成 pgvector embedding，`embedding_ready` 后可进入 RAG 召回。
- 新增 `apps/agents` 中的 `EMAIL_REPLY` Agent/任务流。
- `EMAIL_REPLY` 支持根据客户邮件、客户资料、触达历史、Q&A、邮件模板和合规话术生成建议回复。
- 白名单客户、固定 FAQ、首次邮件触达、低风险场景在通过硬拦截后允许自动发送。
- 其他场景进入运营/销售人工确认、编辑后发送或标记已发送。
- 全链路写入审计和质量指标。

**第五阶段 MVP 验收线：**

后台发布知识 → embedding ready → 客户邮件进入待回复 → `EMAIL_REPLY` 生成建议 → 规则判断自动发送或人工确认 → 发送/标记发送 → 审计和质量指标可查。

**专业建议沉淀：**

- 不把批量营销、复杂收件箱同步、多账号路由、报价自动承诺纳入第五阶段 MVP。
- 第五阶段重点验证“内容治理 + RAG + 自动回复 Agent + 合规拦截 + 审计闭环”。
- 真实邮件发送可以接入，但自动发送范围必须被白名单、低风险场景和硬拦截严格约束。

### 问题 16：第五阶段实施优先级和 Story 拆分顺序

**决定：选择 A。**

第五阶段采用“数据与治理底座优先”的实施顺序：

1. `P5-E1 数据底座`：补齐 Prompt、知识库、邮件回复、自动回复草稿、发送记录、审计和质量指标所需的数据模型。
2. `P5-E2 Prompt 入库治理`：将文件 Prompt 入库，支持后台草稿编辑、发布新版本、默认版本切换和回滚。
3. `P5-E3 Q&A/邮件回复知识库`：统一知识库底座，按内容类型、业务场景、语言、风险等级和自动回复权限管理。
4. `P5-E4 pgvector embedding 异步`：发布后异步生成 embedding，支持状态、重试、模型版本和召回过滤。
5. `P5-E5 后台管理页面`：在 `apps/admin` 中新增 Prompt 模板管理、邮件回复内容管理、Q&A 知识库管理。
6. `P5-E6 自动回复规则与审计`：实现自动发送准入、硬拦截、DNC、风险判断和全链路审计。
7. `P5-E7 EMAIL_REPLY Agent`：在 `apps/agents` 中新增独立 Agent/任务流，通过 `apps/api` 内部接口编排。
8. `P5-E8 邮件发送通道`：接入真实邮件发送适配层，支持预览、人工确认、白名单自动发送、失败重试和退信记录。
9. `P5-E9 质量指标与端到端验收`：形成知识命中、采纳、编辑、发送、回复、风险拦截和 Go/No-Go 复盘指标。

**专业建议沉淀：**

- 先做数据与治理底座，避免把 Prompt、知识库、自动发送规则写死在 Agent 流程里。
- 后台页面必须接真实 API，不做 seed 静态页。
- `EMAIL_REPLY` Agent 必须在规则、知识、Prompt、审计都具备基础能力后接入，减少返工。

## 当前阶段性收敛

截至问题 16，已形成第五阶段核心方向：

- 范围：端到端闭环 MVP。
- 自动回复：低风险自动发送 + 其他场景人工确认。
- Prompt：全部入库，后台草稿编辑，发布生成新版本，可回滚。
- 知识库：Q&A 与邮件回复内容共用统一知识库底座，按类型和场景过滤。
- pgvector：发布后异步生成 embedding，ready 后进入 RAG。
- 后台：Prompt、邮件回复、Q&A 三类页面拆开，角色权限分层。
- Agent：在 `apps/agents` 新增独立 `EMAIL_REPLY` Agent，通过 `apps/api` 内部接口读写业务数据。
- 邮件：接入真实发送通道，但自动发送受白名单、低风险和硬拦截约束。
- 审计：AI 建议与最终发送内容分开存，全链路可回放。

### 问题 17：第五阶段是否纳入收件箱同步能力

**决定：选择 B。**

第五阶段先做“待回复邮件表 + 手动/API 导入”，预留真实邮箱同步适配层：

- 建立 `email_threads` / `email_messages` 或等价数据模型。
- 支持后台手动新增客户来信。
- 支持 API 导入待回复邮件，便于后续对接邮箱服务、Webhook 或人工导入脚本。
- 支持邮件线程、客户关联、待回复状态、已回复状态、失败状态和归档状态。
- `EMAIL_REPLY` Agent 从待回复队列获取邮件，不直接绑定具体邮箱协议。
- IMAP、Gmail、Outlook、企业邮箱同步作为后续增强，不纳入第五阶段核心 MVP。

**专业建议沉淀：**

- 第五阶段需要真实待回复数据结构，否则 Agent 只能做一次性文本生成，无法形成业务闭环。
- 不建议一开始做完整多邮箱同步，OAuth、IMAP 差异、反垃圾策略和安全审核会拉长周期。
- 通过 API 导入和待回复队列表，可以先验证自动回复策略、RAG 召回、人工确认、发送记录和审计闭环。

### 问题 18：自动回复多语言与邮件语气策略

**决定：选择 B。**

自动回复采用“识别客户来信语言，匹配同语言知识和模板”的策略：

- 优先使用客户来信语言回复。
- 知识库、Q&A、邮件回复模板必须标注 `language`。
- RAG 召回默认按客户来信语言过滤。
- 优先支持俄语和英语，后续再扩展其他市场语言。
- 若语言识别不确定，或缺少同语言 `embedding_ready` 内容，则禁止自动发送，转人工确认。
- 可生成中文内部摘要 + 目标语言草稿，方便运营/销售审核。

**邮件语气策略：**

- 默认商务、简洁、低承诺。
- 不主动承诺价格、库存、交付周期、合同条款和付款条件。
- 不使用夸张营销话术。
- 对首次触达保持礼貌、清晰、低打扰。
- 对客户问题优先引用已发布知识库和邮件模板。

**专业建议沉淀：**

- 不建议完全由 LLM 自由决定语言和语气。
- 多语言自动发送必须依赖同语言知识条目和模板，否则容易出现语义偏差或外发质量不可控。
- 自动回复输出中应包含 `detected_language`、`reply_language`、`language_confidence` 和是否允许自动发送的判断。

### 问题 19：第五阶段 Go/No-Go 验收指标

**决定：选择 B。**

第五阶段 Go/No-Go 验收采用“功能 + 质量 + 风险 + 业务结果”的综合指标：

**功能指标：**

- Prompt 入库覆盖率 100%。
- Prompt 后台可查看、草稿编辑、发布新版本、默认版本切换和回滚。
- Q&A 与邮件回复内容可后台草稿、审核、发布。
- 已发布知识可异步生成 pgvector embedding。
- `EMAIL_REPLY` Agent 可完成待回复邮件处理闭环。

**质量指标：**

- 已发布知识 embedding ready 率 ≥ 95%。
- AI 回复建议生成成功率 ≥ 90%。
- 人工采纳率 ≥ 50%。
- 自动回复必须引用已发布知识或邮件模板。
- 经常被大幅编辑或拒绝的知识条目进入优化队列。

**风险指标：**

- 自动发送硬拦截准确执行率 100%。
- DNC/勿扰客户不自动发送 100%。
- Watch/Invalid（D/E 级）客户不自动发送 100%。
- 投诉、封禁、违规事件为 0。
- 缺少同语言知识、语言识别不确定、知识召回不足时不自动发送 100%。

**业务观察指标：**

- 客户回复率。
- 自动发送成功率。
- 退信率。
- 人工处理节省时间。
- 邮件回复到客户跟进转化情况。

**Go / No-Go 建议：**

- **Go：** 功能闭环跑通，风险指标全部达标，质量指标达到最低线，客户回复率和采纳率表现正向。
- **重跑小范围：** 功能跑通但质量指标不足，例如采纳率低、人工修改幅度高、embedding ready 不稳定。
- **暂停自动发送：** 发生投诉、封禁、违规、DNC 误发、D/E 客户误发、硬拦截失效、语言误判外发等风险事件。

## 头脑风暴阶段总结

第五阶段小范围运行的核心方向已经收敛为：

1. **Prompt 治理：** 当前文件 Prompt 全部入库，文件保留为基线；后台支持草稿编辑、发布新版本、默认版本切换和回滚。
2. **邮件自动回复：** 白名单客户、固定 FAQ、首次邮件触达、低风险场景允许自动发送；其他场景由 AI 生成建议，人工确认后发送或标记已发送。
3. **知识库：** Q&A 与邮件回复内容共用统一知识库底座，按内容类型、业务场景、语言、风险等级和自动回复权限区分。
4. **pgvector：** 内容发布后异步生成 embedding，只有 `embedding_ready` 且已发布内容可进入 RAG 召回。
5. **后台管理：** `apps/admin` 拆分 Prompt 模板管理、邮件回复内容管理、Q&A 知识库管理，角色权限分层。
6. **Agent：** 在 `apps/agents` 新增独立 `EMAIL_REPLY` Agent/任务流，通过 `apps/api` 内部接口获取上下文、检索知识、保存草稿、执行规则检查和发送提交。
7. **邮件通道：** 接入真实邮件发送，但自动发送必须经过准入规则和硬拦截；先做待回复邮件表和手动/API 导入，预留真实邮箱同步。
8. **多语言：** 识别客户来信语言，优先匹配同语言知识和模板；语言不确定或缺少同语言知识时转人工确认。
9. **审计与质量：** AI 建议和最终发送内容分开存，全链路记录 Prompt、模型、知识命中、风险判断、人工编辑和发送结果。
10. **实施顺序：** 数据与治理底座优先，再推进后台、规则、Agent、邮件发送和端到端验收。
