# 第三阶段小范围运行 - 客户管理工作台与线索完善闭环头脑风暴记录

创建时间：2026-06-04 CST

## Session Overview

**主题：** 第三阶段小范围运行新增客户管理工作台。线索池中的线索必须经过 AI 深挖补全或人工补录，完善客户信息、联系方式、经营状况、意向车型等关键字段后，才允许流转到客户管理，并晋级到 `customers`、`lead_sources`、`contact_methods` 等 core 表。

**目标：**

1. 定义线索完善和客户晋级的业务边界。
2. 设计 AI 深挖补全与人工补录的协同流程。
3. 明确客户管理工作台应展示的数据：客户补全资料、触达历史、意向车型、跟进记录、合规状态等。
4. 明确无法完善线索的处理方式：降级、Watch/Invalid、放弃、失败案例沉淀。
5. 保持不自动社交私信、不自动加好友、不登录后批量采集、不反爬规避、不自动触达的强制边界。

## 已继承的核心边界

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不反爬规避。
- 不抓取非公开数据。
- High 风险来源必须人工审核后才可只读抽取。
- Forbidden 来源不得进入自动抽取。
- Invalid / Watch 不进入触达队列。
- 勿扰客户不得再次进入触达队列。
- C 级客户在报价、合同或实质交易前必须合规复核。
- LLM 缺失字段必须输出 `Unknown`、`null` 或空数组，不得编造。
- 所有 AI 输出必须保留来源证据和审计记录。
- 客户触达仍为人工确认模式，系统可生成草稿和记录历史，但不得自动发送。

## 当前关键判断

第三阶段的重点不是继续扩大线索数量，而是提升线索到客户的转化质量。系统需要从“发现和抽取线索”升级到“补全、判断、晋级、交付、跟进”的客户运营闭环。

建议把数据流定义为：

```text
staging_leads
  -> 线索完善队列
  -> AI 深挖补全 / 人工补录
  -> 完善度与合规准入校验
  -> 可晋级：customers + lead_sources + contact_methods
  -> 客户工作台
  -> 触达历史 / 意向车型 / 跟进记录 / 销售交付
```

不建议把所有有联系方式的线索直接晋级为客户。客户表应代表“可被团队承接的业务对象”，不是原始线索镜像。

## 维度 1：阶段目标与客户晋级边界

### 初步专业建议

第三阶段建议采用“宽线索池、严客户池”的原则：

- 线索池可以保留不完整、不确定、待补全、Watch/Invalid 的线索。
- 客户池只接收完成最低字段补全、来源证据可追溯、联系方式可用、勿扰状态通过、风险等级允许承接的对象。
- AI 深挖补全只能提供候选事实和证据，不得无证据编造字段。
- 人工补录必须记录补录人、补录时间、字段来源和备注。

### 待确认问题 1

客户晋级到 `customers` 的最低准入标准，建议采用哪一种？

A. 严格准入：客户名称 + 国家/城市 + 至少一个联系方式 + 来源证据 + 客户类型 + 经营信号 + 勿扰校验通过。

B. 平衡准入：客户名称 + 国家/城市 + 至少一个联系方式 + 来源证据 + 勿扰校验通过；客户类型、经营信号、意向车型可后续补全。

C. 宽松准入：只要客户名称 + 至少一个联系方式即可晋级客户，其他字段后补。

**我的建议：选择 B。**

理由：

- A 会提高客户质量，但小范围运行阶段可能导致晋级过慢，客服/销售没有足够样本验证承接流程。
- C 会让客户表快速膨胀，容易把原始线索污染到客户池。
- B 更适合第三阶段：先保证来源、联系方式和勿扰校验，再允许客户进入工作台，同时把客户类型、经营信号、意向车型作为完善度评分和后续任务。


### 已确认决定 1

选择 **B：平衡准入** 作为第三阶段客户晋级最低标准。

客户晋级到 `customers` 的最低条件为：

- 客户名称。
- 国家/城市。
- 至少一个联系方式。
- 来源证据。
- 勿扰校验通过。

客户类型、经营信号、意向车型可以后续继续补全，但必须在客户工作台中显示为“待补全”或“待确认”，不能伪装为已确认事实。

同时保留人工操作：

- 运营/客服/销售可以手动补全客户信息、联系方式、经营状况、意向车型、备注等字段。
- 人工可以执行“手动准入”操作，将满足最低标准的线索晋级到 `customers`、`lead_sources`、`contact_methods`。
- 手动补录和手动准入必须记录操作人、时间、字段来源、备注和审计日志。
- 人工补录不得绕过勿扰、Forbidden、High 风险、C 级合规复核等硬规则。

## 维度 2：线索完善方式 - AI 深挖补全与人工补录

### 初步专业建议

第三阶段建议把线索完善拆成两个并行能力：

1. **AI 深挖补全**
   - 输入：当前线索名称、城市、国家、已有联系方式、来源链接、来源证据、AI 摘要。
   - 动作：LLM Agent 总结关键词，生成可审计的搜索查询，读取 Low/Medium 或已审核可读来源，补充客户官网、邮箱、社媒公开链接、经营描述、库存/车型信号等。
   - 输出：候选补全字段 + 来源证据 + 置信度 + 缺失字段 + 是否建议晋级。
   - 限制：不得登录、不得批量抓取社媒、不得自动私信、不得编造未知字段。

2. **人工补录**
   - 输入：运营/客服/销售通过公开资料、电话沟通、邮件回复、展会名片、业务沟通等获得的信息。
   - 动作：手动填写或修正字段，上传备注或证据链接，选择是否手动准入。
   - 输出：字段变更记录、客户晋级记录、跟进记录。
   - 限制：人工补录仍需遵守风险和合规闸门；关键字段应标记来源类型。

建议所有补全字段都带 `source_type`：

- `ai_public_search`：AI 公开搜索补全。
- `manual_public_info`：人工从公开资料补录。
- `manual_customer_reply`：客户回复产生的信息。
- `manual_business_note`：业务人员线下/沟通备注。
- `unknown`：未知来源，不建议作为晋级关键证据。

### 待确认问题 2

AI 深挖补全的自动化程度建议采用哪一种？

A. 全自动补全并自动晋级：AI 找到足够信息后直接写入客户表。

B. AI 自动补全到 staging 完善区，达到最低准入条件后给出“建议晋级”，但由人工确认晋级。

C. AI 只生成搜索关键词和候选信息，不写任何字段，完全由人工录入。

**我的建议：选择 B。**

理由：

- A 风险较高，AI 可能误合并同名客户或把未经确认的信息写入 core 表。
- C 太慢，不能体现第三阶段引入 AI 深挖的效率价值。
- B 能让 AI 承担信息发现、证据整理和补全建议，同时由人工掌握最终准入，适合小范围运行。


### 已确认决定 2

选择 **B：AI 自动补全到 staging 完善区，达到最低准入条件后给出“建议晋级”，但由人工确认晋级**。

补充确认：

- AI 深挖补全不是后台无差别自动跑。
- 必须由人工在线索详情或线索完善队列中点击 **“深挖线索”** 后启动。
- AI 深挖结果先写入 `staging_leads` 的完善区或后续新增的线索补全结果表，不直接写入 `customers`。
- 当 AI 补全后满足最低准入条件时，系统展示 **“建议晋级客户”**。
- 最终晋级到 `customers`、`lead_sources`、`contact_methods` 必须由人工确认。
- 如果 AI 深挖后仍无法满足最低准入条件，系统应提示缺失字段，并允许人工继续补录、调整等级或放弃线索。

专业判断：

- 人工触发深挖可以控制 LLM 成本，把算力集中在运营认为有价值的线索上。
- AI 写 staging 完善区而非 core 表，可以避免误写正式客户数据。
- 人工确认晋级保留了业务判断和合规责任边界。

## 维度 3：线索完善区的数据结构与字段来源

### 初步专业建议

第三阶段建议不要直接把 AI 深挖字段覆盖原线索字段，而是引入“补全候选层”：

```text
staging_leads 原始抽取字段
  + lead_enrichment_results AI/人工补全候选
  + 字段级来源与置信度
  + 人工采纳/拒绝
  -> promote_to_customer
```

推荐新增或抽象以下结构：

1. `lead_enrichment_results`
   - 记录一次深挖或人工补录批次。
   - 字段：`id`、`staging_lead_id`、`enrichment_type`、`triggered_by`、`status`、`input_snapshot_json`、`output_json`、`evidence_links`、`confidence_score`、`missing_fields`、`recommended_action`、`created_at`。

2. `lead_enrichment_field_candidates`
   - 记录字段级候选值。
   - 字段：`field_name`、`candidate_value`、`source_type`、`source_url`、`evidence_note`、`confidence_score`、`accepted_by`、`accepted_at`、`rejected_reason`。

3. `promotion_review`
   - 记录晋级客户的人工确认。
   - 字段：`reviewer_id`、`decision`、`review_note`、`accepted_fields_json`、`created_customer_id`、`created_at`。

这样做的好处：

- 原始 AI 抽取、AI 深挖补全、人工补录、最终客户字段之间有清晰边界。
- 字段可逐个采纳，避免“一键覆盖全部”。
- 审计上能解释每个客户字段从哪里来。
- 后续如果发现 AI 错误，可以追溯并撤销采纳字段。

### 待确认问题 3

线索完善区的字段采纳方式建议采用哪一种？

A. 批量采纳：AI 深挖结果整体通过后，一次性覆盖 staging 字段并用于晋级。

B. 字段级采纳：AI 或人工补全结果按字段展示，人工可逐个采纳、修改或拒绝，再晋级客户。

C. 只记录最终结果：不保存候选字段细节，只保存晋级后的客户字段。

**我的建议：选择 B。**

理由：

- A 操作快，但容易把错误字段一起带入客户表。
- C 审计弱，后续无法解释客户字段来源。
- B 最适合当前项目的风控要求：既能提升补全效率，又能保留字段级证据、人工判断和可追溯性。


### 已确认决定 3

选择 **B：字段级采纳**。

线索完善区采用字段级候选和人工采纳机制：

- AI 深挖或人工补录的结果按字段展示。
- 人工可逐个采纳、修改或拒绝字段。
- 每个字段保留来源类型、来源链接、证据备注、置信度、采纳人、采纳时间。
- 只有被采纳的字段才能参与客户晋级。
- 未被采纳字段保留在补全结果中，用于审计和失败案例复盘，但不得污染 `customers` core 数据。

推荐后续实现：

- 新增 `lead_enrichment_results` 保存一次补全任务。
- 新增 `lead_enrichment_field_candidates` 保存字段级候选。
- 晋级客户时保存 `accepted_fields_json`，保证 `customers`、`lead_sources`、`contact_methods` 的字段可追溯。

## 维度 4：客户管理工作台信息架构

### 初步专业建议

客户管理工作台不应只是客户列表，而应是客服/销售的承接工作台。建议围绕“谁该处理、为什么处理、下一步做什么”组织信息。

客户列表建议包含：

- 客户名称、国家、城市、客户类型。
- 客户等级：A/B/C，Watch/Invalid 不进入客户工作台。
- 联系方式摘要：邮箱、电话、Telegram/WhatsApp/VK 等公开联系方式。
- 来源完整度：官网、地图、公开目录、人工补录等。
- 完善度评分：基础信息、联系方式、经营信号、意向车型、触达记录。
- 跟进状态：待分配、待首次触达、已触达待回复、有回复待销售、销售跟进、暂停、勿扰。
- 意向车型摘要：品牌、车型、车龄、数量、预算、关注点。
- 下一步动作：分配客服、生成草稿、记录发送、补充车型、发起 C 级合规复核、交付销售。

客户详情建议分区：

1. 客户画像：基础信息、客户类型、经营状况、规模信号。
2. 联系方式：邮箱、电话、社媒、官网表单，并显示勿扰校验。
3. 来源证据：从线索晋级继承的证据链。
4. 意向车型：品牌、车型、车龄、数量、预算、交付关注点。
5. 触达历史：草稿、人工发送、回复、拒绝联系。
6. 跟进记录：客服/销售备注、下一步计划、负责人。
7. 合规状态：C 级复核、勿扰、High 来源边界、报价/合同前检查。

### 待确认问题 4

客户工作台列表的默认排序建议采用哪一种？

A. 按创建时间倒序：最新晋级客户排前面。

B. 按下一步动作优先级：今日待跟进、C 级待合规、已回复待销售、待首次触达、待补全。

C. 按客户等级排序：C 级、B 级、A 级依次展示。

**我的建议：选择 B。**

理由：

- 客户工作台的核心是推进动作，不是静态查看。
- 创建时间排序容易让高价值但即将超时的客户被压下去。
- 等级排序有价值，但不如“下一步动作”直接服务客服/销售日常工作。
- 可以在筛选 Tab 中保留等级筛选，但默认排序应按运营优先级。


### 已确认决定 4

选择 **B：按下一步动作优先级排序**。

客户工作台默认排序优先级：

1. 今日待跟进。
2. C 级待合规复核。
3. 已回复待销售承接。
4. 待首次触达。
5. 待补全客户信息。
6. 销售跟进中。
7. 暂停/低优先级。

等级、创建时间、国家、城市、负责人、客户类型可作为筛选条件，但不作为默认主排序。

专业判断：客户工作台应首先服务“下一步动作”，让客服/销售进入页面后立即知道该处理谁、为什么处理、处理什么。

## 维度 5：客户状态流转与团队交付

### 初步专业建议

第三阶段建议新增客户状态流转，避免客户进入 `customers` 后变成静态档案。

建议状态：

```text
待分配
  -> 待首次触达
  -> 已触达待回复
  -> 有回复待判定
  -> 有意向待销售
  -> 销售跟进中
  -> 成交/长期跟进/暂停/勿扰/无效
```

其中：

- A/B 级客户默认交给客服处理初次触达和意向确认。
- C 级客户可以交给销售，但报价、合同或实质交易前必须完成合规复核。
- 客户拒绝联系时，立即进入勿扰，后续不得进入触达队列。
- 客户无回复不等于无效，应按 SLA 和次数策略进入暂停或长期跟进。
- 线索完善不足但满足最低准入的客户，可进入“待补全”状态，不应阻塞基础承接。

团队交付建议：

- 客服队列：A/B 级、待首次触达、待回复、待补充需求。
- 销售队列：C 级、有明确车型/数量/预算/采购周期的客户。
- 合规队列：C 级报价前、High 来源二次边界、客户要求敏感承诺时。
- 运营队列：待补全、疑似重复、来源证据不足、深挖失败。

### 待确认问题 5

客户晋级后，A/B/C 级的默认交付策略建议采用哪一种？

A. A/B/C 全部先给客服，由客服判断是否转销售。

B. A/B 给客服，C 级进入销售候选，但报价/合同前必须合规复核。

C. A 给客服，B/C 直接给销售，提高转化速度。

**我的建议：选择 B。**

理由：

- A 过于保守，C 级高意向客户可能被客服队列拖慢。
- C 过于激进，B 级还未确认需求时直接给销售会浪费销售精力。
- B 更符合前期已确认的分工：低/中意向由客服处理，高意向由出口销售承接，C 级保留合规复核闸门。


### 已确认决定 5

选择 **B：A/B 给客服，C 级进入销售候选，但报价/合同前必须合规复核**。

客户晋级后的默认交付策略：

- A 级客户：进入客服队列，优先补充需求和完成首次触达。
- B 级客户：进入客服队列，重点确认车型、采购频率、预算和合作意向。
- C 级客户：进入销售候选队列，但报价、合同或实质交易前必须完成合规复核。
- Watch / Invalid 不进入客户工作台和触达队列。
- 勿扰客户不得再次进入触达队列。

团队队列建议：

- 客服队列：A/B 级、待首次触达、待回复、待补充需求。
- 销售队列：C 级、有明确车型/数量/预算/采购周期的客户。
- 合规队列：C 级报价前、High 来源边界、客户要求价格/物流/清关/交付承诺时。
- 运营队列：待补全、疑似重复、来源证据不足、深挖失败。

## 维度 6：深挖线索 Agent 的输入输出与成本控制

### 初步专业建议

第三阶段的“深挖线索”不建议做成一个自由搜索 Agent，而应做成受控任务：

```text
人工点击深挖线索
  -> 读取 staging_lead 当前字段和证据
  -> LLM 总结关键词/搜索查询
  -> 风险策略过滤可搜索渠道
  -> 执行公开搜索/读取
  -> LLM 抽取补全字段
  -> schema + 证据校验
  -> 写入线索完善区
  -> 给出建议晋级/继续补全/降级/放弃
```

输入建议包括：

- 当前客户名称、别名、国家、城市。
- 当前联系方式。
- 当前来源链接和证据摘要。
- 当前 AI 分级、缺失字段、客户类型。
- 人工备注和人工指定关键词。
- 允许搜索的渠道范围和风险边界。

输出必须包括：

- 搜索关键词和查询语句。
- 命中的公开来源链接。
- 字段级候选值。
- 字段级证据备注。
- 置信度。
- 缺失字段。
- 是否满足最低晋级条件。
- 推荐动作：建议晋级、继续补全、降级、放弃、人工复核。
- LLM prompt 版本、模型、token、耗时和任务审计。

成本控制建议：

- 只允许人工触发，不做全量后台自动深挖。
- 每条线索每天限制深挖次数，例如 1-2 次。
- 每次深挖限制搜索查询数，例如 3-5 个。
- 每次深挖限制候选链接读取数，例如 5-10 个。
- 对 Watch/Invalid/勿扰/Forbidden 不允许深挖。
- 深挖失败要进入失败案例库，避免重复消耗。

### 待确认问题 6

深挖线索 Agent 的运行配额建议采用哪一种？

A. 宽松配额：每条线索可多次深挖，直到补全成功。

B. 受控配额：每条线索每天最多 1-2 次，每次最多 3-5 个查询、5-10 个公开链接。

C. 严格配额：每条线索只允许深挖一次，失败后只能人工补录。

**我的建议：选择 B。**

理由：

- A 容易造成 LLM 成本和搜索读取成本失控，也可能对同一低质量线索反复消耗。
- C 太严格，可能因为一次搜索关键词不准就错失可补全线索。
- B 更适合小范围运行：有成本边界，又保留二次尝试空间。


### 已确认决定 6

选择 **B：受控配额**。

深挖线索 Agent 运行配额：

- 必须由人工点击 **“深挖线索”** 后启动。
- 每条线索每天最多 1-2 次深挖。
- 每次深挖最多生成 3-5 个搜索查询。
- 每次深挖最多读取 5-10 个公开链接。
- Watch / Invalid / 勿扰 / Forbidden 线索不允许深挖。
- 深挖失败或无有效补全结果时，必须记录失败原因，避免重复消耗。

专业判断：受控配额可以控制 LLM 成本、搜索读取成本和低质量线索反复消耗，同时保留关键词不准时的二次修正空间。

## 维度 7：无法完善线索的退出机制

### 初步专业建议

第三阶段不能只设计“晋级客户”，也必须设计“不晋级”的路径。否则线索池会堆积大量无法承接的数据，客服/销售会失去对系统的信任。

建议无法完善线索的处理动作：

1. **继续补全**
   - 缺少少量关键字段，仍有明确来源或线索价值。
   - 可再次人工补录或触发下一次深挖。

2. **调整等级**
   - AI 或人工发现线索价值低于原等级。
   - 可从 B/C 降为 A、Watch 或 Invalid。

3. **转 Watch**
   - 信息不足、暂时无法判断，但不应立即删除。
   - Watch 不进入触达队列，不进入客户工作台。

4. **转 Invalid**
   - 明确不是车商、无车辆销售相关性、联系方式无效、重复、媒体/招聘/维修/配件等无效类型。
   - Invalid 不进入触达队列，不进入客户工作台。

5. **放弃线索**
   - 多次深挖无结果、人工判断不值得继续处理。
   - 放弃必须记录原因，后续用于关键词和渠道质量优化。

6. **标记勿扰**
   - 客户拒绝联系或出现明确不再联系要求。
   - 勿扰为硬状态，不得再次进入触达队列。

建议所有退出动作都写入审计，并进入失败案例/低质量案例统计，用于优化 Source Discovery、Lead Extraction 和 Deep Enrichment prompt。

### 待确认问题 7

无法完善线索的默认处理策略建议采用哪一种？

A. 保守保留：无法完善的线索都留在待补全队列，避免错失机会。

B. 有限保留：允许继续补全 1-2 轮，仍不满足准入则必须转 Watch/Invalid/放弃，并记录原因。

C. 快速清理：一次深挖失败就直接放弃，保证队列干净。

**我的建议：选择 B。**

理由：

- A 会让线索池长期堆积，客服/运营难以判断优先级。
- C 过于激进，可能错过需要人工补录或二次关键词调整的线索。
- B 更适合小范围运行：保留有限尝试空间，同时强制退出，保证队列质量。


### 已确认决定 7

选择 **B：有限保留**。

无法完善线索的默认处理策略：

- 允许继续补全 1-2 轮。
- 补全仍不满足客户最低准入条件时，必须转 Watch、Invalid 或放弃。
- 转 Watch、Invalid 或放弃必须记录原因。
- 无法完善线索不得长期停留在待补全队列。
- 退出原因进入失败案例/低质量案例统计，用于优化关键词、渠道、prompt 和深挖策略。

专业判断：该策略可以避免线索池无限堆积，同时保留有限纠错和人工补录空间。

## 维度 8：客户数据模型与现有表关系

### 初步专业建议

第三阶段应继续坚持 raw/staging/core 分层：

```text
staging_leads               原始线索与抽取结果
lead_enrichment_results     线索深挖/人工补全批次
lead_enrichment_field_candidates 字段级候选与采纳记录
customers                   已准入客户主档
lead_sources                客户来源证据
contact_methods             客户联系方式
outreach_records            触达历史
customer_followups          跟进记录，若现有 outreach_records 不足可新增
customer_vehicle_intents    客户意向车型，建议新增或扩展现有匹配表
compliance_reviews          C 级合规复核
```

建议不要把所有客户业务数据都塞进 `customers` 一张表。`customers` 应保持客户主档，车型意向、跟进记录、触达历史、联系方式、来源证据都应独立建模，便于一对多管理和审计。

推荐表设计方向：

1. `lead_enrichment_results`
   - 记录 AI 深挖或人工补录批次。
   - 关联 `staging_leads`。
   - 保存输入快照、输出摘要、缺失字段、推荐动作、任务审计。

2. `lead_enrichment_field_candidates`
   - 字段级候选值。
   - 支持采纳、修改、拒绝。

3. `customer_vehicle_intents`
   - 客户意向车型。
   - 字段：品牌、车型、年份/车龄、数量、预算、采购频率、交付地、关注点、状态、来源。

4. `customer_followups`
   - 客户跟进记录。
   - 字段：客户 ID、负责人、跟进方式、跟进内容、客户反馈、下一步动作、下一次跟进时间、是否触发勿扰/合规。

5. `customer_assignments` 或扩展客户表字段
   - 负责人、团队、分配状态、SLA。

### 待确认问题 8

客户意向车型和跟进记录的建模方式建议采用哪一种？

A. 简化建模：先把意向车型和跟进记录都放在 `customers` 的 JSON 字段中，开发最快。

B. 正规建模：新增 `customer_vehicle_intents` 和 `customer_followups` 独立表，保留一对多和审计能力。

C. 混合建模：短期 JSON，后续再迁移为独立表。

- JSON 开发快，但后续查询、筛选、统计、SLA 和销售交接会很快受限。
- 第三阶段目标已经是客户管理工作台，建议一开始就把意向和跟进独立建模，避免很快返工。


### 已确认决定 8

选择 **B：正规建模**。

第三阶段客户意向车型和跟进记录采用独立表：

- 新增 `customer_vehicle_intents`，承载客户意向车型、品牌、车型、年份/车龄、数量、预算、采购频率、交付地、关注点、状态和来源。
- 新增 `customer_followups`，承载客户跟进记录、负责人、跟进方式、客户反馈、下一步动作、下一次跟进时间、是否触发勿扰/合规等。
- `customers` 保持客户主档，不承载大量一对多业务过程数据。
- `contact_methods` 继续承载客户联系方式。
- `lead_sources` 继续承载客户来源证据。
- `outreach_records` 继续承载触达历史；如与 follow-up 记录边界重叠，后续产品设计阶段需明确二者关系。

专业判断：客户意向车型和跟进记录天然是一对多关系，独立建模更利于后续筛选、统计、SLA、销售交接和审计。

## 维度 9：页面与操作入口边界

### 初步专业建议

第三阶段建议明确两个工作台职责：

1. **线索池 / 线索详情**
   - 处理尚未晋级客户的对象。
   - 主要操作：深挖线索、字段级采纳、人工补录、建议晋级、手动准入、调整等级、转 Watch/Invalid、放弃线索。
   - 核心目标：把不完整线索变成可承接客户，或明确退出。

2. **客户工作台 / 客户详情**
   - 处理已经晋级客户的对象。
   - 主要操作：分配负责人、查看客户档案、管理联系方式、记录触达历史、维护意向车型、记录跟进、发起合规复核、交付客服/销售。
   - 核心目标：推进客户转化和团队承接。

不建议在客户工作台里继续做大量线索补全，也不建议在线索池里做完整 CRM 跟进。否则页面职责会混乱，用户不知道应该在哪里操作。

推荐入口：

- 线索列表卡片：增加“深挖线索”“补录”“建议晋级/手动准入”“放弃”。
- 线索详情页：增加“完善区”Tab，展示 AI/人工字段候选和采纳状态。
- 客户工作台：展示客户状态、负责人、下一步动作、意向车型摘要和最近触达。
- 客户详情页：展示客户主档、联系方式、来源证据、意向车型、触达历史、跟进记录、合规状态。
- 客户跟进页：新增/编辑跟进记录，并可设置下一次跟进时间。

## 维度 10：合规、审计与硬门禁

### 初步专业建议

第三阶段客户进入工作台后，系统风险从“信息采集风险”扩展到“触达、报价、销售交接风险”。因此需要把合规规则做成硬门禁，而不是页面提示。

建议硬门禁：

1. **勿扰门禁**
   - 客户或线索一旦标记勿扰，禁止生成触达草稿、禁止记录新的主动触达、禁止进入客服/销售触达队列。
   - 只能允许查看历史、记录被动回复或合规备注。

2. **C 级复核门禁**
   - C 级客户可进入销售候选。
   - 在报价、合同、付款、物流、清关、交付周期等动作前，必须存在通过状态的合规复核。

3. **来源证据门禁**
   - 客户晋级时必须有来源证据。
   - AI 深挖补全字段必须有字段级证据或人工来源说明。

4. **自动触达禁止门禁**
   - 系统不得自动发送邮件、短信、社媒消息或加好友。
   - 触达助手只能生成草稿和记录人工发送。

5. **字段审计门禁**
   - 手动补录、字段采纳、字段修改、客户晋级、客户放弃都必须记录操作人和时间。

6. **High/Forbidden 门禁**
   - High 来源客户必须保留风险标签和审核记录。
   - Forbidden 来源不得作为客户晋级关键来源。

建议审计事件：

- `lead_deep_enrichment_started`
- `lead_enrichment_field_accepted`
- `lead_enrichment_field_rejected`
- `lead_promoted_to_customer`
- `lead_marked_watch_invalid_abandoned`
- `customer_assigned`
- `customer_outreach_draft_generated`
- `customer_manual_outreach_recorded`
- `customer_do_not_contact_marked`
- `customer_compliance_review_requested`
- `customer_compliance_review_passed_or_rejected`

### 待确认问题 10

第三阶段的合规门禁强度建议采用哪一种？

A. 提示型门禁：页面提示风险，但允许用户继续操作。

B. 硬阻断门禁：勿扰、Forbidden、C 级报价前复核、自动触达禁止等关键规则必须阻断操作。

C. 审批型门禁：所有客户晋级、触达、跟进都必须先审批。

**我的建议：选择 B。**

理由：

- A 风险过高，用户可能为了效率绕过关键规则。
- C 过重，会让小范围运行效率过低。
- B 把关键高风险动作硬阻断，把普通跟进留给业务人员操作，安全与效率平衡最好。


### 已确认决定 10

选择 **B：硬阻断门禁**。

第三阶段关键合规规则采用硬阻断：

- 勿扰客户不得生成触达草稿、不得记录新的主动触达、不得进入客服/销售触达队列。
- Forbidden 来源不得作为客户晋级关键来源。
- C 级客户在报价、合同、付款、物流、清关、交付周期等动作前必须完成合规复核。
- 系统不得自动发送邮件、短信、社媒消息或自动加好友。
- 客户晋级必须有来源证据。
- AI 深挖补全字段必须有字段级证据或人工来源说明。
- 手动补录、字段采纳、字段修改、客户晋级、客户放弃必须记录操作人和时间。

普通客户跟进动作保留业务人员操作空间，但触达、报价、合规和勿扰相关动作必须由系统硬阻断。

专业判断：硬阻断门禁能防止关键规则被效率压力绕过，同时不会把所有客户工作都变成审批流，适合第三阶段小范围运行。

## 维度 11：第三阶段指标、验收与暂停条件

### 初步专业建议

第三阶段必须用指标验证“线索完善到客户承接”是否成立。建议指标分为四层：

1. **线索完善效率**
   - 深挖触发线索数。
   - 深挖成功补全率。
   - 字段级采纳率。
   - 人工补录完成率。
   - 平均补全耗时。

2. **客户晋级质量**
   - `staging_leads -> customers` 晋级率。
   - 晋级客户中联系方式完整率。
   - 晋级客户中有意向车型比例。
   - 晋级客户重复率/误晋级率。
   - 放弃/Watch/Invalid 原因分布。

3. **团队承接效率**
   - 客户分配及时率。
   - 首次人工触达及时率。
   - 客户回复率。
   - C 级销售承接率。
   - 跟进记录完整率。

4. **风险与合规**
   - 勿扰误入队列次数，目标为 0。
   - 自动触达违规次数，目标为 0。
   - C 级未复核报价/合同次数，目标为 0。
   - 无证据字段采纳次数，目标为 0。
   - High/Forbidden 违规晋级次数，目标为 0。

建议第三阶段 Go/No-Go：

- Go：深挖补全有效、客户晋级质量可接受、客服/销售愿意承接、风险事件为 0。
- 重跑：深挖关键词/渠道质量不足，但流程和数据模型可行。

## 补充维度：Watch/Invalid 线索清洗 Agent

### 用户补充需求

在进入实施优先级确认前，新增一个 LLM Agent，对线索池中的 Watch / Invalid 线索进行清洗，包含：

- 重复线索识别。
- 重复线索更新补充。
- 无效线索清洗过滤。
- 低质量线索原因归类。
- 可恢复线索重新进入待补全或深挖队列。

### 初步专业建议

该 Agent 建议命名为 **Lead Cleanup Agent** 或 **线索清洗 Agent**，职责独立于深挖线索 Agent。

#### 与深挖线索 Agent 的区别

| Agent | 处理对象 | 目标 | 输出 |
|---|---|---|---|
| 深挖线索 Agent | 人工选择的潜在有效线索 | 补全客户准入字段 | 字段级补全候选、建议晋级 |
| 线索清洗 Agent | Watch / Invalid / 疑似重复 / 低质量线索 | 清理、归并、纠错、恢复可用线索 | 去重建议、归并建议、无效原因、恢复建议 |

#### 建议处理范围

线索清洗 Agent 可处理：

1. **强重复线索**
   - 客户名称 + 联系方式一致。
   - 来源 URL/hash 一致。
   - 官网域名一致且城市一致。

2. **疑似重复线索**
   - 客户名称相似。
   - 城市一致。
   - 联系方式部分重叠。
   - 来源域名相同但页面不同。

3. **可恢复 Watch 线索**
   - 原来信息不足，但新来源或重复线索中补充了联系方式/城市/客户名。
   - 可从 Watch 恢复到 A/B，并进入待补全或深挖候选。

4. **确认无效线索**
   - 维修、配件、保险、招聘、媒体、论坛个人帖、非车辆销售、纯资讯站。
   - 无法联系、明显重复、来源违规、非目标国家/地区。

5. **重复线索补充主记录**
   - 重复线索不直接删除。
   - 可将联系方式、来源证据、经营信号、车型线索补充到主线索或客户记录。
   - 被归并记录标记 `duplicate_of_id`，保留审计。

#### 建议硬规则

- 清洗 Agent 不得自动删除线索，只能标记、归并建议或执行受控归并。
- Watch / Invalid 不得直接进入触达队列。
- 从 Watch 恢复到 A/B/C 必须有新证据或人工确认。
- Invalid 恢复必须人工确认，不能由 LLM 单独恢复。
- 重复归并必须保留来源证据和字段来源。
- 勿扰状态必须向主记录传播，不能因归并丢失。

#### 建议数据结构

可新增或扩展：

1. `lead_cleanup_runs`
   - 记录一次清洗任务。
   - 字段：`id`、`trigger_source`、`status`、`input_filter_json`、`output_summary_json`、`llm_provider`、`llm_model`、`prompt_template_id`、`started_at`、`finished_at`。

2. `lead_cleanup_suggestions`
   - 记录清洗建议。
   - 字段：`id`、`cleanup_run_id`、`staging_lead_id`、`suggestion_type`、`target_lead_id`、`confidence_score`、`reason`、`evidence_json`、`recommended_action`、`review_status`、`reviewer_id`、`reviewed_at`。

`suggestion_type` 建议包括：

- `strong_duplicate`
- `possible_duplicate`
- `merge_contact_method`
- `merge_source_evidence`
- `restore_from_watch`
- `confirm_invalid`
- `mark_abandoned`
- `needs_manual_review`

#### 建议页面入口

- 线索池新增 Tab：`待清洗`、`疑似重复`、`可恢复 Watch`、`确认无效`。
- 线索详情显示“清洗建议”：归并到哪条主线索、补充哪些字段、为什么无效、是否可恢复。
- 管理后台新增清洗统计：重复率、恢复率、确认无效率、归并补充字段数。

A. 聚焦闭环：只做线索完善、客户晋级、客户工作台、跟进记录、清洗 Agent 和核心指标。

## 维度 14：角色、权限与人工操作边界

### 初步专业建议

第三阶段涉及多个高影响操作：字段采纳、手动准入、清洗恢复、重复归并、客户分配、C 级合规复核、勿扰标记。建议不要一开始做复杂 RBAC，但必须做最小角色权限边界。

建议角色：

1. **运营**
   - 可触发深挖线索。
   - 可人工补录。
   - 可字段级采纳/拒绝。
   - 可提交建议晋级。
   - 可处理清洗建议，但恢复 Invalid 或合并到客户需更高权限确认。

2. **客服**
   - 可查看已分配客户。
   - 可记录人工触达和跟进。
   - 可补充客户反馈、意向车型初步信息。
   - 可标记客户拒绝联系/勿扰。
   - 不可绕过 C 级合规复核。

3. **销售**
   - 可查看 C 级和销售候选客户。
   - 可维护意向车型、采购数量、预算、交付关注点。
   - 可记录销售跟进。
   - 报价/合同前必须触发并通过合规复核。

4. **合规/管理员**
   - 可审核 C 级复核。
   - 可确认 High/Forbidden 边界相关争议。
   - 可恢复 Invalid。
   - 可确认高风险重复归并。
   - 可查看完整审计。

建议最小权限原则：

- 手动准入客户：运营、管理员可做；客服/销售可提交补录，但不直接准入未满足条件线索。
- 恢复 Invalid：必须管理员或合规确认。
- 标记勿扰：客服、销售、运营都可做，且立即生效。
- 取消勿扰：必须管理员或合规确认。
- C 级报价前复核：必须合规/管理员通过。
- 重复归并：强重复可运营确认；疑似重复和客户级归并建议管理员确认。

### 待确认问题 14

第三阶段权限模型建议采用哪一种？

A. 简化权限：所有登录用户都能深挖、补录、晋级、归并、恢复和跟进。

B. 最小角色权限：运营、客服、销售、合规/管理员分工，关键高风险动作需要更高权限确认。

C. 完整 RBAC：每个按钮、字段、状态都配置细粒度权限。

**我的建议：选择 B。**

理由：

- A 风险过高，容易误晋级、误恢复 Invalid、误取消勿扰或绕过 C 级复核。
- C 对小范围运行过重，会拖慢开发和验证。
- B 足够覆盖关键风险，同时保持实现成本可控。


### 已确认决定 14

选择 **B：最小角色权限**。

第三阶段采用运营、客服、销售、合规/管理员的最小角色权限模型：

- 运营：可触发深挖线索、人工补录、字段级采纳/拒绝、提交或执行符合条件的手动准入、处理普通清洗建议。
- 客服：可查看已分配客户、记录人工触达和跟进、补充客户反馈与初步意向车型、标记勿扰。
- 销售：可查看 C 级和销售候选客户、维护意向车型、采购数量、预算、交付关注点、记录销售跟进。
- 合规/管理员：可审核 C 级复核、处理 High/Forbidden 争议、恢复 Invalid、确认高风险重复归并、取消勿扰、查看完整审计。

关键权限边界：

- 手动准入客户：运营、管理员可做；必须满足最低准入和硬门禁。
- 恢复 Invalid：必须合规/管理员确认。
- 标记勿扰：运营、客服、销售均可做，立即生效。
- 取消勿扰：必须合规/管理员确认。
- C 级报价/合同前复核：必须合规/管理员通过。
- 强重复归并：运营可确认；疑似重复和客户级归并建议管理员确认。

专业判断：该权限模型能覆盖第三阶段关键风险，同时不引入完整 RBAC 的复杂度。

## 维度 15：第三阶段 Agent 与现有 Agent 的边界

### 初步专业建议

第三阶段新增两个 Agent，但不应替代第二阶段已有 Agent：

1. **Source Discovery Agent**
   - 负责发现来源候选。
   - 输出 `lead_source_candidates`。
   - 不负责客户补全和客户晋级。

2. **Lead Extraction Agent**
   - 负责从已准入来源抽取初始线索。
   - 输出 `staging_leads`。
   - 不负责深挖补全和客户晋级。

3. **Lead Deep Enrichment Agent / 深挖线索 Agent**
   - 由人工在线索详情点击触发。
   - 输入单条或少量 staging 线索。
   - 目标是补全客户准入字段。
   - 输出字段级补全候选和建议晋级。
   - 不直接写 `customers`。

4. **Lead Cleanup Agent / 线索清洗 Agent**
   - 定时扫描 Watch / Invalid / 疑似重复 / 低质量线索。
   - 输出清洗建议。
   - 关键动作由人工确认。
   - 不自动删除线索，不自动恢复 Invalid。

建议服务编排关系：

```text
Source Discovery Agent -> lead_source_candidates
Lead Extraction Agent  -> staging_leads
Deep Enrichment Agent  -> lead_enrichment_results / field_candidates
Cleanup Agent          -> lead_cleanup_runs / cleanup_suggestions
Promotion Service      -> customers / lead_sources / contact_methods
Customer Workbench     -> customers + intents + followups + outreach history
```

如果后续采用 LangChain / LangGraph，建议先把第三阶段两个 Agent 设计成显式节点流程：

- 深挖线索图：`load_lead -> build_keywords -> search_public_sources -> extract_candidates -> validate_evidence -> write_enrichment_candidates -> recommend_action`
- 清洗线索图：`load_watch_invalid -> detect_duplicates -> classify_invalid_reason -> find_restore_candidates -> write_cleanup_suggestions -> wait_human_review`

但短期仍应保留现有 FastAPI + APScheduler + PostgreSQL 审计作为主控，不要一次性重构全部 Agent。

### 待确认问题 15

第三阶段两个新增 Agent 的编排方式建议采用哪一种？

A. 继续沿用现有自研服务编排：FastAPI Service + APScheduler + AgentThreadRunner + PostgreSQL 审计，先不引入 LangGraph。

B. 第三阶段开始局部试点 LangGraph：只在深挖线索 Agent 和清洗 Agent 内使用显式图流程，外层仍由现有调度和审计主控。

C. 全面迁移到 LangGraph：把 Source Discovery、Lead Extraction、Deep Enrichment、Cleanup 全部重构为 LangGraph。

**我的建议：选择 B。**

理由：

- A 最稳，但深挖和清洗都存在多步骤、分支、人工确认等待，继续纯自研会让流程状态越来越分散。
- C 风险过高，会影响第二阶段已经跑通的来源发现和线索抽取链路。
- B 是更适合当前现状的折中方案：第三阶段新增 Agent 用 LangGraph 表达复杂流程，现有 Agent 暂不重构，外层调度、锁、审计和数据库仍沿用当前架构。


### 已确认决定 15

选择 **B：第三阶段开始局部试点 LangGraph**，并补充确认：

- 新建独立 Agent 项目来做局部试点 LangGraph。
- 新 Agent 项目优先承载：
  - `Lead Deep Enrichment Agent` / 深挖线索 Agent。
  - `Lead Cleanup Agent` / Watch/Invalid 线索清洗 Agent。
- 现有后端 `Source Discovery Agent`、`Lead Extraction Agent`、APScheduler、Redis lock、`agent_task_runs` 审计暂不重构。
- 外层调度、任务审计、数据库写入、人工确认、风险硬门禁仍由现有 FastAPI 后端主控。
- LangGraph 只负责新增 Agent 内部的多步骤流程表达、分支、状态流转和人工等待点。

建议新建 Agent 项目定位：

```text
apps/agents 或 apps/agent-runtime
  -> LangGraph workflows
  -> Deep Enrichment graph
  -> Cleanup graph
  -> shared schemas
  -> tools: LLMClient / PublicSearch / PublicPageReader / EvidenceValidator
  -> API/worker adapter connecting apps/api
```

推荐边界：

- Agent 项目不直接绕过后端写 core 表。
- Agent 项目输出结构化结果，由 `apps/api` 的 Service 层校验后写入 PostgreSQL。
- Agent 项目可以读任务输入、调用 LLM、搜索公开来源、生成候选字段或清洗建议。
- Agent 项目不得自动晋级客户、自动归并客户、自动恢复 Invalid、自动触达客户。
- 所有 Agent run 必须回写任务状态和审计信息。

专业判断：

- 独立 Agent 项目可以隔离 LangGraph 试点风险，避免影响第二阶段已跑通的调度和抽取链路。
- 新增 Agent 的流程复杂度更高，适合用图表达；现有 Source Discovery / Lead Extraction 当前已可运行，不宜一次性迁移。
- 后续如果 LangGraph 试点稳定，再评估是否将 Lead Extraction 或 Source Discovery 迁移到同一编排体系。

## 当前头脑风暴阶段性结论

第三阶段已形成以下核心方向：

1. 客户晋级采用平衡准入：客户名称、国家/城市、至少一个联系方式、来源证据、勿扰校验通过。
2. 保留人工补全和手动准入。
3. 深挖线索由人工点击触发，AI 补全到 staging 完善区，人工确认后晋级。
4. 字段级采纳，保留字段来源、证据、置信度和人工采纳审计。
5. 客户工作台默认按下一步动作优先级排序。
6. A/B 级交给客服，C 级进入销售候选但报价/合同前必须合规复核。
7. 深挖 Agent 采用受控配额。
8. 无法完善线索有限保留，1-2 轮后必须 Watch/Invalid/放弃。
9. 意向车型和跟进记录独立建模。
10. 页面新增客户工作台、客户详情、跟进记录；线索页只负责晋级前动作。
11. 勿扰、Forbidden、C 级复核、自动触达禁止等采用硬阻断门禁。
12. 核心成功指标为有效客户承接率。
13. 新增 Watch/Invalid 线索清洗 Agent，定时生成建议，人工确认执行。
14. 第一批开发从数据底座、线索晋级 API 和清洗建议 API 开始。
15. 第三阶段范围聚焦闭环，不扩展成完整 CRM。
16. 权限采用运营、客服、销售、合规/管理员的最小角色权限模型。
17. 新建独立 Agent 项目局部试点 LangGraph，优先承载深挖线索 Agent 和清洗 Agent。
