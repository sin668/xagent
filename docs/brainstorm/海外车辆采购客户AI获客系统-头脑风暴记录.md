# 海外车辆采购客户 AI 获客系统 - 头脑风暴记录

创建日期：2026-05-25
最近更新：2026-05-26

## 1. 原始业务目标

通过 AI 与现有 Agent 工具，面向海外车辆采购客户完成线索发现、基础信息整理、初步触达、意向筛选、客户沉淀、CRM 登记与交付给客服/出口/线下团队。

## 2. 已知客户来源渠道

- YouTube
- Facebook
- Telegram
- X
- 企业官网
- 地图标注信息
- 其他因国家和市场不同而变化的本地渠道

## 3. 关键关注点

- 海外平台对自动化、AI 操作、批量触达、中国地区访问或账号行为的限制与封禁政策
- 方案是否可持续落地，而不只是技术上“能跑”
- 数据库与 CRM 的客户信息结构
- 客户交付到客服、出口团队、线下团队的流程
- 初期可以暂不做深度意向筛选，但应保留后续筛选能力

## 5. 当前讨论阶段：战略维度

### 待确认问题

- 战略优先级：先验证单一市场的成交闭环，还是先搭建多市场线索池？
- 第一阶段优先市场：中亚/俄罗斯周边。

### 初步建议

- 建议优先选择 1-2 个目标国家或区域做 MVP，而不是一开始覆盖所有渠道和国家。
- MVP 的战略目标应定义为“稳定产出可交付的合格线索”，而不是“最大化抓取数量”。
- 平台封禁与合规风险应作为战略约束，而不是上线后再补救的问题。

## 6. 已确认决定

- 第一阶段采用“先验证成交闭环”的战略路径：选择 1-2 个国家或区域，跑通“找客户 -> 触达 -> 回复 -> 登记 -> 交付团队”的完整流程。
- 平台封禁、AI 自动化限制、中国地区访问或账号风险作为前置风控动作，与成交闭环验证同步进行。
- 第一阶段不追求最大化抓取数量，而追求稳定产出可交付、可跟进、可沉淀的客户线索。
- 第一阶段市场方向选择：中亚/俄罗斯周边。
- 该市场方向需要重点评估：制裁与贸易限制、跨境支付、物流路线、车辆准入、本地平台可访问性、社交平台触达风险。
- 第一阶段主攻客户画像：当地车商/二级经销商。
- 第一阶段拓展渠道线索：个人买家、KOL、汽车博主引流。
- 进口贸易商/清关物流相关公司暂不作为第一阶段重点，但后续可作为合作伙伴线索补充。
- 第一阶段主打车辆品类：二手车、准新车、库存车。
- 主打卖点需要围绕：价格优势、车况透明、配置清单、可出口手续、物流路线、清关协作、交付周期。
- 第一阶段主战场：俄罗斯。
- 俄罗斯市场机会较大，但必须将制裁、支付、物流、平台账号、数据合规和触达合规作为战略红线，在任何自动化获客方案前先做可行性判断。
- 第一阶段成功指标：以“有效线索”为主，以“触达回复”为辅，以“成交机会”为观察指标。
- 有效线索建议至少包含：客户名称、国家/地区、经营类型、联系方式、来源渠道、来源链接、车辆相关性、初步备注、数据采集时间。
- 触达回复指标需要在平台规则允许和账号风控可控的前提下验证，不以违规批量私信作为增长手段。
- 成交机会作为观察指标，用于判断客户是否可交付给出口团队、客服团队或线下团队继续推进。
- AI 角色定位：长期战略目标为“AI + 人工混合 SDR 系统”，MVP 阶段先实现“AI 线索研究员 + AI 销售助理”。
- MVP 阶段 AI 负责搜索、识别、整理、去重、分类客户，并生成触达话术、客户画像和跟进建议；具体触达动作由人工审核执行。
- 第一阶段不建议直接做 AI 自动加好友、批量私信、自动跟进，避免平台封禁、垃圾信息、误触达和合规风险。
- 第一阶段不做事项：
  - 不做大规模自动私信或自动加好友。
  - 不承诺全平台覆盖。
  - 不做复杂意向评分模型，先采用规则标签和人工复核。
  - 不直接把未验证线索推给销售团队。
  - 不绕过平台规则做规避式自动化。
- 第一阶段节奏：采用“2 周 PoC + 4 周 MVP”的两段式策略。
- 2 周 PoC 目标：验证俄罗斯车商线索能否稳定找到、整理、去重、人工触达，并初步评估渠道封禁和触达风险。
- 4 周 MVP 目标：形成稳定线索数据库、渠道 SOP、人工审核触达流程、CRM 登记结构和客户交付机制。
- 竞品分析重点顺序：汽车出口同行/车源平台 > 俄罗斯本地车商既有采购网络 > 数据采集/销售线索工具 > 传统外贸获客服务商。
- 竞品维度首先关注业务竞品和替代采购路径，而不是先研究工具功能。
- 竞品研究目标产出：
  - 渠道地图：识别俄罗斯汽车出口/采购相关客户与同行活跃平台。
  - 客户触达话术库：沉淀价格、车况、物流、清关、付款、交付周期等卖点表达。
  - 竞品卖点/报价结构表：了解同行展示车型、价格区间、库存结构和批发能力。
  - 平台风控与可行性清单：判断哪些渠道适合公开采集、人工触达、半自动辅助或暂不进入。
- 后续竞品调研边界：以公开网页调研为主，平台内行为验证暂不自动化。
- 平台内注册、登录、加好友、私信、入群、批量查看等高风险行为，仅做政策研究和人工小样本验证，不作为第一阶段 AI 自动化动作。
- 俄罗斯竞品/渠道调研优先级：搜索引擎与官网/地图标注 + 中国出口同行公开渠道先行，俄罗斯本地汽车平台 + 社交/内容平台跟进。
- 先行渠道价值：更容易通过公开信息采集客户和竞品资料，平台风控相对较低，适合 PoC 快速产出有效线索。
- 跟进渠道价值：俄罗斯本地汽车平台和社交/内容平台更贴近真实客户与市场，但需先评估使用条款、反爬限制、账号风险和触达边界。
- 竞品研究同时作为销售话术库建设输入。
- 每个竞品或渠道样本除记录身份和来源外，还应提取供应能力话术、信任建立话术和合作模式话术。
- 话术库后续用于 AI 销售助理生成触达内容，但 MVP 阶段仍需人工审核后发送。
- 业务承接流程：AI 先负责线索去重、信息补全、初筛和打标签，然后再分配给客服或出口销售。
- 客服团队处理低/中意向线索，负责初步沟通、联系方式确认、微信/WhatsApp/Telegram 引导和基础需求确认。
- 出口销售团队处理高意向线索，负责车型匹配、报价、合同、付款、物流和清关等深入推进。
- 线下/当地合作团队作为后续能力接入，暂不作为第一阶段必备承接方。
- 线索交付标准：
  - A 级基础线索：客户名称、国家/城市、来源链接、客户类型、至少一个联系方式；先入库补全，不急于交付。
  - B 级增强线索：在 A 级基础上增加主营车型、是否卖二手/进口车、社交/官网活跃度、初步匹配原因；可交付客服团队。
  - C 级高意向线索：在 B 级基础上增加已回复、表达采购兴趣、询价、愿意添加微信/WhatsApp/Telegram 或提供需求；可交付出口销售团队。
- 第一阶段不把“注册微信”作为硬性指标，而把“可持续联系”作为硬性指标。
- 联系方式策略：尊重客户习惯，支持 WhatsApp、Telegram、Email、电话、VK、微信等多种方式；对 C 级高意向客户再引导微信，便于深入报价、发送车源和成交跟进。
- 线索跟进 SLA：B 级线索需在 48 小时内首次跟进，C 级线索需在 24 小时内首次跟进。
- CRM 或登记表中必须记录：分配时间、首次跟进时间、跟进人、跟进结果、下一步动作。
- 第一阶段增加轻量车源/报价表，不直接接入复杂 ERP 或库存系统。
- 轻量车源/报价表建议字段：品牌、车型、年份、里程、配置、车况、价格、可出口状态、图片/视频、有效期、备注。
- 轻量车源/报价表用于支持 AI 生成初步回复建议，并帮助客服/销售快速匹配客户需求。
- 客户经营状况第一阶段不做复杂黑箱评分，采用可解释字段：
  - 活跃度：高/中/低/未知。
  - 规模信号：大/中/小/未知。
  - 进口/二手相关性：强/中/弱/未知。
  - 证据链接/备注：记录判断依据，便于人工复核。
- 触达自动化边界：
  - 2 周 PoC：完全人工发送，AI 只生成草稿和建议。
  - 4 周 MVP：可尝试人工确认后的半自动发送，但仅限邮件或官网表单等低风险渠道。
  - 社交平台私信、加好友、入群互动等动作不纳入第一阶段自动化范围，仅允许人工小样本、可控验证。
- 产品形态路径：2 周 PoC 采用“表格 + Agent 工作流”，4 周 MVP 升级为“轻量 CRM 看板”，第一阶段不做完整 SaaS 系统。
- PoC 阶段用表格承载客户库、车源/报价表、渠道记录和触达记录；Agent 负责搜索、补全、打标签、去重和生成话术建议。
- MVP 阶段增加线索列表、状态流转、分配、跟进记录、车源表和话术建议等轻量 CRM 能力。
- PoC 阶段至少建立 5 张表：
  - 客户线索表：客户名称、国家、城市、客户类型、联系方式、来源链接、等级、标签、状态、负责人。
  - 渠道来源表：平台、关键词、搜索方式、国家/语言、采集规则、风险等级、备注。
  - 车源/报价表：品牌、车型、年份、里程、车况、价格、可出口状态、图片/视频、有效期。
  - 触达记录表：客户、触达渠道、话术版本、发送人、发送时间、回复状态、下一步。
  - 话术库表：场景、语言、渠道、客户类型、话术内容、风险提示、适用条件。
- 线索状态流转：
  - 新采集：AI 或人工刚找到，还没清洗。
  - 待补全：缺联系方式、地区、客户类型、来源证据等关键信息。
  - 待复核：AI 已补全并打标签，需要人工确认是否有效。
  - 可交付客服：达到 B 级标准，适合客服初步跟进。
  - 客服跟进中：客服已接手，正在确认需求或联系方式。
  - 可交付销售：达到 C 级标准，已回复或有明确需求。
  - 销售跟进中：出口销售接手报价、车型匹配、合同等。
  - 无效/暂不匹配：重复、无联系方式、非目标客户、风险过高、不相关。
  - 沉淀观察：暂时无明确意向，但有价值，后续可再营销。
- AI 在产品中的呈现方式：
  - 优先做 AI 建议面板：每条线索显示推荐等级、推荐原因、缺失信息、下一步建议和推荐话术。
  - 保留 AI/Agent 证据日志：记录 Agent 做过哪些搜索、访问了哪些来源、提取了哪些证据、为什么判断该客户有效。
  - 后续增加 AI 批处理能力：例如一键去重、一键补全、一键生成俄语触达话术、一键分级，但所有批处理结果进入人工复核队列。
- MVP 角色范围：
  - 线索运营/管理员：查看全部线索、渠道、AI 日志、重复项、分配和质量统计。
  - 客服：查看分配给自己的 B 级线索、推荐话术、联系方式、跟进记录和下一步动作。
  - 出口销售：查看 C 级高意向线索、客户需求、匹配车源、报价建议和跟进历史。
  - 管理者：MVP 阶段先通过简单统计页或表格汇总查看渠道效果、线索数量、有效率、回复率、SLA 和团队跟进情况。
- 语言策略：第一阶段系统内部以中文为主，面向客户的内容优先俄语；后续再逐步补成中俄双语体系。
- 外部客户内容包括：触达话术、车源介绍、报价摘要、跟进模板、常见问题回复。
- 第一阶段入口形态：网页看板 + 表格同步。
- 网页看板负责主要操作、分配、跟进和查看；表格负责底层数据、导入导出和初期手工维护。
- 线索详情页建议包含 6 个区域：
  - 客户基础信息：名称、国家、城市、客户类型、等级、状态、负责人、联系方式。
  - 来源与证据：来源平台、来源链接、截图/摘录、采集时间、AI 判断依据。
  - 经营状况判断：活跃度、规模信号、进口/二手相关性、证据备注。
  - AI 建议：推荐等级、缺失信息、推荐下一步、俄语触达草稿、风险提醒。
  - 跟进记录：跟进人、时间、渠道、客户回复、下一步动作。
  - 车源匹配：根据客户类型或需求推荐可发的二手/准新/库存车。
- 消息发送产品边界：
  - PoC 阶段：生成话术 + 记录发送结果，系统不直接发送消息。
  - MVP 后半段：可尝试人工确认后的邮件或官网表单发送。
  - 第一阶段不做系统内社交私信、加好友或入群自动化。
- 产品定位优先作为“话术和触达记录中心”，而不是“自动群发工具”。
- MVP 做轻量数据仪表盘，不做复杂 BI。
- 仪表盘指标包含：
  - 渠道指标：每个渠道采集数量、有效线索数、B/C 级线索数、无效率。
  - 触达指标：已触达数、回复数、回复率、愿意继续沟通数。
  - 团队跟进指标：B/C 级线索 SLA 是否超时、客服/销售处理量、待处理队列。
  - 业务机会指标：询价数、车型需求数、车源匹配数、进入销售推进数。
- 技术栈定位：Agent 负责采集与辅助处理，LLM 负责理解与生成，CRM/数据库负责状态，人工负责审核和触达。
- Codex/Claude Code 等研发型 Agent 适合用于搭建表格结构、脚本、CRM 原型、数据清洗逻辑、文档和 SOP。
- Hermes/OpenClaw 等浏览器/操作型 Agent 适合用于低风险公开页面调研、采集流程验证和平台可访问性检查，但必须受合规边界约束。
- LLM 核心能力用于信息抽取、翻译、分类、去重辅助、客户画像、俄语触达话术生成和跟进建议。
- 数据底座路径：PoC 使用飞书表格承载协作数据，MVP 再同步到 PostgreSQL 等结构化数据库。
- 飞书表格用于快速验证字段、流程、人工复核和团队协作；数据库用于后续轻量 CRM、权限控制、审计和扩展。
- 采集技术优先级：搜索引擎 + 公开网页抓取 > 平台官方 API > 浏览器辅助采集。
- 第一阶段明确不做模拟登录、批量操作、反爬规避或绕过平台限制的采集方式。
- 浏览器辅助采集仅用于小样本验证和公开页面流程验证，不作为大规模采集手段。
- MVP 技术栈方向：
  - 移动端：uni-app + Vue3。
  - Web 前端：Vue3。
  - 后端：Python + FastAPI。
  - 数据库：PostgreSQL。
  - 原则：优先采用现有成熟主流框架和组件，不在第一阶段自研基础设施。
- 后端 MVP 采用 FastAPI + PostgreSQL，建议配套 SQLAlchemy/SQLModel、Alembic，以及 Celery/RQ/Arq 等任务队列方案之一。
- 如果需要后台管理能力，优先补充轻量 Admin，不以 Django 作为第一阶段主框架。
- AI/LLM 接入策略：PoC 直接调用大模型 API + 简单脚本，MVP 再引入轻量 Agent 编排。
- AI 能力优先用于网页信息抽取、俄语翻译、客户分类、去重辅助、话术生成和跟进建议。
- 每次 AI 判断都需保存输入、输出、模型、时间、证据链接和执行结果，便于审计、复核和纠错。
- 多 Agent 协同适合研发和流程验证，不直接暴露给业务用户作为黑箱自动执行能力。
- 飞书表格与 MVP 数据库同步策略：
  - PoC 阶段使用手动导入导出 CSV。
  - MVP 阶段使用飞书 API 定时同步到 PostgreSQL。
  - 第一阶段暂不做双向同步，避免数据冲突。
  - 后续系统稳定后，逐步迁移到数据库为主，飞书仅作为导入导出或轻量协作入口。
- 任务执行策略：
  - PoC 阶段不上任务队列，使用脚本手动运行并写入飞书。
  - MVP 阶段增加轻量任务队列，处理批量搜索、网页抓取、AI 抽取、翻译、去重、表格同步、话术生成和 SLA 提醒。
  - 第一阶段暂不引入 Airflow/Temporal 等重型工作流平台。
- 安全与合规总原则：
  - 平台规则优先：每个渠道先评估 ToS、robots、公开 API、账号规则和自动化限制，再决定采集与触达方式。
  - 数据保护优先：对客户联系方式、聊天记录、来源证据、AI 判断记录做权限控制、脱敏、审计和最小化访问。
  - 贸易与制裁合规前置：俄罗斯市场涉及车辆品类、交易主体、支付路径、物流路线和客户身份等风险，必须在业务推进前做红线判断。
- 平台风险分级：
  - 低风险渠道：官网公开联系方式、公开企业目录、地图标注、公开搜索结果；可做公开采集和人工触达。
  - 中风险渠道：YouTube、公开 Telegram 频道、公开 VK 页面、公开论坛；可做公开信息提取，但触达需人工、小量、保留记录。
  - 高风险渠道：需要登录的平台、私域群、批量私信、加好友、频繁访问、模拟人工操作；不做自动化，只允许人工小样本验证。
  - 禁止渠道/行为：绕过反爬、盗用账号、购买异常账号、批量骚扰、规避平台限制、抓取非公开数据。
- MVP 中每个渠道需配置风险等级、允许动作和禁止动作；Agent 执行前必须读取渠道规则，不允许自行越界。
- 客户数据保存原则：采用数据最小化原则，只保存业务跟进必要信息。
- 来源链接优先于大量保存网页原文；关键证据可保存有限摘录或截图，但不做无差别页面快照。
- 聊天记录仅保存业务相关摘要和关键沟通，不保存无关私人内容。
- 联系方式、聊天记录、来源证据和 AI 判断记录需要做权限控制和访问审计。
- 俄罗斯贸易与制裁风险处理方式：
  - 建立贸易合规红线清单，覆盖车型/车辆品类、交易主体、地区、支付路径、物流路线、清关和最终用途等风险点。
  - C 级高意向线索在报价、合同或实质交易推进前，必须进入人工合规复核节点。
  - AI 可以提示潜在风险，但不能替代法务、贸易合规或业务负责人判断。
  - 如进入实际交易阶段，建议尽早引入外部法务或贸易合规顾问做专项确认。
- AI 生成话术合规机制：
  - 先建立经过审核的话术模板，再由 AI 在模板框架内生成具体内容。
  - AI 输出时同时给出风险提示，例如价格承诺、交付承诺、清关承诺、平台敏感行为、过度营销或翻译歧义。
  - 所有对外话术在第一阶段必须人工确认后发送。
  - 不同渠道配置不同话术规则，邮件、官网表单、Telegram、WhatsApp、VK 等渠道不使用同一套触达尺度。
- 平台政策核查范围：
  - 搜索与公开网页：Google、Yandex、企业官网、公开目录、地图标注。
  - 俄罗斯本地平台：Avito、Drom、Auto.ru、VK。
  - 国际社交/内容平台：YouTube、Facebook/Instagram、X、Telegram。
  - 通讯与触达渠道：Email、WhatsApp、Telegram、官网表单。
- 平台政策核查输出字段：允许动作、谨慎动作、禁止动作、账号风险、中国地区风险、建议替代方案。
- PoC 执行只从低风险渠道开始，高风险渠道先做政策研究和人工小样本验证。

### 平台政策初步联网核查记录

核查时间：2026-05-25。以下为公开资料初步判断，后续进入执行前需再次核查官方最新条款。

- YouTube：官方条款禁止未经许可使用机器人、botnet、scraper 等自动方式访问服务，公共搜索引擎需遵守 robots.txt 或取得书面许可。PoC 不应直接抓取 YouTube 页面或自动化互动，优先使用公开搜索结果、官方 API 或人工查看。来源：https://tv.youtube.com/tv/terms
- X：官方开发者指南明确强调应使用官方 API，非 API 的 scraping 和浏览器自动化属于高风险/禁止方向；自动化账号和主动动作需遵守自动化规则。PoC 不做 X 抓取、自动关注、自动私信。来源：https://docs.x.com/developer-guidelines ，https://help.x.com/articles/20174732
- Meta/Facebook/Instagram：Meta 对自动化数据收集有专门条款和权限边界，未授权 scraping 风险高。PoC 不做 Facebook/Instagram 自动抓取和自动私信，仅做公开资料人工/小样本核查。来源：https://www.facebook.com/legal/automated_data_collection_terms
- Telegram：官方服务条款禁止 spam/scam；Telegram 对陌生人骚扰和异常消息行为有账号限制机制。PoC 不做 Telegram 自动私信、自动加群、批量触达，只研究公开频道/群介绍和人工小样本验证。来源：https://telegram.org/tos/eu ，https://www.telegram.org/faq_spam/
- WhatsApp：WhatsApp Business 政策强调用户体验、反垃圾信息、官方 Business API、模板消息和 24 小时窗口等规则。PoC 不使用非官方 WhatsApp Web 自动群发；MVP 如接入 WhatsApp，应走官方 Business Platform/合规 BSP，并基于用户同意和人工确认。来源：https://whatsappbusiness.com/policy/
- Google Maps/Places：Google Maps Platform 条款包含 No Scraping，Places API 对缓存/存储也有限制。PoC 不把 Google Maps 作为可自由抓取的数据源；如使用地点数据，应优先官方 API，并注意存储限制。来源：https://cloud.google.com/maps-platform/terms ，https://developers.google.com/maps/documentation/places/web-service/policies
- Drom：公开资料显示 Drom 有面向卖家的自动上传/授权工具和许可协议，但这不等于允许第三方抓取数据。PoC 可把 Drom 作为人工研究和渠道观察对象，自动化采集需进一步核查许可。来源：https://www.drom.ru/misc/autoload/ ，https://www.drom.ru/commerce/license/
- Auto.ru：Auto.ru 有正式用户协议，执行前需进一步核查其中关于自动访问、数据使用、账号行为和商业触达的具体条款。来源：https://yandex.ru/legal/autoru_terms_of_service/ru/index.html
- Avito/VK/Yandex：仍需进一步核查最新官方条款、API 文档、robots 和账号规则；暂按中高风险渠道处理，不进入 PoC 自动化采集范围。
- 账号与身份策略：
  - PoC 阶段不登录平台，只做公开资料研究和低风险公开信息整理。
  - MVP 阶段可对低/中风险渠道建立合规专用业务账号，明确账号用途、负责人、允许动作、禁止动作和审计记录。
  - 公司官方账号只用于人工、正式沟通，不用于自动化实验或高风险平台行为。
  - 不使用个人账号作为正式运营资产，不购买异常账号，不做批量账号规避。
- 客户拒绝/勿扰机制：
  - CRM 中设置“拒绝联系/勿扰”状态。
  - 客户明确拒绝后，停止所有渠道继续触达，不通过更换渠道绕过客户意愿。
  - 邮件或官网表单触达时提供停止联系选项或明确的拒绝回复路径。
  - AI 生成触达任务或话术时自动排除“勿扰”客户。
- 可扩展性优先级：国家/区域 > 渠道 > 车辆品类 > 自动化程度。
- 扩展原则：先复制已验证成交闭环，再逐步扩大渠道和品类，最后提升自动化程度。
- 新增国家/区域时需要配置化：
  - 国家/地区配置：语言、时区、币种、城市、主要口岸、物流路线。
  - 渠道配置：每个国家常用平台、关键词、风险等级、允许动作、禁止动作。
  - 客户画像配置：车商、KOL、个人买家、租赁/车队等不同客户类型权重。
  - 话术与合规配置：语言、称呼、触达习惯、敏感承诺、合规红线。
  - 车源匹配配置：偏好车型、价格区间、排放/准入规则。
- 第一阶段架构中应优先将国家、渠道、话术和风险规则做成配置，避免系统被俄罗斯市场写死。
- 权限扩展路径：
  - PoC 阶段不做复杂权限，依赖飞书表格权限和人工管理。
  - MVP 阶段做角色权限和客户归属，区分线索运营、客服、出口销售和管理者。
  - 第二阶段增加国家/区域权限，实现不同区域团队的数据隔离和协作。
- AI 扩展路径：
  - PoC 阶段采用单一大模型，降低复杂度。
  - MVP 阶段预留多模型路由和轻量评测能力，可按抽取、翻译、分类、话术等任务选择模型。
  - 第二阶段再引入多 Agent 分工，例如搜索 Agent、抽取 Agent、合规 Agent、话术 Agent、CRM Agent。
  - MVP 阶段开始建立轻量评测样本集，用于测试抽取准确率、俄语质量和分类一致性。
- 数据结构扩展方向：
  - PoC 表格阶段可以采用一条线索对应一个主要来源，但字段设计需预留多来源能力。
  - MVP 数据库采用“客户主体 + 多来源记录 + 多联系方式”的结构。
  - 第二阶段可进一步扩展为“客户主体 + 联系人 + 渠道账号 + 门店/分支机构”。
- 2 周 PoC 最小交付物：
  - 俄罗斯渠道与竞品初版地图：以搜索/官网/地图/中国出口同行为主，并标注平台风险等级。
  - 飞书五张表：客户线索、渠道来源、车源/报价、触达记录、话术库。
  - 100-200 条初步线索：至少达到 A/B 级，重点验证能否稳定形成 B 级线索。
  - AI 清洗与补全脚本/流程：去重、分类、俄语/中文摘要、推荐等级、推荐话术。
  - 人工触达小样本：不做自动发送，选少量 B/C 级线索人工触达并记录回复。
  - PoC 复盘报告：渠道有效率、线索质量、回复情况、平台风险和 MVP 建议。
- 2 周 PoC 不做网页 CRM 开发，重点验证业务闭环和渠道可行性。
- 4 周 MVP 交付范围：
  - 轻量 CRM Web 看板：线索列表、状态流转、分配、筛选和详情页。
  - 飞书 API 单向同步到 PostgreSQL。
  - AI 建议面板：推荐等级、缺失信息、下一步、俄语话术、风险提示。
  - Agent/AI 执行日志：保存来源、输入、输出、模型、证据链接和执行结果。
  - 车源/报价表与线索匹配。
  - 触达记录与 SLA：B 级 48 小时，C 级 24 小时。
  - 轻量仪表盘：渠道、触达、SLA、业务机会。
  - 安全合规配置：渠道风险等级、允许动作、禁止动作、勿扰状态。
- 4 周 MVP 不做自动社交触达、不做复杂 BI、不做多国家完整扩展。
- PoC 验收标准：
  - 渠道可行性：至少找到 3 个可持续获取俄罗斯车商线索的低/中风险渠道。
  - 线索质量：100-200 条线索中，至少 30%-50% 可达到 B 级增强线索。
  - AI 效率：AI 能稳定完成去重、补全、分类、摘要和话术建议，人工复核后准确率达到可接受水平。
  - 触达反馈：小样本人工触达后，有真实回复或明确负反馈；不要求成交，但要能判断客户是否愿意沟通。
  - 风险可控：没有账号封禁、平台投诉、明显违规采集或批量骚扰行为。
  - 团队可接：客服/销售认为线索字段、话术和交付方式可用。
- PoC 通过标准：渠道可持续 + B 级线索比例达标 + 小样本触达有反馈 + 风险可控 + 团队愿意接。
- PoC/MVP 团队职责：
  - 业务负责人：定义目标市场、车辆品类、线索质量、销售承接标准，并最终判断 PoC 是否继续。
  - 线索运营/数据负责人：管理飞书表格、去重、补全、复核和分配。
  - AI/Agent 操作负责人：运行搜索、抽取、清洗、话术生成流程，记录执行日志和问题。
  - 客服负责人：承接 B 级线索，小样本触达，反馈话术效果。
  - 出口销售负责人：承接 C 级线索，判断客户需求、车型匹配和报价可行性。
  - 合规/风控负责人：审核平台风险、话术边界、贸易红线和客户勿扰机制。
- PoC 阶段可以一人兼多岗，但每项职责必须有人负责。
- PoC 暂停或转向条件：
  - 有效线索比例过低：100-200 条线索中，B 级线索低于 20%，需调整渠道或客户画像。
  - 触达反馈极差：小样本触达几乎无人回复，或多数明确拒绝，需调整话术、渠道或需求假设。
  - 平台风险过高：出现账号限制、投诉、访问异常或明显违反条款风险，应停止该渠道。
  - 销售团队不愿承接：字段不够、线索不准、需求不匹配，需重做交付标准。
  - 贸易/支付/物流不可行：客户有兴趣但实际交易路径不可落地，需调整市场策略。
- 暂停/转向机制不是失败机制，目标是在证据基础上调整渠道、关键词、地区、话术或客户画像。
- 新增知识库/SOP 维度，作为第 9 个讨论维度。
- 知识库/SOP 建议沉淀：
  - 渠道 SOP：每个平台怎么找客户、用什么关键词、什么动作允许/禁止。
  - 线索审核 SOP：A/B/C 级怎么判、哪些证据必须有、如何去重。
  - 触达 SOP：不同渠道怎么发、俄语话术怎么审核、拒绝后怎么处理。
  - 销售交付 SOP：B 级给客服、C 级给销售、什么情况下升级或退回。
  - 合规 SOP：平台政策、贸易红线、账号规则、客户勿扰。
- 知识库/SOP 作为 PoC 交付物之一，避免经验只停留在人脑里。
- SOP 维护方式：
  - PoC 阶段使用飞书文档 + 表格备注维护，方便业务团队协作和快速更新。
  - MVP 阶段将关键 SOP 嵌入 CRM 页面，例如渠道风险提示、线索评级规则、话术审核提醒。
  - 研发侧同步维护 Markdown 文档，便于版本管理和后续系统实现。
- 建立关键词库作为 PoC 核心资产之一。
- 关键词库分类：
  - 客户关键词：汽车经销商、二手车 dealer、进口车、批发、车行、auto dealer 等俄语/英语/中文表达。
  - 车辆关键词：二手车、准新车、库存车、中国车、进口车、批发车源、品牌/车型关键词。
  - 渠道关键词：Telegram channel、VK group、dealer website、auto import Russia、Drom dealer 等。
  - 排除关键词：招聘、维修、配件、保险、贷款、新闻、无关论坛帖等。
  - 地区关键词：Moscow、Saint Petersburg、Vladivostok、Novosibirsk、Kazan 等俄语/英语地名。
- 关键词库按语言、渠道、客户类型和有效率持续迭代。
- 建立失败案例库作为 PoC 核心资产之一。
- 失败案例库记录：
  - 无效线索案例：为什么看起来像客户，但实际不是目标客户。
  - 错误分类案例：AI 把维修店、配件商、媒体号等误判为车商。
  - 触达失败案例：话术不合适、渠道不合适、客户明确拒绝。
  - 平台风险案例：某渠道访问异常、账号限制、疑似违反规则。
  - 交易不可行案例：客户有兴趣，但支付、物流、清关、车型不适配。
- 失败案例库用于提升 AI 规则、关键词排除逻辑和人工审核质量。
- 建立客户 FAQ/异议处理库，采用中文内部版 + 俄语客户版。
- FAQ/异议处理库覆盖：
  - 价格类：批发价、议价、是否包含物流/清关、最新报价表。
  - 车况类：事故车、里程、检测报告、图片/视频。
  - 出口与物流类：能否发俄罗斯、到达周期、口岸、清关责任。
  - 付款与信任类：付款方式、合同、车辆真实性、过往案例。
  - 合作模式类：长期供货、定向找车、批发价更新、Telegram/WhatsApp/微信。
- AI 生成客户回复时必须优先引用经过审核的 FAQ/异议处理库，不允许自由承诺价格、物流、清关或付款条件。
- 新增成本与 ROI 维度，作为第 10 个讨论维度。
- PoC 阶段需记录基础成本，不做复杂财务模型，但要能判断渠道是否值得继续投入、AI 是否节省人力。
- 成本与 ROI 维度关注：
  - 线索成本：每条有效线索的人工时间、AI 成本、工具成本。
  - 触达成本：每次人工触达、翻译、跟进、客服处理成本。
  - 渠道 ROI：不同渠道产出的 B/C 级线索比例、回复率、机会率。
  - 团队效率：AI 介入前后，每人每天可处理线索数量。
  - 成交机会价值：每个 C 级线索潜在订单量、毛利、复购可能。
- PoC 设置四类成本上限：
  - 时间上限：2 周内完成，不无限延期。
  - 人力上限：限定每天用于线索采集、复核和触达的投入时间。
  - 工具/API 成本上限：限制 AI/API/数据工具等费用。
  - 渠道试错上限：某渠道采集 30-50 条仍没有 B 级线索时，暂停或调整该渠道。
- ROI 指标层级：
  - PoC 阶段关注每条有效线索成本和每条回复成本。
  - MVP 阶段在 PoC 指标基础上增加每个销售机会成本。
  - 长期阶段关注每个成交成本和复购价值。
- 最终收敛方案：采用“方案 B：2 周 PoC + 4 周 MVP 连续推进”。
- 选择理由：先以低风险方式验证俄罗斯二手/准新/库存车商线索闭环，再将验证有效的流程产品化为轻量 CRM 和 AI 辅助工作流。
- 不采用方案 A 作为最终方案的原因：仅做 PoC 难以形成团队协作和持续运营能力。
- 不采用方案 C 作为最终方案的原因：直接产品化建设会在渠道、线索质量和合规风险尚未验证前过度开发。
