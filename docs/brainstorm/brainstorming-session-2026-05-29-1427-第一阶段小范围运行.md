# 第一阶段小范围运行 - BMAD 头脑风暴记录

创建时间：2026-05-29 14:27 CST

## Session Overview

**主题：** 海外车辆采购 AI 获客系统进入第一阶段小范围运行。

**目标：**

1. 继续探索 Low/Medium 风险渠道收集线索，拓展更多渠道和方式。
2. 探索 High 风险渠道如何相对安全地收集线索。
3. 补充更多可能的线索来源和获客方式。
4. 讨论线索收集智能体如何实现，以及如何接入大模型。
5. 讨论知识库如何实现，以及如何接入大模型。
6. 第一阶段不再使用飞书表格，所有数据直接进入 PostgreSQL。
7. 补充第一阶段小范围运行需要讨论的其他维度。

## 已继承的核心边界

- 不自动社交私信。
- 不自动加好友。
- 不登录后批量采集。
- 不做反爬规避。
- 勿扰客户不得再次进入触达队列。
- AI 输出必须保留来源证据、模型输入输出和审计记录。
- C 级线索在报价、合同或实质交易前必须合规复核。
- High 风险渠道默认不进入自动任务，只允许政策研究、人工小样本或转化为 Low/Medium 来源后二次取证。

## 讨论记录

### 维度 1：Low/Medium 风险渠道拓展

#### 已确认决定

第一阶段小范围运行采用 **A + B + C 并重**：

- A 类稳定低风险主渠道：官网、公开企业目录、搜索引擎结果、品牌/经销商官网列表。
- B 类中风险补充渠道：Google/Yandex 地图、公开论坛、Drom 公开商户页、公开视频描述页。
- C 类行业链路渠道：汽车展会名录、协会会员名单、B2B 目录、物流/清关合作伙伴转介绍。

#### 专业建议

- A 类渠道负责稳定产出可审计线索，作为 PostgreSQL 直入库的主数据来源。
- B 类渠道负责扩大覆盖面，但只允许低频、人工核验、小批量公开文本处理；优先把平台内线索迁移到官网、公开邮箱、公开电话等可复核来源。
- C 类渠道质量可能更高，但拓展成本和商务协作成本也更高；第一阶段应与 A/B 并行探索，但建议控制样本量，作为高质量来源池建设。

#### 运行边界

- Low 渠道可以进入智能体任务队列，但必须保留来源链接和证据摘录。
- Medium 渠道可以进入小范围运行，但需要渠道规则、限量策略和异常暂停机制。
- High/Forbidden 渠道不得混入 A/B 自动任务。

### 维度 2：High 风险渠道相对安全的线索发现方式

#### 已确认决定

第一阶段允许 **AI/浏览器 Agent 读取公开 High 风险页面**，但必须满足以下前提：

- 不登录。
- 不互动。
- 不触达。
- 不自动私信。
- 不自动加好友。
- 不入群。
- 不抓取非公开数据。
- 不绕过验证码、访问限制、频率限制或反爬机制。
- 不保存粉丝、好友、评论关系链、私信内容、个人画像等非业务必要信息。

High 风险渠道在第一阶段的定位不是正式采集源，而是 **只读发现入口**。如果从 High 页面发现客户名称、官网、公开邮箱、公开电话、WhatsApp、Telegram 或其他公开商务联系方式，需要转到官网、地图、公开目录或搜索结果进行二次复核后再入库。

#### 专业建议

- High 渠道任务应单独建任务类型，例如 `high_risk_public_discovery`，不得和 Low/Medium 自动采集任务混跑。
- 每次任务必须声明平台、关键词、页面 URL、执行人/执行 Agent、风险等级、允许动作、禁止动作和最大样本数。
- 浏览器 Agent 只允许执行“打开公开 URL、读取公开文本、截取有限证据、提取跳转官网/公开商务联系方式”。
- High 渠道提取结果默认状态应为 `Watch` 或 `Needs Low/Medium Verification`，不得直接进入触达队列。
- 只有当线索在 Low/Medium 来源完成二次取证后，才允许进入 `待复核` 或 `待触达`。
- 出现验证码、登录墙、访问异常、平台警告或账号限制时，立即暂停该渠道任务。

#### 建议数据字段

- `source_risk_level`: High。
- `source_usage_type`: public_discovery_only。
- `requires_secondary_verification`: true。
- `secondary_verification_source`: 官网/地图/公开目录/搜索结果。
- `agent_allowed_actions`: read_public_page, extract_business_contact, capture_limited_evidence。
- `agent_forbidden_actions`: login, message, follow, friend_request, join_group, scrape_comments, scrape_followers, bypass_rate_limit。
- `queue_eligible`: false，直到二次复核完成。

### 维度 3：其他可能的渠道和方式

#### 已确认决定

其他渠道拓展采用 **数量为主，质量观察池为辅**：

- 主流程继续扩大搜索、地图、公开目录、公开平台覆盖，目标是获得更多线索样本。
- 同步建立质量观察池，每周补充少量高质量渠道样本，例如协会、展会、B2B、物流/清关、检测/维修机构、进口车服务商等。
- A/B/C 三类渠道并重，不再把 C 类行业链路渠道后置。

#### 专业建议

- A 类负责稳定性，B 类负责覆盖面，C 类负责质量和成交接近度。
- 第一阶段可以为每类渠道设定不同 KPI：
  - A 类：线索数量、联系方式完整率、来源证据完整率。
  - B 类：新增发现率、二次复核成功率、异常/风控事件数。
  - C 类：B/C 级线索比例、可交付率、销售愿意承接比例。
- C 类渠道不宜用“批量采集”思路，应更像“行业网络拓展”：少量、高质量、强证据、强人工复核。

#### 候选补充渠道池

- 汽车展会官网参展商名录。
- 俄罗斯/中亚汽车经销商协会会员名录。
- 汽车批发市场、汽车城、二手车市场官网或地图页。
- B2B 平台公开企业页。
- 物流、清关、报关、检测、维修机构的合作客户或公开案例。
- 竞品公开客户案例、评论区中出现的企业名，仅作为发现入口，不采集个人互动数据。
- 俄语汽车论坛公开商户帖。
- 本地黄页、商会、工商目录。

### 维度 4：线索收集智能体与大模型接入

#### 已确认决定

第一阶段线索收集智能体采用 **受控自动发现**：

- Agent 可以根据渠道规则、关键词库、地区配置自动生成搜索任务和候选 URL。
- Agent 可以读取允许范围内的公开页面文本。
- Agent 可以调用大模型进行抽取、补全建议、分级、风险提示和下一步建议。
- Agent 可以将结构化结果直接写入 PostgreSQL。
- Agent 不允许自动社交私信、自动加好友、登录后批量采集、反爬规避或绕过平台限制。

#### 专业建议

不要做单一“大 Agent”，而是拆成受控的任务型智能体：

1. **渠道发现 Agent**
   - 输入：国家、城市、渠道、关键词、风险等级、每日样本上限。
   - 输出：候选 URL、候选客户名、渠道风险、发现理由。
   - 限制：只生成候选，不直接触达。

2. **公开页面读取 Agent**
   - 输入：候选 URL、渠道规则。
   - 输出：公开文本、页面标题、少量证据摘录、读取状态。
   - 限制：不登录、不绕过、不高频、不采集非公开内容。

3. **线索抽取 Agent**
   - 输入：公开文本、来源 URL、渠道元数据。
   - 输出：客户名称、国家、城市、客户类型、联系方式、活跃度、规模信号、进口/二手相关性、证据备注。
   - 限制：缺失字段必须输出 Unknown、null 或空数组，不允许编造。

4. **分级与风控 Agent**
   - 输入：抽取结果、渠道风险、证据、业务规则。
   - 输出：A/B/C/Invalid/Watch、推荐原因、缺失信息、下一步动作、建议交付团队、是否需要合规复核。
   - 限制：Invalid/Watch 不进入触达队列；C 级报价/合同前必须合规复核；High 来源必须二次复核。

5. **入库与审计 Agent**
   - 输入：结构化线索、AI 输入输出、任务日志。
   - 输出：PostgreSQL 记录、审计日志、去重结果、复核队列项。
   - 限制：任何无来源证据的 AI 输出不得进入有效线索池。

#### 大模型接入建议

- 第一阶段采用“规则 + LLM”混合方式，不完全依赖大模型判断。
- LLM 用于非结构化文本理解、俄语/英语/中文翻译、客户画像、分级原因生成和缺失信息识别。
- 规则用于渠道权限、风险等级、勿扰、重复检测、C 级复核、触达队列准入。
- 每次 LLM 调用保存：
  - prompt 版本。
  - 输入文本摘要或引用。
  - 输出 JSON。
  - 模型名称。
  - 时间。
  - 来源链接。
  - 任务 ID。
  - 校验结果。

#### 推荐任务流

`channel_plan -> search_task -> candidate_url -> page_snapshot -> llm_extraction -> rule_validation -> llm_grading -> dedupe -> postgres_upsert -> human_review_queue`

#### 第一阶段自动化边界

- 可自动：生成搜索任务、读取公开允许页面、抽取、分级、去重、入库、生成复核建议。
- 需人工：High 风险二次确认、B/C 线索复核、触达发送、客户回复判断、销售交付确认。
- 禁止：自动社媒私信、自动加好友、登录批采、反爬规避、编造联系方式、绕过勿扰。

### 维度 5：知识库与大模型接入

#### 已确认决定

第一阶段知识库采用 **PostgreSQL + pgvector 统一承载**：

- PostgreSQL 保存结构化业务规则、渠道规则、线索、任务、审计、复核记录和交付记录。
- pgvector 保存语义型知识条目的向量，用于 RAG 检索。
- 不再使用飞书表格作为主数据载体。

#### 知识库分层

1. **规则型知识库**
   - 渠道风险规则。
   - 允许动作/禁止动作。
   - 字段字典。
   - 线索分级规则。
   - 勿扰规则。
   - C 级合规复核规则。
   - 贸易合规红线。
   - 任务准入和暂停条件。

2. **语义型知识库**
   - 俄语关键词库。
   - FAQ 与异议处理。
   - 俄语触达模板。
   - 车型、车况、配置、报价知识。
   - 物流、清关、付款、合作模式说明。
   - 竞品卖点和公开案例。
   - 失败案例库。
   - 渠道 SOP。

#### 大模型接入方式

- LLM 在执行抽取、分级、话术生成、复核建议前，先检索相关知识条目。
- 规则型知识不作为“参考建议”，而作为硬约束由程序执行。
- 语义型知识作为 RAG 上下文提供给 LLM，要求模型引用知识条目 ID 或来源。
- 所有 LLM 输出必须写入 `ai_audit_logs`，保存输入、输出、模型、prompt 版本、知识条目引用、来源证据和校验结果。

#### 推荐表设计方向

- `knowledge_collections`: 知识库集合，例如 channel_sop、faq、script_template、vehicle_knowledge、compliance_rules。
- `knowledge_items`: 知识条目，保存标题、正文、语言、国家、适用渠道、状态、来源、审核状态。
- `knowledge_embeddings`: 条目向量，使用 pgvector。
- `rule_configs`: 结构化规则配置，例如渠道动作边界、等级准入、触达限制。
- `prompt_versions`: prompt 模板和版本。
- `ai_audit_logs`: LLM 调用审计。

#### 专业建议

- 第一阶段只需要做“够用的 RAG”，不要做复杂知识图谱。
- 规则必须结构化，不能只放在文档里让大模型理解。
- 知识条目需要状态字段：draft、reviewing、approved、deprecated。
- 只有 approved 知识才允许进入生产 Agent 检索。
- 每次知识库更新应记录版本，避免话术和规则漂移。

### 维度 6：PostgreSQL 直入库替代飞书表格

#### 已确认决定

PostgreSQL 直入库采用 **宽进严出**：

- AI/Agent 采集和抽取结果先进入 `raw` 与 `staging`。
- 只有规则校验、去重、来源证据检查、人工复核通过后，才进入 `core` 正式业务表。
- 所有 AI 调用、规则判断、人工操作和状态变化进入 `audit`。
- 飞书表格不再作为第一阶段主数据载体。

#### 推荐数据分层

1. **raw 层**
   - 保存采集任务、候选 URL、页面标题、公开文本摘要、来源证据、截图/摘录引用、采集状态。
   - 允许重复，允许低质量，主要用于追溯。

2. **staging 层**
   - 保存 AI 抽取后的候选客户、联系方式、城市、客户类型、经营信号、推荐等级、推荐原因。
   - 缺失字段保留 Unknown、null 或空数组。
   - Invalid、Watch、High 未二次复核线索不能进入触达队列。

3. **core 层**
   - 正式客户表、联系方式表、来源表、车源/报价表、触达记录表、分配/交付表、合规复核表。
   - 只保存复核通过或规则准入通过的数据。

4. **audit 层**
   - AI 输入输出。
   - prompt 版本。
   - 模型名称。
   - 知识库引用。
   - 来源证据。
   - 操作人/Agent。
   - 状态流转。
   - 错误和风险事件。

#### 入库闸门

- 来源链接缺失：不得进入 core。
- 证据备注缺失：不得进入 core。
- 联系方式缺失：可进入 staging，但不得进入待触达队列。
- Invalid/Watch：不得进入触达队列。
- High 来源：必须完成 Low/Medium 二次复核后才能进入 core。
- C 级：进入 core 后仍需合规复核，报价/合同前必须阻断。
- 勿扰：不得再次进入触达队列。

#### 专业建议

- 第一阶段先实现 `raw -> staging -> core` 的最小闭环，不急于做复杂数据湖。
- 所有 upsert 都需要幂等键，例如 source_url_hash、normalized_name + contact_hash、external_source_id。
- PostgreSQL 中应明确区分 `source_risk_level`、`lead_grade`、`review_status`、`queue_status`，避免一个状态字段承载太多含义。
- 管理后台应优先做 staging 复核页面，而不是先做复杂 CRM 页面。

### 维度 7：第一阶段运行治理、指标、成本、合规与暂停机制

#### 已确认决定

第一阶段小范围运行的 Go/No-Go 核心判断指标采用 **有效线索数量优先**。

#### 专业建议

虽然主指标是有效线索数量，但必须设置辅助硬闸门，避免只追数量导致数据污染或合规风险：

- 有效线索数量：主指标。
- B/C 级线索比例：质量辅助指标。
- 来源证据完整率：数据可信度硬指标。
- 联系方式完整率：可触达性指标。
- High 误入自动任务次数：必须为 0。
- 勿扰误入触达队列次数：必须为 0。
- 投诉、封禁、验证码异常、平台警告：触发暂停评估。
- 销售/客服愿意承接比例：判断是否真有业务价值。

#### 建议 Go/No-Go 口径

- Go：连续 1-2 周能稳定产出足量有效线索，且来源证据完整、合规事件为 0、客服/销售愿意承接。
- 重跑：有效线索数量不足，但渠道无明显合规风险，说明关键词、渠道组合或抽取策略需要调整。
- 暂停：出现投诉/封禁/违规风险、High 渠道越界、勿扰误触达、AI 编造严重、销售完全不愿承接。

#### 第一阶段建议观察指标

- 每日新增候选 URL 数。
- 每日进入 staging 的候选线索数。
- 每日进入 core 的有效线索数。
- 各渠道有效线索数。
- 来源链接完整率。
- 证据备注完整率。
- 联系方式完整率。
- 重复率。
- Invalid/Watch 比例。
- B/C 线索比例。
- 人工复核通过率。
- 每条有效线索平均 LLM 成本。
- 每条有效线索平均人工复核耗时。
- 风险事件数。

### 维度 8：待进一步收敛的问题

#### 已确认决定：运行规模

第一阶段采用 **小规模运行**：

- 每天约 100 条候选线索。
- 控制每日候选 URL、staging 候选线索、core 有效线索和人工复核数量。
- 以验证自动发现、PostgreSQL 直入库、AI 抽取分级、人工复核和渠道风险控制为主。

#### 专业建议

建议初始运行配额：

- 每日候选 URL：100-200 个。
- 每日 staging 候选线索：约 100 条。
- 每日 core 有效线索目标：20-50 条。
- 单个 Medium 渠道每日样本：20-50 条。
- 单个 High 渠道每日只读公开页面样本：5-20 条，且必须二次复核。
- 每日人工复核上限：不超过实际客服/运营可处理量。

## 阶段性设计摘要

### 渠道策略

- A/B/C 并重，拓展更多线索来源。
- A 类低风险渠道提供稳定基础样本。
- B 类中风险渠道扩大覆盖面，但要限量、可暂停、可追溯。
- C 类行业链路渠道并行探索，用于提高线索质量。
- High 风险渠道允许 AI/浏览器 Agent 只读公开页面，但不登录、不互动、不触达，结果必须二次复核后才可入正式流程。

### 智能体策略

- 使用受控自动发现，不做完全自主 Agent。
- 拆分为渠道发现、公开页面读取、线索抽取、分级风控、入库审计、人工复核队列等任务型 Agent。
- LLM 用于文本理解、抽取、翻译、分级原因、缺失信息识别和话术建议。
- 规则用于渠道边界、风险等级、勿扰、去重、C 级复核和触达准入。

### 数据策略

- 不再使用飞书表格作为主载体。
- PostgreSQL 采用 raw/staging/core/audit 四层。
- 采用宽进严出：先收集和抽取，再校验、去重、复核，最后进入正式客户库。
- 知识库采用 PostgreSQL + pgvector，结构化规则与语义 RAG 统一承载。

### 运行指标

- Go/No-Go 主指标：有效线索数量。
- 辅助硬闸门：来源证据完整、High 不越界、勿扰不误触达、无投诉/封禁/违规风险、客服/销售愿意承接。

### 初始规模

- 每天约 100 条候选线索。
- 每日 core 有效线索目标建议 20-50 条。
- High 渠道只读公开页小样本，且不直接进入触达队列。
