# AI协同开发执行标准

**文档类型**: 规范标准  
**版本号**: V1.0  
**文档状态**: 项目适配版  
**创建日期**: 2026-05-05  
**最后更新**: 2026-05-09  
**适用范围**: XAgents 仓库  
**维护人**: 项目协作团队

---

## 1. 目标

本标准用于统一 XAgents 当前仓库的 AI 协同方式，避免 `BMAD`、`Codex`、`superpowers`、`gstack`、`Frontend-Design` 以及各类自动化脚本在同一阶段重复判断、重复产出、互相覆盖结论，或者在缺少真相源的情况下直接推进实现。

本项目采用的默认分工是：

- `BMAD`：负责业务收敛、PRD、架构、Epic / Story、Readiness、变更分级
- `Codex`：负责读取代码与文档、实现单个 Story、执行命令、修复问题、回写结果
- `superpowers`：负责计划拆解、TDD、系统化调试、完工前验证
- `gstack`：负责独立评审、QA、安全检查、发布前收口
- `Frontend-Design`：负责 UI / UX 设计任务、原型与体验方案收敛，并确保设计输出符合仓库正式设计规范

核心原则只有一句话：

**一个阶段只认一个主负责人。**

---

## 2. 项目适配结论

XAgents 不是一个泛 AI 助手项目，而是围绕 **海外获客高质量线索闭环** 的协同系统。当前仓库的协同开发标准，必须首先服务这个业务边界。

当前项目的适配结论如下：

- UI / UX 设计任务（含原型、交互、视觉、信息架构、组件样式、主题与颜色策略）默认由开源工具 `Frontend-Design` 主负责。
- 所有 UI / UX 设计产物及相关实现，必须符合 `docs/01_标准规范/UI-UE设计规范_V1.00.md` 与 `docs/01_标准规范/颜色规范使用指南_V1.00.md`。
- 所有页面型需求必须遵守 `PRD -> UX -> 角色/页面矩阵 -> 关键原型确认 -> Epic/Story 开发 -> 页面域验收` 的顺序；详细规则见 `docs/01_标准规范/页面产品化收口与原型确认执行标准_V1.00.md`。
- `prototypes/mvp-mobile-agent/` 下的原型预览可作为体验参考，但在未被冻结为正式真相源前，不能替代 PRD、架构、Story 和 validation。

推荐研发操作系统：

```text
BMAD 负责冻结边界与拆解任务
Codex 负责按 Story 实现与回写
superpowers 负责计划、TDD、调试与完工前验证
gstack 负责独立评审、QA、安全与发布前收口
Frontend-Design 负责 UI / UX 设计、原型与体验规范收敛
```

如果当前环境尚未直接接入 `gstack`，则必须用现有仓库的独立评审、测试、validation 和执行结果留痕承担同等职责，门禁不降级。

## 3. 核心原则

### 3.1 真相源优先

所有正式结论必须回写到仓库文件，不能只停留在聊天记录里。

### 3.2 规则先于功能

功能只是交付形态，规则才是实现和测试的约束对象。任何需求变更都必须先明确规则范围，再进入实现。

### 3.3 一阶段一个主负责人

同一阶段只允许一个角色拥有最终裁决权。其他角色可以提出输入，但不能越权改范围、改结论、改门禁。

### 3.4 单 Story 实现

`Codex` 默认一次只实现一个 Story 或一个边界清楚的大任务分片，不在编码中途重新定义产品边界。

### 3.5 先验证后收口

未完成测试、验证、独立评审、状态回写、风险记录和执行结果留痕，不得标记任务完成。

### 3.6 稳定结论必须落盘

稳定技术决策、约束、坑点、修复方案、评审结论和收尾摘要，都必须写进仓库文档或实现产物，不能只保留在会话上下文中。

### 3.7 不允许静默扩需求

没有正式输入包、没有明确 `in-scope / out-of-scope`、没有 Story 或变更分级时，不允许在实现阶段自行扩展需求。

### 3.8 原型可辅助，不可越权

HTML 原型、静态预览、假数据 Demo 可以帮助校准产品感觉，但在未被冻结为正式真相源前，不能单独覆盖 PRD、架构、Story 或规则清单。

### 3.9 业务边界优先于工程冲动

哪怕工程上“顺手可以一起做”，也不能越过项目已冻结的业务边界：

- 不重做外部系统能力
- 不默认把高风险动作做成全自动黑盒
- 不把金融协作扩成平台自营金融
- 不把知识、记忆、规则、事实混成一个模糊系统

### 3.10 门禁升级只能增强，不能弱化

当变更级别升高、风险升高、影响面扩大或发现真相源冲突时，门禁只能增强，不能因为赶进度而自行简化。

### 3.11 设计任务主责明确

涉及 UI / UX 的设计任务，包括但不限于原型、页面布局、交互方案、视觉样式、组件表现、主题和颜色策略时，默认由开源工具 `Frontend-Design` 主负责；其他角色只能提供边界、实现和评审输入，不能绕过该工具直接冻结最终设计结论。

### 3.12 页面型增量需求不得跳步

凡是新增页面、修改页面结构、调整角色入口、改变终端归属、重做关键交互，或会影响页面域验收口径的增量需求，必须按以下顺序推进：

`PRD -> UX -> 角色/页面矩阵 -> 关键原型确认 -> Epic / Story 开发 -> 页面域验收`

不允许从 `PRD` 直接跳到 `Epic / Story`、前端实现或页面验收，也不允许在没有角色 / 页面矩阵和关键原型确认的情况下，让研发凭口头描述反推页面方案。

如果变更仅属于纯后端、纯规则、纯脚本或不影响用户可见页面域的 `L0 / L1` 小改，可不强制走完整原型链路；但一旦影响角色、页面、终端、导航、交互或验收口径，就必须恢复完整顺序。

## 4. 真相源顺序

当聊天记录、原型、文档和代码冲突时，XAgents 默认按以下顺序判断：

1. 当前用户明确确认的需求指令与当前批次已确认的变更包
2. `agents.md` 中的仓库铁律、锁规则和强制门禁
3. 已冻结的 `PRD`、`UX` 设计规格、角色 / 页面矩阵、关键原型确认包、架构文档、Epic / Story 拆分、Readiness 报告
4. `docs/implementation-artifacts/sprint-status.yaml`
5. 当前 Story 主文件、对应 validation、当前会话执行结果文件
6. 已正式生效的规则清单、需求追踪矩阵、页面域验收口径、核查总表
7. 已实现代码、测试和脚本
8. `docs/` 下的规范与说明文档，业务源文档、补充说明和专题规范
9. `prototypes/mvp-mobile-agent/` 下的原型快照
10. 历史聊天记录

原则：

- 如果同层文档互相冲突，必须回到变更定义或 Story 校验阶段，不允许只凭口头解释覆盖真相源。
- 如果代码实现与 Story、validation 或冻结规则冲突，默认认为代码跑偏，除非重新冻结变更。
- 如果原型体验与正式产品边界冲突，默认以冻结文档为准；若要抬升原型优先级，必须先把原型约束正式落盘。
- 如果冲突集中在角色入口、页面归属、页面结构、关键交互或页面域验收口径，必须回到 `PRD -> UX -> 角色/页面矩阵 -> 关键原型确认` 链路重新冻结，不允许直接用 Story 或代码覆盖上游设计结论。

## 5. 角色边界

### 5.1 BMAD

`BMAD` 负责：

- 业务想法收敛
- PRD、架构、Epic / Story 拆分
- `in-scope / out-of-scope` 冻结
- Readiness 判断
- 变更分级
- 测试与验收口径的上游定义

`BMAD` 不负责：

- 代替 `Codex` 做日常实现
- 代替 `gstack` 做独立 QA
- 在实现阶段频繁改变边界

### 5.2 Codex

`Codex` 负责：

- 读取代码、文档、测试和现有事实
- 实现单个 Story 或明确分片
- 执行本地命令、验证、调试
- 修复问题并回写代码、测试、文档、状态和执行结果
- 在完工前显式记录未完成项、残留风险和下一接力点

`Codex` 不负责：

- 重新裁定已冻结的产品边界
- 在没有正式输入包时直接扩需求
- 静默跳过测试、评审、validation、状态同步和执行结果留痕

### 5.3 superpowers

`superpowers` 负责：

- 计划拆解
- `TDD`
- 系统化调试
- 完工前验证
- 评审反馈消化与验证策略补强

`superpowers` 不负责：

- 代替 `BMAD` 定义产品边界
- 代替 `Codex` 长时间承担主要业务实现
- 在未重开变更的情况下扩大范围

### 5.4 gstack

`gstack` 负责：

- 独立代码评审
- QA
- 安全检查
- 发布前收口
- 对照需求和边界检查“测试通过但产品跑偏”的问题

如果当前环境尚未接入 `gstack`，则由仓库现有流程中的：

- validation
- 多轮独立评审
- 测试产物
- 执行结果留痕
- 最终收口记录

承担同等职责，但门禁要求不降级。

### 5.5 Frontend-Design

`Frontend-Design` 负责：

- UI / UX 设计任务主导
- 原型、页面结构、交互流程、信息架构和组件表现方案收敛
- 对照 `UI-UE设计规范` 与 `颜色规范使用指南` 校准设计输出
- 为前端实现、Story 验证和设计评审提供可执行设计基线

`Frontend-Design` 不负责：

- 替代 `BMAD` 冻结业务范围
- 替代 `Codex` 完成实现与代码回写
- 脱离正式设计规范直接产出最终设计结论

### 5.6 自动化层

自动化脚本、锁机制、Qdrant、批量审计脚本、模板生成器等配套能力，只能承担以下职责：

- 编排阶段顺序
- 限制写集冲突
- 汇总测试和评审结果
- 生成留痕、报告和辅助基线

不能承担的职责：

- 替代角色裁决
- 私自改写真相源优先级
- 未经确认直接改变产品范围

## 6. 变更分级

### 6.1 L0：文档或小修复

适用：

- 文案、说明、注释、目录整理
- 不改变业务边界的小 bug 修复
- 不触发跨模块连锁影响的规范修正

最小输入：

- 问题描述
- 影响范围
- 验证方式

### 6.2 L1：单 Story 变更

适用：

- 一个用户价值明确的 Story
- 一个模块内的清晰实现分片
- 不跨越既定产品边界的功能补齐

最小输入：

- `Story ID`
- 用户价值
- `in-scope / out-of-scope`
- 验收标准
- 测试路径
- 风险说明

### 6.3 L2：中变更

适用：

- 跨多个模块
- 需要同步修改规则、矩阵、架构或状态机口径
- 涉及价格、审批、路由、知识治理、记忆治理、事实与规则边界、权限、安全、审计、外部契约

最小输入：

- 变更包
- 规则清单
- 需求追踪矩阵
- Story 拆分结果
- 验收与延期项

### 6.4 L3：大变更

适用：

- 产品边界调整
- 核心架构重构
- 数据模型大改
- 外部系统协作模式重大变化
- 安全、合规、金融边界或发布模式变化

最小输入：

- `BMAD` 冻结后的 PRD / 架构结论
- 分阶段实施计划
- 回滚或降级策略
- 独立评审与发布前收口门禁

原则：

- 未分级的需求不进入编码。
- 价格、审批、路由、金融、知识、记忆、合规类改动默认不按 L0 处理。
- 变更级别越高，验证、评审、留痕和回写要求越强。

## 7. 本仓库强制门禁适配

### 7.1 BMAD 阶段铁律：8.10 / 8.11 / 8.12

当前仓库明确采用以下阶段约定：

- `8.10`：Create Story / 建档
- `8.11`：Validate Story / 校验可开发性
- `8.12`：Dev Story / 实现与验证

强制规则：

1. `8.10`、`8.11`、`8.12` 每个步骤完成后，必须执行两轮独立的多维度评审。
2. 两轮评审都必须显式记录 `结论 / 发现项 / 修正结果`。
3. 若第一轮修正后仍出现新增实质问题，第二轮继续以独立视角复核。
4. 只有在连续两轮评审都没有新增实质阻塞问题时，当前步骤才可视为完成。
5. 未完成双轮评审，不得进入下一步。

推荐评审维度：

- 需求与 Story 对齐
- `in-scope / out-of-scope` 是否清晰
- 证据链是否完整
- 验收标准是否可验证
- 风险与依赖是否显式
- 文档、代码、测试、状态是否同步
- 是否违反业务边界、审批边界、价格边界、路由边界、知识或记忆治理边界

### 7.2 Story 写锁是开发前置门禁

进入任何 Story 的开发、校验、评审或收口前，必须先获取本地 Story 写锁。

基本要求：

- 未持有 Story 写锁，不得修改该 Story 主文件、validation。
- 未持有 Story 写锁，不得修改该 Story 写集内的实现文件、测试文件、脚本文件。
- 如果当前 Story 还会写执行结果文件或共享文档，必须在获取锁时通过 `--add <repo/path>` 显式声明。
- 同一 owner 同一时刻只应持有一个活跃 Story 锁。
- 完成、暂停或切换 Story 时，必须及时释放锁；长时间开发需续租。

推荐命令：

```powershell
.\scripts\story-lock.ps1 -Action acquire -Story <story_key> -Owner <session_name> -Add docs/implementation-artifacts/<session_name>-执行结果.md
.\scripts\story-lock.ps1 -Action renew -Story <story_key> -Owner <session_name>
.\scripts\story-lock.ps1 -Action release -Story <story_key> -Owner <session_name>
```

补充说明：

- 不属于受保护写集的通用规范文档，例如 `docs/` 下的普通规范文件，默认可按普通文档流程修改。
- 但如果某个 Story 明确要求回写某份共享规范文档，则应在 Story 锁获取时通过 `--add` 纳入覆盖。

### 7.3 `sprint-status.yaml` 与总表采用双层锁

以下共享文件采用“Story 锁 + 共享短锁”双层约束：

- `docs/implementation-artifacts/sprint-status.yaml`
- `docs/planning-artifacts/三源文档全量核查总表.md`

规则如下：

- 只有在当前 owner 已持有覆盖该路径的 Story 写锁前提下，才允许真正写入前获取共享短锁。
- 写完后必须立即执行 `path-release` 释放。
- 目标单次占用不超过 `30` 秒。
- 共享短锁只覆盖实际写入动作，不覆盖思考、分析、编码、测试或等待时间。

推荐命令：

```powershell
.\scripts\story-lock.ps1 -Action path-acquire -RepoPath docs/implementation-artifacts/sprint-status.yaml -Owner <session_name> -TtlSeconds 30
.\scripts\story-lock.ps1 -Action path-release -RepoPath docs/implementation-artifacts/sprint-status.yaml -Owner <session_name>
.\scripts\story-lock.ps1 -Action path-acquire -RepoPath docs/planning-artifacts/三源文档全量核查总表.md -Owner <session_name> -TtlSeconds 30
.\scripts\story-lock.ps1 -Action path-release -RepoPath docs/planning-artifacts/三源文档全量核查总表.md -Owner <session_name>
```

### 7.4 会话执行结果文件必须独立回写

矩阵并行开发或多会话协同时，每个会话只应回写自己的执行结果文件：

- `docs/implementation-artifacts/<session_name>-执行结果.md`

以下共享文件只允许在批次收尾阶段由单一协调会话统一更新：

- `matrix-batch-<批次编号>-分配表.md`
- `业务构想/需求实际实施进展覆盖矩阵.md`

### 7.5 Git hooks 与受保护路径校验

仓库 Git hooks 通过以下命令安装：

```powershell
npm run story:hooks:install
```

安装后：

- `pre-commit`
- `pre-push`

会校验受保护路径是否已被 active Story 锁覆盖。

要求：

- 凡是提交或推送受保护改动，必须显式设置 `STORY_LOCK_OWNER`。
- 共享短锁只用于运行期串行写入，不替代 hooks 对 Story 锁覆盖的要求。

### 7.6 测试、验证与未执行说明

任何代码、脚本、契约、状态机、规则变更，都应执行与影响面相匹配的验证：

- 单元测试
- 集成测试
- 关键路径手测
- 契约 / 状态 / 文档一致性检查
- 必要时的回归检查

要求：

- 能跑的测试必须跑。
- 不能跑的测试必须明确写出未执行原因、影响范围和下一步补救点。
- 不允许用“理论上应该没问题”替代验证记录。

### 7.7 Qdrant 开发记忆隔离规则

若当前会话启用 Qdrant 作为 `Codex` 开发记忆增强，必须遵守项目专属隔离规范。

推荐基线：

- 共用现有 Qdrant 服务实例
- 单独使用 collection：`caragents_codex_memory`
- 必须使用 metadata 作为第二层过滤

最低 metadata 过滤建议：

- `project_key = x_tradein_agent`
- `repo_key = caragents`
- `memory_scope = codex_dev`
- `status = active`

原则：

- 开发记忆只沉淀决策、约束、坑点、修复、评审结论和总结。
- 不写入客户身份证、手机号、金融材料、审批原件、实时业务敏感真相数据。
- 开发记忆记“怎么做项目”，不记“客户是谁、订单真相是什么”。

### 7.8 文件与命令安全

默认安全规则：

- 未经授权不得删除文件
- 禁止使用破坏性 Git 命令
- 不擅自创建或切换分支来绕过当前工作区事实
- 不把密钥、令牌、Cookie、真实个人敏感信息写入仓库
- 不用聊天解释代替实际回写

## 8. 推荐工作流

涉及页面、角色入口、终端归属、关键交互或页面域验收口径的增量需求，当前仓库强制采用以下顺序：

`PRD -> UX -> 角色/页面矩阵 -> 关键原型确认 -> Epic / Story 开发 -> 页面域验收`

责任划分：

- `BMAD` 负责冻结 `PRD`、边界和后续 `Epic / Story` 拆分输入。
- `Frontend-Design` 负责 `UX`、角色 / 页面矩阵、关键原型确认与页面域验收基线。
- `Codex` 负责在上游输入冻结后按 `Epic / Story` 实现，不反向发明页面方案。

门禁要求：

- 页面型需求不得从 `PRD` 直接跳到 `Epic / Story` 开发。
- 未形成角色 / 页面矩阵与关键原型确认结论，不得启动相关页面 Story。
- 页面完成后默认按页面域验收，不按“单页能打开”或“接口已通”判定完成。

### 8.1 阶段零：业务发现与收敛

主负责人：`BMAD`

最低产出：

- 业务背景
- 角色矩阵
- 核心闭环
- 候选规则清单
- 非目标与开放问题

### 8.2 阶段一：定义变更

主负责人：`BMAD`

最低产出：

- `PRD`
- 架构方向
- 变更分级
- `Epic / Story` 拆分
- 规则清单
- 需求追踪矩阵

若变更涉及页面、角色、终端或关键交互，还必须在进入 Story 阶段前补齐 `UX` 输入包，并转交 `Frontend-Design` 继续收敛角色 / 页面矩阵与关键原型确认。

进入下一阶段的条件：

- 已明确一期做什么、不做什么
- 已明确角色、关键路径和关键人工确认点
- 已明确不重做拍卖 / 出海 / 自营金融边界

### 8.3 阶段二：实现准备就绪

主负责人：`BMAD`

最低产出：

- 已确认的 `in-scope / out-of-scope`
- Readiness 结论
- 风险与延期项
- 测试路径
- 文档回写点

如果当前需求属于页面型增量需求，则进入本阶段前应已完成：

- `UX` 设计规格或等效增量设计结论
- 角色 / 页面矩阵
- 关键原型确认
- 页面域验收口径

### 8.4 阶段三：8.10 Create Story

主负责人：`BMAD` 或在其冻结边界下由 `Codex` 建档

最低要求：

- 形成单独 Story 主文件
- 明确用户价值、依赖、落点、验收标准、边界和风险
- 完成两轮独立多维度评审并显式记录

### 8.5 阶段四：8.11 Validate Story

主负责人：`BMAD`

最低要求：

- 明确 Story 是否 `Ready / Conditional Ready / Needs Work`
- 校验上下游依赖、验收标准、测试路径、风险与门禁是否完整
- 完成两轮独立多维度评审并显式记录

### 8.6 阶段五：8.12 Dev Story

主负责人：`Codex`

协作层：

- `superpowers` 负责计划、TDD、调试和完工前验证

执行约束：

- 每次只实现一个 Story 或一个边界清楚的分片
- 实现前先拿 Story 写锁
- 发现需求冲突、技术不可行或风险显著升高时，必须回到上游重新裁决
- 完成后必须执行双轮独立评审并显式记录

### 8.7 阶段六：独立评审与收口

主负责人：`gstack` 或等效独立流程

最低动作：

- 代码 / 文档 / 状态一致性复核
- 测试与风险复核
- 安全与边界检查
- 发布阻塞项或收口结论

### 8.8 阶段七：回写与复盘

主负责人：`Codex`

最低动作：

- 回写 Story、validation、执行结果
- 必要时回写 `sprint-status.yaml`
- 记录已完成、未完成、验证命令、残留风险和下一接力点
- 释放 Story 锁和共享短锁

## 9. Story 输入包与完工定义

### 9.1 开工前最小输入

每个 Story 开工前至少要有：

- `Story ID`
- 标题与用户价值
- `in-scope`
- `out-of-scope`
- 关联业务场景或角色
- 影响模块 / 文件落点
- 验收标准
- 测试路径
- 风险与依赖
- 文档回写点

如果是页面型 Story，还至少要有：

- 对应的角色 / 页面矩阵条目
- 对应的关键原型确认条目，或显式说明该页为何不属于关键原型页
- 页面终端归属
- 页面域验收口径

推荐补齐项：

- Given / When / Then 形式的验收条目
- 明确的失败路径与人工兜底入口
- 所需会话结果文件路径

### 9.2 完工定义

每个 Story 收口时至少要满足：

- 代码或文档改动已完成
- Story 主文件与 validation 已同步
- 关键验证已执行，或明确记录未执行原因
- 双轮独立评审已完成，且无新增实质阻塞问题
- 页面型 Story 已完成页面域验收，不以“单页能打开”替代
- `sprint-status.yaml` 已在需要时完成短临界区更新
- 会话执行结果文件已回写
- 残留风险与下一接力点已写清
- Story 锁与共享短锁已释放

不满足以上条件，不标记为 `done`。

---

## 10. 评审标准与记录要求

### 10.1 第一轮评审建议维度

- 是否符合需求原意
- 是否存在明显漏项、错项、越界项
- 证据链是否完整
- 文档与代码是否同步
- 测试是否覆盖主路径

### 10.2 第二轮评审建议维度

- 是否仍存在新增实质问题
- 修正后是否引入回归
- 风险、状态、执行结果是否同步
- 边界、审批、价格、路由、知识 / 记忆治理是否被破坏
- 结论是否可以支持进入下一阶段或正式收口

### 10.3 评审记录最低格式

每轮评审至少记录：

- `评审轮次`
- `评审结论`
- `发现项`
- `修正结果`
- `是否还有新增实质阻塞问题`

建议直接追加到对应 validation 文件末尾，避免评审结论散落在聊天记录或临时笔记中。

## 11. 常见禁止动作

以下行为默认禁止：

- 未冻结范围就直接进入编码
- 未拿 Story 锁就修改受保护路径
- 页面型增量需求从 `PRD` 直接跳到 `Epic / Story` 或开发，跳过 `UX`、角色 / 页面矩阵或关键原型确认
- 把共享短锁当成长时间占用锁
- 只改代码，不回写 Story / validation / 执行结果
- 把页面验收降格为“单页能打开”或“接口能返回”
- 只在聊天里说明“已验证”，但仓库里没有任何留痕
- 用原型、口头解释或历史聊天覆盖当前真相源
- 在实现中途私自扩大一期范围
- 把外部系统能力重新做进本项目

## 13. 一页式执行基线

如果只记最少几条，请记下面这版：

1. 页面型需求先冻结 `PRD -> UX -> 角色/页面矩阵 -> 关键原型确认`，再拆 `Epic / Story` 与写代码。
2. `8.10 / 8.11 / 8.12` 每一步完成后都必须做两轮独立多维度评审。
3. Story 开发、校验、评审、收口前必须先拿 Story 写锁。
4. 写 `sprint-status.yaml` 和总表时，必须先拿共享短锁，写完立即释放。
5. 代码、Story、validation、执行结果、状态必须同步回写。
6. 页面完成后默认按页面域验收，不按“单页能打开”收口。
7. 关键价格、审批、路由、金融边界、知识与记忆治理边界不能静默破坏。
